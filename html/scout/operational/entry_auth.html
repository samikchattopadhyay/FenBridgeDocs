Workflow Overview

Mobile + OTP is the sole authentication path for FenBridge. Country defaults to India (+91); 10-digit Indian mobile validated. The same entry screen serves all roles — after successful authentication, users are routed to their role-specific dashboard. For newly registered users (farmers/poppers), scouts register them on their behalf, and they receive their profile pre-populated when they first log in. Consent is captured explicitly, and OTP verification is real-time SMS delivery.

Purpose
Capture the user's mobile number and request an OTP.
Business Requirement
Mobile + OTP is the sole auth path. Country defaults to India (+91); 10-digit Indian mobile validated. Rate-limited to prevent OTP spam. The same screen serves all roles — Splash routes to dashboard by role after auth succeeds.
User Experience (UX)
Single-field screen. Auto-focus, numeric keypad. Continue enabled on valid 10-digit number. Helper line clarifying "We'll send you a 6-digit OTP".

• Enter 10-digit mobile
• Tap "Send OTP" → OTP Verification screen
• Tap country code → country picker (rarely used in Year 1)
Technical Details
Validation: Indian mobile regex ^[6-9][0-9]{9}$. Calls Supabase auth.signInWithOtp({ phone }) with phone in E.164 (+91XXXXXXXXXX). Supabase forwards OTP via configured SMS provider (MSG91 / Twilio). Rate limit: 3 OTPs per number per hour (server enforced); client shows a friendly 60s cooldown after each send. No user record needs to pre-exist — Supabase creates an auth row on first send; the FenBridge profile (users table) is created/linked on first successful verify. Fully offline-incompatible (requires network); offline message: "You need to be online to log in."
Design Details
Top bar: none (this is the entry screen). Brand mark (small, top-centre). Headline: "Log in to FenBridge" (Title 2). Sub-line: "Enter your mobile number — we'll send you an OTP" (text-md). Database Schema
No new tables. Uses Supabase auth.users (managed) and the existing users table that links auth ID to FenBridge profile. The users table must expose: id, auth_user_id, role, mobile, mobile_verified_at, status (pending_approval / active / suspended), created_via_scout_id (nullable, set for scout-assisted onboarding).

Successful scout login: route to Scout Dashboard. Token persisted in local secure storage; refresh handled by Supabase SDK. Auto-read OTP from SMS where Android Autofill API supports it.

Backend Architecture & Implementation Guide
Entry & Authentication Flow

React Component Decomposition & Reusability

Entry & Authentication uses a minimal, single-purpose component set focused on auth flow. Follow atomic design: atoms (inputs) → molecules (form groups) → templates (screens).

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: All auth state managed by Supabase SDK
• Reuse across all user roles: Same auth flow for scouts, farmers, poppers, buyers
• Minimal footprint: Auth screens are lightweight, no heavy dependencies

Auth Flow Sequence

1. Splash: Check secure storage for valid JWT → if found & valid, resolve role & redirect to dashboard. Otherwise, proceed to Mobile Entry.

2. Mobile Entry: User enters 10-digit number → "Send OTP" → Supabase auth.signInWithOtp({ phone: "+91..." }) → SMS sent via MSG91/Twilio → proceed to OTP Verification.

3. OTP Verification: User enters 6-digit OTP → auto-verify or tap Verify → Supabase auth.verifyOtp({ phone, token, type: 'sms' }) → on success: lookup/create users row → resolve role → store JWT in secure storage → redirect to role-specific dashboard.

Error Handling: Invalid OTP → clear field, show "Incorrect OTP — try again" → allow 3 retries before brief lockout → "Resend OTP" becomes available after 30s.

Common Patterns & Constraints

  • Offline: Auth flows require network (SMS delivery). Offline message: "You need to be online to log in."
  • Rate Limiting: 3 OTPs per phone per hour (server-enforced); UI shows 60s cooldown per send.
  • Role Resolution: After successful verify, fetch users.role. If no role (brand-new auth user), show "Contact a scout" message.
  • Token Storage: JWT stored in secure storage (iOS Keychain / Android Keystore); Supabase SDK handles refresh on expiry.
  • Cross-Role: Same auth screens serve all user types (scouts, farmers, poppers, buyers). Dashboard routing happens post-auth.
