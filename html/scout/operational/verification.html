Scout Operations

Lot Verification Inspection

Complete 7-step workflow for inspecting and verifying incoming lots of raw makhana with quality assessment, reference samples, and approval queue.

Scout inspects a newly received lot of raw makhana at the warehouse or collection center. Captures quality data (color, moisture, defects), takes reference photos, and submits for admin approval. Lot enters pending_approval state until verified.

Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedure Architecture (7 Procedures)

The Lot Verification Inspection workflow consists of 7 tRPC procedures for lot intake, quality assessment, photo capture, and submission.

Procedure Breakdown:
• 1 Public procedure: Initiate lot verification
• 5 Protected procedures: Quality data, photos, reference samples
• 1 Protected procedure: Submit verification (atomic transaction, 4 inserts)

File: /src/server/api/routers/lot.ts

export const lotRouter = t.router({ // Screen 1: Init Lot Verification initLotVerification: publicProcedure, // Screens 2-4: Quality Assessment submitQualityMetrics: protectedProcedure, submitLotPhotos: protectedProcedure, submitReferenceSamples: protectedProcedure, // Screens 5-6: Assessment Details submitDefectAnalysis: protectedProcedure, submitGradeAssignment: protectedProcedure, // Screen 7: Submit submitLotVerification: protectedProcedure, });

Atomic Transaction (Screen 7 - Submit)

  • 1. Insert lot_inspections record (status=pending_approval)
  • 2. Insert lot_inspection_photos (array, up to 10 per lot)
  • 3. Insert lot_inspection_reference_samples (array)
  • 4. Insert lot_inspection_events (audit trail)
  • → On success: Trigger async Inngest job lot.inspection-submitted
  • → On failure: Rollback all 4 inserts (no partial state)

Core Database Tables (Drizzle ORM)

lot_inspections (Main Record)

CREATE TABLE lot_inspections ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_id UUID NOT NULL REFERENCES lots(id), scout_id UUID NOT NULL REFERENCES scouts(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), status ENUM('pending_approval','approved','rejected'), color_grade VARCHAR(50), moisture_percentage DECIMAL(5,2), defect_percentage DECIMAL(5,2), overall_grade VARCHAR(10), notes TEXT, submitted_at TIMESTAMPTZ NOT NULL, approved_by_admin_id UUID REFERENCES users(id), approved_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT NOW() );

lot_inspection_photos (Photos)

CREATE TABLE lot_inspection_photos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_inspection_id UUID NOT NULL REFERENCES lot_inspections(id), photo_url TEXT NOT NULL, photo_type ENUM('full_lot','close_up','defect'), caption TEXT, created_at TIMESTAMPTZ DEFAULT NOW() );

lot_inspection_reference_samples (Lab Samples)

CREATE TABLE lot_inspection_reference_samples ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_inspection_id UUID NOT NULL REFERENCES lot_inspections(id), sample_id VARCHAR(50) UNIQUE NOT NULL, weight_grams DECIMAL(8,2), lab_test_status ENUM('pending','completed','failed'), lab_result_url TEXT, created_at TIMESTAMPTZ DEFAULT NOW() );
⚠️ Encryption:
• Photo URLs: Signed/expiring S3-style URLs (Supabase Storage)
• Reference sample data: Plain text (no PII)

Input Validation & Error Handling

Validation Strategy

All inputs validated with Zod schemas:
• Quality metrics: moisture 0-30%, defects 0-100%
• Photos: max 10 per lot, JPEG/PNG only, <5MB each
• Reference samples: weight > 0, unique sample_ids
• Grade: must be A, B, C, or D

Async Jobs with Inngest (Serverless Queue)

1. lot.inspection-submitted (Chained)

Trigger: Screen 7 "Submit Verification"
Steps: Verify lot exists → Calculate composite grade → Notify admin → Update status
Retry: 3 attempts with exponential backoff

2. lot.approved (Event-Triggered)

Trigger: Admin approves lot in dashboard
Steps: Update lot status → Send SMS to farmer → Update inventory

Security Best Practices

Sensitive Data Handling

Never Log:
✗ Full photo URLs (log only domain)
✗ Scout's exact GPS coordinates

Safe to Log:
✓ Lot ID, warehouse ID, scout ID
✓ Quality metrics, grades
✓ Event types and timestamps

Photo Upload Security

  • Store photos in Supabase Storage with RLS (scout can only view own uploads)
  • Generate signed URLs with 24-hour expiration
  • Scan photos for malware before storage
  • Compress images to max 2MB before upload

Lot Verification Workflow
7 Steps
1Initialize Lot Verification
Lot selection screen
Purpose
Scout selects the incoming lot to be verified.
User Experience
Dropdown list of unverified lots with lot ID, supplier, quantity, date received.
Technical Details
Wizard context initialized: { lot_id, warehouse_id, scout_id }. No DB write yet.
Design Details
Lot selection dropdown with timestamps, quick search by lot ID.
2Quality Metrics Assessment
Quality assessment form
Purpose
Captures primary quality indicators: color, moisture, defects.
Business Requirement
Color grade (A/B/C/D), moisture % (0-30), visible defect %. Required for lot approval.
User Experience
Form with sliders for moisture/defects, color grade selector.
Technical Details
Validation: moisture 0-30% (decimals), defects 0-100%, color A/B/C/D.
3Lot Photos
Photo grid preview
Purpose
Scout captures reference photos of the lot at intake.
Business Requirement
Minimum 3 photos required: full lot view, close-up of quality, defect areas. Max 10 photos.
User Experience
Photo grid with camera integration, type selector (full lot/close-up/defect).
Technical Details
Photos compressed (JPEG, max 5MB), uploaded to Supabase Storage, queued for sync.
4Reference Samples
Sample details form
Purpose
Scout collects sample(s) from the lot for lab testing.
Business Requirement
Unique sample ID, weight in grams. Samples sent to lab for confirmation testing.
User Experience
Form to enter sample IDs and weights, add multiple samples.
Technical Details
Sample IDs must be unique across all lots. Weight > 0g. Status: pending_lab_test.
5Defect Analysis
Defect checklist
Purpose
Scout identifies specific defect types to categorize quality.
Business Requirement
Check for: foreign material, discoloration, damage, under-formed. Count if possible.
User Experience
Checklist of defect types with optional count fields.
Technical Details
Stored in context as defect_types array. Used to refine overall grade.
6Overall Grade Assignment
Grade summary
Purpose
Final quality grade based on all assessments.
Business Requirement
Auto-calculated from color + moisture + defects, or override by scout. Grade A/B/C/D.
User Experience
Show calculated grade, allow manual adjustment with confirmation.
Technical Details
Server-side grade calc: A (color A + moisture <15% + defects <5%), B/C/D by metric combinations.
7Submit Verification
Review summary
Purpose
Final review and submission of lot verification.
User Experience
Summary card showing all metrics, photos count, grade. Final confirmation CTA.
Technical Details
Atomic insert: lot_inspections, lot_inspection_photos, lot_inspection_reference_samples, lot_inspection_events. Status: pending_approval. Triggers async job.
Design Details
Summary sections with edit links, final confirmation checkbox, submit button.
