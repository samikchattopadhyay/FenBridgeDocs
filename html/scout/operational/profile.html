tRPC Procedures (5 Screens)

This dashboard manages regional boundaries, language locales, and local database capacity metrics.

State Operations:
• Locale and configurations persist locally. Account status queries run securely against territory registries.

Key Procedures

  • getScoutProfile — Protected. Fetches current scout details and mappings.
  • updateScoutProfile — Protected. Updates basic contact details.
  • getTerritoryDetails — Protected. Returns assigned district villages.

React Component Decomposition (5 Screens)

The Scout Profile workflow breaks into 10 reusable components: 6 shared across Scout module workflows, 4 workflow-specific for scout profile settings. Components follow atomic design principles with strict TypeScript interfaces.

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% components shared across 20 workflows
• Variant management: Props define UI variants, not separate components

Screen-by-Screen Component Breakdown

Screen 1: Edit Personal Details
Purpose: Modify scout personal information (name, phone, avatar)
Layout: TopAppBar → ScoutAvatarCard → TextInput/PhoneInput → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Edit Personal Details", menu
• ScoutAvatarCard (workflow-specific) — Avatar image + scout name + badge number display
• TextInput (shared) — "Full Name" with validation
• PhoneInput (shared) — "+91 Mobile" with read-only flag if verified
• CTAFooter (shared) — "Update Profile" (enabled on change), "Cancel"
State Management: Profile edits in local component state, submitted via updateScoutProfile()
Validation: Name length 3-100, valid phone format, phone must match verification if changing
Screen 2: Assigned Geographical Territory
Purpose: View assigned district boundaries and operational villages
Layout: TopAppBar → TerritoryMapCard → TerritoryVillagesCard → SupportContactCard → ReadOnly
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Assigned Territory", menu
• TerritoryMapCard (workflow-specific) — Interactive map showing:
  ├ District boundary polygon (orange/cyan outline)
  ├ Village markers with names
  └ Scout location marker
• TerritoryVillagesCard (workflow-specific) — List of assigned villages:
  ├ Village name + panchayat
  ├ Household count
  └ Distance from scout location
• SupportContactCard (workflow-specific) — Emergency/regional support contact details
State Management: Territory data fetched via getTerritoryDetails(), cached locally
Interaction: Read-only view (no edits, back button only)
Offline Support: Territory boundaries cached in SQLite, works fully offline
Screen 3: Toggle Language Settings
Purpose: Select application UI language (English, Hindi, Maithili)
Layout: TopAppBar → LanguageSelectionCard → InfoBanner → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Language Settings", menu
• LanguageSelectionCard (workflow-specific) — Radio button group:
  ├ RadioButton (shared) — "English" with flag icon
  ├ RadioButton (shared) — "हिंदी (Hindi)" with flag icon
  └ RadioButton (shared) — "मैथिली (Maithili)" with flag icon
• InfoBanner (shared) — "App interface will update immediately"
• CTAFooter (shared) — "Save Language" (instant application), "Cancel"
State Management: Selected language_code in local state, persisted to device settings
Side Effect: Upon selection, app interface reloads with new language dictionaries
Validation: Only valid language codes (en, hi, mai)
Screen 4: Database Sync Status
Purpose: Display local database capacity and sync health metrics
Layout: TopAppBar → SyncMetricsCard + StorageUtilizationCard → QueueMetricsCard → ClearCacheCard → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Sync Status", menu
• SyncMetricsCard (workflow-specific) — Key metrics display:
  ├ Last sync timestamp
  ├ Sync status (synced/pending/failed)
  ├ Network connection status (online/offline)
  └ Pending items count
• StorageUtilizationCard (workflow-specific) — Circular progress chart:
  ├ Used vs. total capacity in MB
  ├ Color gradient (green <70%, yellow 70-90%, red >90%)
  └ Cache size breakdown (photos, documents, inspections)
• QueueMetricsCard (workflow-specific) — Queue status:
  ├ Pending submissions (forms waiting sync)
  ├ Queued photos (offline uploads)
  └ Last successful sync date
• ClearCacheButton (workflow-specific) — CTA to clear non-essential cache
• CTAFooter (shared) — "Back" (read-only view)
State Management: Metrics queried from SQLite, auto-refresh every 30s
Interaction: Read-only view + cache clear action
Screen 5: Logout Confirmation Details
Purpose: Safely logout with warning about unsynced data
Layout: TopAppBar → LogoutWarningBanner → UnsyncedItemsCard → ConfirmationCheckbox → LogoutCTA
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Logout", menu
• WarningBanner (shared) — Red critical: "You have unsynced items. They will be lost on logout."
• UnsyncedItemsCard (workflow-specific) — Shows pending items if any:
  ├ Count of pending forms (farm registrations, inspections)
  ├ Count of queued photos
  ├ Timestamp of last sync
  └ "Sync now" action link
• ConfirmationCheckbox (shared) — "I understand my unsync data will be lost"
• LogoutCTA (workflow-specific) — Dark crimson button "Logout" (disabled until checkbox)
• CTAFooter (shared) — "Cancel" secondary action
State Management: Unsync status queried from sync queue table
Logout Flow: On confirm, wipes auth tokens + local session data, redirects to login
Validation: Checkbox must be checked before logout enabled

Cross-Workflow Component Reuse Pattern

⚠️ CRITICAL: Zero Duplication Rule

Already Shared Across Multiple Workflows:
• TextInput, PhoneInput — All shared. DO NOT recreate.
• RadioButton — New shared component (language, settings, selection lists). Create ONCE for reuse across 20 workflows.
• InfoBanner, WarningBanner — Shared. DO NOT create new banner variants.
• ConfirmationCheckbox, CTAFooter, TopAppBar — All shared. DO NOT recreate.

This Workflow's New Components:
Only 4 components are UNIQUE to Scout Profile:
1. ScoutAvatarCard — Displays scout avatar + badge (profile-specific)
2. TerritoryMapCard — Interactive map with boundaries + village markers
3. SyncMetricsCard + StorageUtilizationCard + QueueMetricsCard — Diagnostics cards
4. UnsyncedItemsCard — Warning card listing pending items

DO's:
✓ Check COMPONENT_ARCHITECTURE_GUIDE.md BEFORE creating any component
✓ Create RadioButton as shared component (used in language selection, settings, status filters)
✓ Reuse TextInput, PhoneInput, InfoBanner, WarningBanner, ConfirmationCheckbox directly
✓ Create only truly unique components (ScoutAvatarCard, TerritoryMapCard)
✓ Use props for variant display (info vs. warning banner, success vs. error colors)

DON'Ts:
✗ DO NOT create LanguageButton or LanguageOption components — use RadioButton with variant props
✗ DO NOT create separate SyncStatusBanner — use WarningBanner component
✗ DO NOT duplicate banner components — reuse InfoBanner/WarningBanner with severity prop
✗ DO NOT create ProfileHeaderCard twice — reuse ScoutAvatarCard pattern
✗ DO NOT create custom logout button — use CTAFooter with dark crimson variant
✗ DO NOT create new input components — reuse TextInput/PhoneInput with props

Consistency & Mobile Responsive Guidelines

  • ✓ Spacing: 16px gap between sections, 24px padding inside cards, 80px footer height
  • ✓ Colors: Use CSS variables (--accent-primary: #06b6d4, --text-main: #f3f4f6)
  • ✓ Typography: Outfit 800 for titles, Inter 400/600 for body, 12-14px labels
  • ✓ Mobile Breakpoints: 360px (mobile), 600px (tablet), 1024px (desktop)
  • ✓ Form Validation: Frontend mirrors backend Zod, real-time error display
  • ✓ Accessibility: aria-label on all buttons, labels paired with inputs, focus states
  • ✓ Map Display: TerritoryMapCard uses standard map library (Mapbox/Google Maps) with offline tile caching
  • ✓ Chart Display: StorageUtilizationCard uses circular progress (canvas or SVG), color gradient for capacity warnings
  • ✓ Logout Flow: Critical warning banner + checkbox requirement + crimson CTA for confirmation
  • ✓ Read-Only Screens: Territory and Sync screens are read-only with back button only
  • ✓ State Management: Store profile edits locally, language preference in device settings, sync metrics from SQLite
  • ✓ Error Handling: Consistent red border + message for validation errors

Database Schemas (Drizzle ORM DDL)

scouts (Profile Header Registry)

CREATE TABLE scouts ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id), badge_number TEXT NOT NULL UNIQUE, full_name TEXT NOT NULL, district_assignment TEXT NOT NULL, active BOOLEAN NOT NULL DEFAULT true, created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

Strict Input Validations & Error Enforcements

  • Phone Format: Verified before updating profiles.

Offline Caching & Queue Synchronization

  • Offline Storage: Profile and territory metadata cache inside SQLite tables to permit offline navigation.

Security & Media Protection Protocols

  • Territory Lockouts: Scouts cannot register lots or farmers located outside their assigned boundaries.
