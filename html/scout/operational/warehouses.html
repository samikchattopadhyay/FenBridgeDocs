Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Warehouses Directory workflow provides a hierarchical audit trail and traceability interface for supply-chain visibility. The 3-screen flow navigates from warehouse discovery, to performance profiling, to single-lot traceability with full chronological history. This workflow is read-only—designed for Scouts and Admin users to verify compliance and troubleshoot logistics.

Key patterns: Search and filtering, hierarchical navigation, tabbed detail views, timeline visualization of lot journey.

Screen Decomposition

Screen 1: Warehouse Directory

Searchable list of all warehouses in the scout's district. Filters by fitness status and distance. Shows lot counts and last check date.

TopAppBar { title: "Warehouses", backBtn } | StepperPanel { active: 1 } | SearchInput { placeholder: "Search warehouses..." } | SelectInput[] { fitness: ["Fit", "Unfit"], distance: ["Within 5km", "5-10km", "10+km"] } | DataTable { headers: ["Warehouse", "Popper", "Status", "Lots", "Last Check"], rows: warehouseList, onRowClick: selectWarehouse } | CTAFooter { "Done" → home }

Screen 2: Warehouse Profile

360-degree view of a warehouse. Tabbed view with overview (fitness history), lots (active seals), and fitness logs (audit trail).

TopAppBar { title: warehouse.name, backBtn } | StepperPanel { active: 2 } | TabNavigation { tab_1: "Overview", tab_2: "Active Lots", tab_3: "Fitness Logs" } | [Tab 1: Overview] StatusBadge { status: warehouse.fitness_status, color: green|red } | TextInput[] { location, popper_name, last_check_date [read-only] } | [Tab 2: Lots] DataTable { sealed_lots, in_transit_lots } | [Tab 3: Logs] DataTable { history: fitness_check_records, chronological } | CTAFooter { "Navigate" | "Start Verification" | "Back" }

Screen 3: Lot Audit Detail

Single Source of Truth for a lot. Chronological timeline from initial inspection through seal, re-checks, and current status.

TopAppBar { title: "Lot " + lot.id, backBtn } | StepperPanel { active: 3 } | TextInput[] { lot_id, seal_id, grade_cert_id [read-only] } | StatusBadge { status: lot.current_status } | Timeline { events: [ { date, event: "Initial Inspection", grade }, { date, event: "Seal Applied", seal_id }, { date, event: "Re-check", verdict }, { date, event: "Current Status", status } ] } | CTAFooter { "Download Grade Cert" [PDF] | "Back" }

State Management & Data Flow

Navigation & Selection State

const [navigationState, setNavigationState] = useState({ screen: 1, // 1=directory, 2=profile, 3=lot_detail selected_warehouse_id: string | null; selected_lot_id: string | null; search_query: string; filters: { fitness_status: "Fit" | "Unfit" | null; distance_radius: "5km" | "10km" | "50km" | null; }; active_tab: 1 | 2 | 3; // Screen 2 tabs });

Data Loading & Aggregation

// Screen 1: Fetch warehouse list (pre-cached) const warehouseList = await db.warehouses .where({ district_id: scoutDistrict }) .select(['id', 'name', 'popper_id', 'fitness_status', 'last_check_at']) .orderBy('name'); // Screen 2: Aggregate warehouse details const warehouseProfile = { ...warehouse, active_lots_count: (SELECT COUNT(*) FROM lots WHERE warehouse_id = $1 AND status IN ('sealed', 'in_transit')), fitness_check_history: (SELECT * FROM warehouse_fitness_checks WHERE warehouse_id = $1 ORDER BY check_date DESC) }; // Screen 3: Full lot traceability const lotAuditTrail = { ...lot, initial_inspection: (SELECT * FROM lot_inspections WHERE lot_id = $1 LIMIT 1), seal_record: (SELECT * FROM lot_seals WHERE lot_id = $1), recheck_logs: (SELECT * FROM recheck_logs WHERE lot_id = $1 ORDER BY created_at), current_status: lot.status };

Offline Cache Strategy

// Screen 1: Full warehouse list cached (max 1000 records) // Index: district_id + fitness_status + distance Cache: warehouseList (1 week TTL) // Screen 2: Fetch on demand with fallback to cache Cache: warehouseProfile (3 day TTL per warehouse) // Screen 3: Deep fetch on demand, no offline access // (Lot traceability requires latest server data) NoCache: lotAuditTrail (always fetch fresh)

Common Patterns & Reusability

  • Hierarchical Navigation: 3-screen drill-down (list → detail → deep detail) reused pattern across all directory workflows. Extract as <HierarchicalNavigator screens={[...]} onSelect={callback} />
  • Searchable Data Table: Search + filter + sort pattern (Screen 1) reused from Registered Users and Ponds Directory. Centralize <SearchableTable data={[...]} filters={[...]} />
  • Timeline Visualization: Chronological timeline (Screen 3) applied to any audit-trail or traceability scenario. Extract as <AuditTimeline events={[...]} />
  • Tabbed Profile View: Tabbed detail screen (Screen 2) mirrors pattern from Farmer/Popper profiles. Consistent UX across module.
  • Read-Only Principle: Entire workflow is audit/reporting only—no mutations. No offline sync needed. Designs show confidence in data freshness.
