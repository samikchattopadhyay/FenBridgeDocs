Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (6 Screens)

This proxy wizard facilitates registering spot supply lots for offline farmers, triggering quality visits.

State Operations:
• Submission inserts records in spot_listings, and if pond-sourced, generates corresponding lots records linked as pending_verification.

Key Procedures

  • scout.spotOffer.initProxy — Protected. Fetches assigned farmers list and queries corresponding active pond indices.
  • scout.spotOffer.getAvailability — Protected. Computes available inspection calendar slots based on district calendars and travel distances.
  • scout.spotOffer.submitProxy — Protected. Atomic mutator. Inserts a new row in spot_listings, initializes a child lots row, and logs a scheduled visit_schedules event.

React Component Decomposition (6 Screens)

The Submit Spot Offer workflow breaks into 12 reusable components: 8 shared across Scout module workflows, 4 workflow-specific for spot listing mechanics. Components follow atomic design principles with strict TypeScript interfaces.

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% components shared across 20 workflows
• Variant management: Props define UI variants, not separate components

Database Schemas (Drizzle ORM DDL)

spot_listings (Market Registry Header)

CREATE TABLE spot_listings ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), farmer_id UUID NOT NULL REFERENCES users(id), lot_id UUID NOT NULL REFERENCES lots(id), scout_id UUID REFERENCES users(id), status TEXT NOT NULL DEFAULT 'pending_verification' CHECK (status IN ('pending_verification','listed','reserved','sold','cancelled')), scout_visit_scheduled_at TIMESTAMPTZ NOT NULL, estimated_parameters JSONB NOT NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_spot_listings_farmer ON spot_listings (farmer_id); CREATE INDEX idx_spot_listings_status ON spot_listings (status);

visit_schedules (Audit Verification Booking)

CREATE TABLE visit_schedules ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), farmer_id UUID NOT NULL REFERENCES users(id), scout_id UUID NOT NULL REFERENCES users(id), pond_id UUID REFERENCES ponds(id), scheduled_for TIMESTAMPTZ NOT NULL, purpose TEXT NOT NULL CHECK (purpose IN ('pond_inspection','lot_verification','seal_application')), status TEXT NOT NULL DEFAULT 'scheduled' CHECK (status IN ('scheduled','in_progress','completed','cancelled')), spot_listing_id UUID REFERENCES spot_listings(id) );

Row-Level Security (RLS)

-- Scout can only submit spot offers CREATE POLICY scout_spot_offer_write ON spot_listings FOR INSERT WITH CHECK (scout_id = current_user_id()); -- Visit schedules visible only to the assigned scout CREATE POLICY scout_visit_schedules_scope ON visit_schedules USING (scout_id = current_user_id());

Strict Input Validations & Error Enforcements

  • Volume Bounds: Estimated volumes must be greater than zero. Volume declarations above typical capacities of the chosen pond raise warnings.

Offline Caching & Queue Synchronization

  • Draft Caching: Details cache locally inside SQLite. Sync requests write scheduled inspection slots on server databases if slots are still open.

Security & Media Protection Protocols

  • Zone Enforcement: Only permits submitting offers for farmers registered inside the scout's designated zone bounds.
