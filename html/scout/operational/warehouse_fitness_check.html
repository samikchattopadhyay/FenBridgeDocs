tRPC Procedures (5 Screens)

The Warehouse Fitness Check operates on a session-draft wizard lifecycle before executing an atomic write transactional block.

Procedure Flow Strategy:
• Screen 1 (confirmWarehouseArrival): GPS coordinate matching with standard geo-fencing (50m bounds).
• Screens 2-3 (submitFitnessChecklist, submitFitnessPhotos): Validates and caches checklist parameters and evidence image URLs.
• Screen 4 (submitWarehouseFitnessVerdict): Commits transaction into relational tables.

Key Procedures

  • scout.warehouse.confirmArrival — Protected. Checks scout coordinate distance to the popper warehouse profile. Returns arrival match validation.
  • scout.warehouse.submitFitnessVerdict — Protected. Atomic transaction executing:
    1. Insert warehouse_fitness_checks row (verdict outcome).
    2. Bulk insert warehouse_fitness_items (checklist details).
    3. Bulk insert warehouse_fitness_photos (media references).
    4. Update parent warehouses.fitness_status.

Database Schemas (Drizzle ORM DDL)

warehouse_fitness_checks (Header Table)

CREATE TABLE warehouse_fitness_checks ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES users(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), checked_at TIMESTAMPTZ NOT NULL DEFAULT now(), check_type TEXT NOT NULL CHECK (check_type IN ('initial','periodic')), gps_lat NUMERIC(9,6) NOT NULL, gps_lng NUMERIC(9,6) NOT NULL, gps_accuracy_m NUMERIC(4,1) NOT NULL, humidity_pct NUMERIC(5,2), temperature_c NUMERIC(4,1), cleanliness_score SMALLINT CHECK (cleanliness_score BETWEEN 1 AND 10), pest_control_ok BOOLEAN, verdict TEXT NOT NULL CHECK (verdict IN ('fit','unfit')), notes TEXT ); CREATE INDEX idx_wfc_warehouse ON warehouse_fitness_checks (warehouse_id, checked_at DESC);

warehouse_fitness_items (Checklist Criteria)

CREATE TABLE warehouse_fitness_items ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), fitness_check_id UUID NOT NULL REFERENCES warehouse_fitness_checks(id) ON DELETE CASCADE, criterion_code TEXT NOT NULL CHECK (criterion_code IN ('dry_space','elevated_floor','off_ground_pallets','dampness_check')), result TEXT NOT NULL CHECK (result IN ('pass','fail')), remarks TEXT ); CREATE INDEX idx_wfi_check ON warehouse_fitness_items (fitness_check_id);

warehouse_fitness_photos (Evidence Media)

CREATE TABLE warehouse_fitness_photos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), fitness_check_id UUID NOT NULL REFERENCES warehouse_fitness_checks(id) ON DELETE CASCADE, photo_url TEXT NOT NULL, caption TEXT );

Row-Level Security (RLS)

-- Scouts only access fitness checks for their assigned warehouses CREATE POLICY scout_fitness_check_scope ON warehouse_fitness_checks USING ( warehouse_id IN ( SELECT warehouse_id FROM scout_assignments_warehouses WHERE scout_id = current_user_id() AND active = true ) ); CREATE POLICY scout_fitness_check_write ON warehouse_fitness_checks FOR INSERT WITH CHECK (scout_id = current_user_id());

Strict Input Validations & Error Enforcements

Server-Side Core Rules:
• GPS distance check to the warehouse profile must be ≤ 50 meters.
• All 4 checklist criteria are mandatory. Failing any critical criteria (e.g., dampness_check) forces the overall verdict to 'unfit'.

HTTP Status Code Mapping

StatusError CodeTriggering Event / Condition
403WAREHOUSE_GEO_FENCE_VIOLATIONDistance between current scout coordinates and registered warehouse > 50 meters
400WEAK_GPS_SIGNALScout coordinate lock accuracy > 100 meters (50m ideal; 50–100m triggers warning)
422INCOMPLETE_FITNESS_CHECKLISTMissing results on one or more mandatory checklist items

Offline Caching & Queue Synchronization

  • Offline Cache: Serializes transaction data inside client SQLite sync_queue if the network is absent during submission.
  • Conflict Mitigation: Employs transaction UUIDs on the device client to guarantee idempotency and avoid duplicate submissions.

Security & Media Protection Protocols

  • Scouts only have access to warehouses assigned to their active operational zones.
  • Camera files are compressed client-side (WebP, target <400KB) and uploaded directly to Cloudflare R2 via short-lived presigned POST URLs obtained from scout.media.getUploadUrl.
Warehouse Fitness Steps Directory
5 Steps
1 Confirm Warehouse & Arrival (GPS Check)
Purpose
Ensures the scout is physically present on-site at the target warehouse profile before initiating the check.
Business Requirement
Verifies mobile GPS coordinates within 50 meters of the warehouse location. Accuracy thresholds: ≤50m = ideal (unlocks); 50–100m = warning (proceeds with caution); >100m = error (blocks).
User Experience (UX)

Top header displays target warehouse ID and name.

Interactive maps calculate distance to the warehouse.

A prominent card displays GPS signal strength, unlocking the primary "Lock Location & Start" button when in bounds.

Technical Details

Retrieves geodetic position using navigator API.

Caches GPS coordinate metrics inside local wizard state context.

Components Used
• TopAppBar (header with warehouse ID and name)
• WarehouseHeader / InfoDisplay (warehouse identification)
• InteractiveMap / MapComponent (distance calculation to warehouse)
• GPSSignalCard / DistanceIndicator (distance display with signal strength)
• GPSAccuracyDisplay (accuracy threshold indicators - ideal/warning/error)
• StatusBadge (GPS lock status indicator)
• Button (Lock Location & Start - enabled when in bounds)
• CTAFooter (action bar with primary button)
2 Fitness Checklist
Purpose
Evaluates essential technical and hygienic parameters at the warehouse.
Business Requirement

Mandatory logging of 4 core criteria: Dry Space, Elevated Floor, Off-Ground Pallets, and Dampness Check.

Failing any item mandates an "Unfit" final verdict.

User Experience (UX)

Renders large, touch-friendly option rows.

Each criterion has clear "Pass" / "Fail" selectors.

Failing critical items automatically highlights warning messages alerting that the warehouse will be graded unfit.

Technical Details

Zod validation ensures all checklist rows are resolved before allowing page advancement.

Stores selections in session cache.

Components Used
• TopAppBar (header with step 2 of 5 indicator)
• ChecklistCard (4 core criteria cards - Dry Space, Elevated Floor, Off-Ground Pallets, Dampness Check)
• RadioButton / SelectInput (Pass/Fail option selectors for each criterion)
• WarningBanner / AlertBanner (warnings for failing critical items)
• StatusDisplay (unfit verdict indicator when any criterion fails)
• Button (Next to proceed)
• CTAFooter (action bar with Next button)
3 Evidence Photos
Purpose
Captures photographic verification documenting warehouse conditions.
Business Requirement

Requires at least 2 photos (General interior layout, close-up of ventilation or flooring).

Images are tagged with caption notes.

User Experience (UX)

Grid list displaying camera triggers.

Progressive loaders track uploading states.

Optional input boxes allow scouts to write captions per image.

Technical Details
Uploads compressed JPEGs (max 2MB) to secure Supabase buckets, caching signed URLs locally inside wizard draft state.
Components Used
• TopAppBar (header with step 3 of 5 progress)
• PhotoGrid (grid list with camera trigger slots)
• PhotoCaptureCard (camera launch buttons for interior and details)
• ProgressLoader / UploadProgress (upload state indicators and loading rings)
• TextInput / TextArea (optional caption notes per image)
• PhotoPreviewCard (image preview replacing camera button after capture)
• Button (Next to proceed)
• CTAFooter (action bar with Next button)
4 Verdict & Submit
Purpose
Reviews all criteria assessments and commits the final verdict.
Business Requirement

Renders calculated final verdict ('Fit' or 'Unfit').

Unfit verdicts must log mandatory reasoning comments.

Executes atomic multi-table inserts.

User Experience (UX)

Detailed review panel.

Calculated status is displayed in large pills (Green checkmark for "Fit", Amber warning for "Unfit").

Notes text area is required for unfit outcomes.

High-contrast primary cyan button reads "Submit Verification".

Technical Details

Invokes submitWarehouseFitnessVerdict procedure.

Automatically updates the parent warehouses.fitness_status column.

On database failure, rolls back all updates.

Components Used
• TopAppBar (header with step 4 of 5 progress)
• ReviewPanel / SummaryCard (detailed criteria assessment review)
• VerdictCard (calculated final verdict display)
• StatusBadge (Fit = green checkmark, Unfit = amber warning)
• TextInput / TextArea (mandatory notes field for unfit verdicts)
• CriteriaDisplay (read-only checklist results review)
• Button (Submit Verification primary action)
• CTAFooter (action bar with Submit button)
5 Submission Confirmation
Purpose
Confirmation page indicating completion, showing local sync queue status.
Business Requirement

Generates and displays unique audit check code (e.g. #WFC-1882) and confirms local database task completions.

User Experience (UX)

Full-screen glassmorphism layout with centered, pulsing green checkmark.

Displays sync state badge ("Synced" or "Offline Queued").

Primary solid cyan CTA "View Today's Tasks" returns scout back to list.

Technical Details

Clears all temporary draft context variables.

Switches local task queue state to completed.

Components Used
• TopAppBar (header with completion state)
• SuccessCard (full-screen glassmorphism layout)
• CheckmarkIcon (large, pulsing green checkmark animation)
• AuditCodeDisplay (unique verification code e.g., #WFC-1882)
• SyncStatusBadge (Synced or Offline Queued status)
• CompletionMessage (confirmation text and sync state details)
• Button (View Today's Tasks primary action)
• CTAFooter (action bar with return button)
