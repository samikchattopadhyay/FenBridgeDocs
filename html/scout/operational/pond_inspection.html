Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (6 Screens)

The Pond Inspection workflow is fueled by a session-context state router that processes sequential forms before committing to the relational backend.

Procedure Flow Strategy:
• Screen 1 (confirmPondArrival): Anti-fraud GPS check comparing current scout mobile telemetry to registered pond boundaries.
• Screens 2-4 (submitPondConditions, submitBundReadings, submitInspectionPhotos): Progressive form cache validations.
• Screen 5 (submitPondInspection): Atomic transaction writing the entire multi-table record.

Key Procedures

  • confirmPondArrival — Protected. Validates scout coordinates within a strict 50m geo-fence of the target pond profile. Saves verification event to audit logs.
  • submitPondInspection — Protected. Atomic procedure that registers:
    1. Main pond_inspections row.
    2. Granular pond_bund_readings details array (Lines 1 to 4).
    3. S3 URL media logs in pond_inspection_photos.
    On transaction success, updates cached task statuses and invokes downstream state updates.

Database Schemas (Drizzle ORM DDL)

pond_inspections (Header Table)

CREATE TABLE pond_inspections ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES scouts(id), pond_id UUID NOT NULL REFERENCES ponds(id), checked_at TIMESTAMPTZ NOT NULL DEFAULT now(), water_level_m NUMERIC(4,2) NOT NULL, algae_density_score INT NOT NULL CHECK (algae_density_score BETWEEN 1 AND 5), readiness_score INT NOT NULL CHECK (readiness_score BETWEEN 0 AND 100), observed_defects_json JSONB, gps_lat NUMERIC(9,6) NOT NULL, gps_lng NUMERIC(9,6) NOT NULL, gps_accuracy_m NUMERIC(4,1) NOT NULL, check_type TEXT NOT NULL DEFAULT 'periodic' CHECK (check_type IN ('initial','periodic')), notes TEXT ); CREATE INDEX idx_poin_pond ON pond_inspections (pond_id); CREATE INDEX idx_poin_scout ON pond_inspections (scout_id);

pond_bund_readings (Line Items)

CREATE TABLE pond_bund_readings ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), pond_inspection_id UUID NOT NULL REFERENCES pond_inspections(id) ON DELETE CASCADE, bund_line_number INT NOT NULL CHECK (bund_line_number BETWEEN 1 AND 4), density_count_m2 INT NOT NULL, maturity_percentage NUMERIC(5,2) NOT NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_pobre_inspection ON pond_bund_readings (pond_inspection_id);

pond_inspection_photos (Evidence Media)

CREATE TABLE pond_inspection_photos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), pond_inspection_id UUID NOT NULL REFERENCES pond_inspections(id) ON DELETE CASCADE, photo_url TEXT NOT NULL, photo_type TEXT NOT NULL CHECK (photo_type IN ('wide_view','sample_view','anomaly')), caption TEXT, created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

Strict Input Validations & Error Enforcements

Zod Validation Schemas

Server-Side Core Rules:
• GPS coordinates: latitude [-90, 90], longitude [-180, 180]. Accuracy must be < 15 meters.
• Water Level: positive numeric float representing depth in meters (e.g. 0.05m to 5.00m).
• Algae density rating: strictly bounded integer from 1 to 5.
• Bund Matrix: density count ≥ 0 per m², and biological maturity float bounded between [0.00, 100.00].

Offline Caching & Queue Synchronization

Offline-First Strategy (SQLite Sync Queue)

If the scout is operating in remote rural areas without reliable internet:

1. Save the entire validated payload locally to the device SQL database inside the sync_queue table as a serialized JSON object.
2. Mark the local task record status as syncing_offline.
3. On network recovery, trigger a background service upload.

Optimistic UI & Conflict Prevention

  • Optimistic State Updates: Reflect the task as "Completed" on the scout's local list view immediately.
  • Concurrency Token: Include a unique generated timestamp UUID on each client payload to prevent redundant transactional inserts.

Security & Media Protection Protocols

GPS Telemetry Integrity

Verification Audit logs: Scout's exact coordinates locked during Screen 1 are stored to prevent mock/spoofed GPS injection attacks. System audits verified distances to trace spoofing activities.

Supabase S3 Signed Media Uploads

  • Photos are captured natively and processed into compressed JPEG objects (max 2MB).
  • Uploaded to secure Supabase S3 buckets using short-lived pre-signed POST URLs generated per session.
  • Public bucket access is disabled; all preview displays resolve expiring URLs (24-hour lifetime).
Pond Inspection Steps Directory
6 Steps
1 Confirm Pond Arrival (GPS Check)
Purpose
Ensures the scout is physically present on-site at the target pond coordinates before allowing any quality or biological data entry (anti-fraud protection).
Business Requirement

Validates live mobile coordinates against the pond's registered location.

Geofence radius must be within 50 meters, with GPS accuracy locked below 15 meters.

User Experience (UX)

Top navigation maps standard return buttons and the pond ID identifier.

Displays an active distance-to-target radar card displaying distance in meters and signal accuracy.

An interactive map visualizes the scout's path, with a locked solid cyan CTA: "Confirm Arrival" that unlocks only when the scout is within range.

User Actions
  • Enable location permissions on the device if requested.
  • Walk toward the pond until the radar card signals a green connection status.
  • Tap the "Confirm Arrival" button to lock coordinate values and advance.
Technical Details

Queries the browser GPS API to retrieve latitude, longitude, and accuracy.

Computes geodetic distance.

On locking, the procedure caches current GPS parameters in local draft state.

2 Assess Pond Conditions
Purpose
Logs core physical and biological parameters representing ecological stability and quality indexes.
Business Requirement

Mandatory logging of current water level depth (meters) and algae density ratings (1 to 5 index, representing bio-load levels).

Check boxes for active weed/pest alerts must be saved.

User Experience (UX)

Includes numeric input elements for water depth.

Biological ratings are captured via an interactive sliding selector scale (1-5) mapping to dynamic colored label indicators (e.g.

1 = Clear, 3 = Moderate, 5 = Eutrophic).

Toggles configure active weed/pest flags.

User Actions
  • Input water level using plus/minus steppers or direct keyboard entry.
  • Drag the algae scale indicator to set the biological load rating.
  • Toggle pest or weed alerts if contamination is present. Tap "Next".
Technical Details

Validates parameters using standard Zod constraints.

Water level must be positive.

Algae score must be an integer between 1 and 5.

Caches values locally.

3 Bund & Edge Readings
Purpose
Logs biological makhana maturity and plant densities across multiple physical bund coordinates to evaluate harvest readiness.
Business Requirement

Must record granular details (Count/m² and Maturity %) for four separate bund lines (Lines 1 to 4).

Dynamically calculates overall averages for compliance logs.

User Experience (UX)

Renders a compact, high-density matrix grid table.

Columns track Bund Line, Density (Count per m²), and Maturity %.

A highlighted summary row at the bottom dynamically calculates and displays overall averages using a bold, bright cyan accent color.

Row checkmarks lock completed coordinates.

User Actions
  • Input numeric density counts and maturity values for each bund line row.
  • Confirm checked status switches per line to lock entries.
  • Verify live computed averages on the bottom row. Tap "Proceed".
Technical Details

Local JavaScript binds inputs and triggers direct mathematical average updates.

Validates that density is positive and maturity is between [0.00%, 100.00%].

Compiles into a child array mapped to pond_bund_readings schema.

4 Pond Photos & Notes
Purpose
Captures visual verification evidence and logs descriptive comments regarding active biological anomalies or structural highlights.
Business Requirement

Requires at least 3 photos (Wide view, close-up sample, biological highlight).

Allows adding optional note descriptions for administrative reviews.

User Experience (UX)

Renders an interactive photo upload deck displaying camera icons and placeholders.

Once uploaded, cards display progressive upload loading rings.

A large, clean notes field includes character-count limits.

User Actions
  • Tap empty photo slots to launch device camera or file selector.
  • Classify each captured photo (Wide view, Sample view, anomaly).
  • Type optional inspection notes inside the text input area. Tap "Review".
Technical Details

Compresses image uploads to max 2MB JPEGs.

Uploads assets to secure Supabase S3 buckets using short-lived pre-signed URLs.

Binds S3 links inside pond_inspection_photos.

5 Review & Submit
Purpose
Final verification check for all compiled physical, chemical, and biological measurements prior to committing the record to the platform databases.
Business Requirement

Must present complete summary cards tracking GPS validation, physical scores, the bund average matrix, and uploaded thumbnails.

Safe validation required before commit.

User Experience (UX)

A vertical scrollable review sheet.

Categorized panels group Profile coordinates, water level and algae indicators, and a read-only bund grid.

High-visibility bottom action bar houses the solid cyan "Submit Inspection" button.

User Actions
  • Review each diagnostic section carefully.
  • Tap inline "Edit" links next to any section to correct entries if needed.
  • Tap the "Submit Inspection" button to write records permanently.
Technical Details

Invokes the submitPondInspection tRPC procedure.

Executes a database transaction to insert into pond_inspections, pond_bund_readings, and pond_inspection_photos simultaneously.

Automatically rolls back all updates if a single block fails.

6 Inspection Confirmation
Purpose
Provides immediate confirmation that the record was registered successfully, highlighting offline/online synchronization states.
Business Requirement

Must generate and display a unique transaction reference code (e.g.

#INS-9942) and update local task queues dynamically.

User Experience (UX)

Premium glassmorphism card layout.

Features a large, glowing, pulsing vector green checkmark ring at the center.

Displays the unique inspection code and sync status details ("Synced with ledger" or "Stored in local queue").

Solid cyan primary CTA "View Today's Tasks" stands alongside outline button option.

User Actions
  • Note or copy the generated inspection code reference.
  • Verify active network sync state badge indicates no errors.
  • Tap the primary button to return to the active task list.
Technical Details

Resets all local state variables.

Updates local cached task list models, switching the target task's state flag to completed.

Clears draft storage files.
