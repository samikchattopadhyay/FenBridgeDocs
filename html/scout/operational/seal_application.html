tRPC Procedures (6 Screens)

Tamper-evident Sealing locks down the inspected makhana bags into the platform ledger via pre-defined rules.

Critical Precondition Guards:
• Guard 1: The lot's latest inspection outcome must be accepted, or downgrade with lots.downgrade_consent_status = 'consented'.
• Guard 2: The host warehouse fitness check status must be fit.

Key Procedures

  • scout.seal.initApplication — Protected. Fetches lot details, latest inspection findings, downgrade consent statuses, and geofence parameters.
  • scout.seal.submitRecord — Protected. Atomic multi-table write committing lot_seals, bulk lot_seal_photos, bulk lot_seal_storage_items, and updating the parent lots status and GC seal reference.
  • scout.media.getUploadUrl — Protected. Issues a short-lived Cloudflare R2 presigned POST URL for direct photo upload from the mobile device. Returns { uploadUrl, fileKey }.
  • scout.media.confirmUpload — Protected. Registers the uploaded file's R2 key and content hash into lot_seal_photos after client confirms successful upload.

Database Schemas (Drizzle ORM DDL)

lot_seals (Primary Header)

CREATE TABLE lot_seals ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), seal_id TEXT UNIQUE NOT NULL, -- physical seal barcode number lot_id UUID NOT NULL REFERENCES lots(id), inspection_id UUID NOT NULL REFERENCES inspections(id), cert_id UUID REFERENCES grade_certificates(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), popper_id UUID NOT NULL REFERENCES users(id), scout_id UUID NOT NULL REFERENCES users(id), task_id UUID REFERENCES scout_tasks(id), net_applied BOOLEAN NOT NULL DEFAULT true, sealed_lat NUMERIC(9,6) NOT NULL, sealed_lng NUMERIC(9,6) NOT NULL, sealed_accuracy_m NUMERIC(6,1), sealed_at TIMESTAMPTZ NOT NULL, shelf_life_days INT NOT NULL CHECK (shelf_life_days IN (30,45)), expires_at DATE NOT NULL, -- sealed_at + shelf_life_days status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','expired','broken','voided')), created_at TIMESTAMPTZ NOT NULL DEFAULT now(), synced_at TIMESTAMPTZ ); CREATE INDEX idx_lotseals_lot ON lot_seals (lot_id); CREATE INDEX idx_lotseals_status ON lot_seals (status, expires_at);

lot_seal_photos (Evidence Media)

CREATE TABLE lot_seal_photos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_seal_id UUID NOT NULL REFERENCES lot_seals(id) ON DELETE CASCADE, slot_type TEXT NOT NULL, -- 'seal_closeup'|'bags_net'|'warehouse_wide'|'extra' storage_path TEXT NOT NULL, captured_at TIMESTAMPTZ NOT NULL );

Row-Level Security (RLS)

-- Only assigned scouts can insert seal records for their warehouses CREATE POLICY scout_seal_own_write ON lot_seals FOR INSERT WITH CHECK ( scout_id = current_user_id() AND warehouse_id IN ( SELECT warehouse_id FROM scout_assignments_warehouses WHERE scout_id = current_user_id() AND active = true ) ); -- Block double-sealing at DB level CREATE UNIQUE INDEX idx_lotseals_active_lot ON lot_seals (lot_id) WHERE status = 'active';

Strict Input Validations & Error Enforcements

  • Seal ID Formatting: Validates against barcode structure formats. Local duplicate checks run immediately.
  • Geofence Limits: Verifies coordinates against registered warehouse locations. Triggers amber warnings if distance > 200m.

Offline Caching & Queue Synchronization

  • Offline-First Scan: Local decoders scan barcodes offline. Commits submit atomically to client SQLite caches.
  • Traceability Reconciliation: On network synchronization, the backend reconciles the new seal reference, triggers server-side PDF regeneration of the grade_certificates, and enqueues the lot into admin pricing list channels.

Security & Media Protection Protocols

  • Guard Rules: Double sealing blocks are strictly enforced. If a lot already contains an active seal, subsequent attempts are blocked.
  • RLS Bounds: Asserts that only assigned scouts can finalize seal registers.
Seal Application Steps Directory
6 Steps
1 Confirm Verified Lot
Purpose
Verifies that the target lot meets all precondition quality standards and is ready for seal placement.
Business Requirement

Guards against unauthorized sealing.

Validates that host warehouse is rated FIT and that popper consent is recorded for downgrades.

User Experience (UX)

Read-only info sheet.

Displays lot identification, assessed grade, and verified weights.

Renders green validation alerts for eligible lots and secure explanations for blocked candidates.

Primary CTA reads "Begin Sealing".

Technical Details
Caches parent inspection_id and cert_id into the sealing context state.
2 Apply Seal & Net
Purpose
Logs the unique physical seal number applied to the lot bags along with on-site GPS verification.
Business Requirement

Captures unique seal barcode reference via scan.

Requires mandatory confirmation check marking that the reusable net was applied.

Logs GPS metadata.

User Experience (UX)

Top actions display scan triggers.

Succesful scans populate alphanumeric seal IDs in monospace font.

Renders mandatory net confirmation toggle.

Displays GPS accuracy locks.

Primary cyan button unlocks on completion.

Technical Details

Validates barcode formatting schemas.

Triggers warnings if GPS coordinate mismatches register.

3 Seal Evidence Photos
Purpose
Captures photographic logs documenting applied seal placement.
Business Requirement
Requires three mandatory photographs: Close-up of the applied seal (ID numbers must be legible), bags with net applied, and a wide-angle warehouse shot.
User Experience (UX)

Standard grid checklist displaying dashed camera buttons.

Previews replace buttons on capture.

Helpful indicators alert scouts that characters on the seal must remain clear.

Technical Details
Compiles photographs into lot_seal_photos mappings in temporary memory.
4 Storage Check at Seal Time
Purpose
Performs a lightweight verification confirming correct placement of the specific lot in the warehouse.
Business Requirement

Logs three parameters: Off the floor platform support, moisture barrier lining, and physical segregation from loose stock.

Fails do not block sealing but alert admin channels.

User Experience (UX)

Three large checkbox cards with Pass/Fail toggles (unset by default).

Failing parameters displays reason list pickers.

Yellow warning bars alert scouts to coordinate quick corrections on failures.

Technical Details

Maintains checks in lot_seal_storage_items formats.

Raises storage_flag = true on any failed item.

5 Shelf-life & Confirm
Purpose
Specifies quality re-inspection durations and commits sealing registers atomically.
Business Requirement
Selects shelf-life duration: 30 days (default) or 45 days. Auto-calculates expires_at date bounds. Requires mandatory checklist signature checks.
User Experience (UX)

Top options render segmented controls picking 30/45 days, updating expiry dates live.

Summary cards show all accumulated metadata.

Prompts signature confirmation checkboxes.

Solid cyan button reads "Submit Seal".

Technical Details

Invokes submitSealRecord database transactions.

Mutates parent status to sealed, linking seal identifiers.

6 Submission Confirmation
Purpose
Confirms successful ledger commitment and guides scouts onward.
Business Requirement

Verifies data persistence and echoes sync queue state flags.

Details onward admin pricing pipelines.

User Experience (UX)

Success modal with pulsing lock icons.

Displays generated seal codes and expiry schedules.

Cyan CTA redirects back to today dashboard list.

Technical Details

Resets temporary memory buffers.

Switches local SQLite queue loaders to completed.
