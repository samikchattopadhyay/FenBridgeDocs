tRPC Procedures (5 Screens)

The Storage Re-check workflow evaluates active tamper-evident lots, routing deteriorated items out of the market pipeline.

Precondition Guard:
• The lot seal status must be exactly active. Sold, voided, or terminally rejected lots block the audit wizard.

Key Procedures

  • scout.storage.initRecheck — Fetches active lot_seals, details remaining shelf-life, and queries historical quality logs.
  • scout.storage.submitRecheckOutcome — Atomic transaction executing:
    1. Inserts storage_rechecks row (status check details).
    2. Conditionally inserts warehouse_storage_failures if outcome is non-clean.
    3. Mutates lot_seals.status (broken, voided, etc.) and updates parent lots.status.

Database Schemas (Drizzle ORM DDL)

storage_rechecks (Primary Header)

CREATE TABLE storage_rechecks ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_seal_id UUID NOT NULL REFERENCES lot_seals(id), lot_id UUID NOT NULL REFERENCES lots(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), popper_id UUID NOT NULL REFERENCES users(id), scout_id UUID NOT NULL REFERENCES users(id), task_id UUID REFERENCES scout_tasks(id), recheck_trigger TEXT NOT NULL CHECK (recheck_trigger IN ('half_life','anomaly_report')), seal_id_match BOOLEAN NOT NULL, seal_integrity TEXT NOT NULL CHECK (seal_integrity IN ('intact','broken')), seal_reason_code TEXT, net_integrity TEXT NOT NULL CHECK (net_integrity IN ('intact','broken')), net_reason_code TEXT, outcome TEXT NOT NULL CHECK (outcome IN ('clean','void','route_reinspection','terminal')), overall_note TEXT, arrival_lat NUMERIC(9,6) NOT NULL, arrival_lng NUMERIC(9,6) NOT NULL, arrival_accuracy_m NUMERIC(6,1), checked_at TIMESTAMPTZ NOT NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT now(), synced_at TIMESTAMPTZ ); CREATE INDEX idx_srk_lot ON storage_rechecks (lot_id, checked_at DESC); CREATE INDEX idx_srk_seal ON storage_rechecks (lot_seal_id);

warehouse_storage_failures (Triage Mappings)

CREATE TABLE warehouse_storage_failures ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), warehouse_id UUID NOT NULL REFERENCES warehouses(id), storage_recheck_id UUID NOT NULL REFERENCES storage_rechecks(id), lot_id UUID NOT NULL REFERENCES lots(id), failure_outcome TEXT NOT NULL, -- 'void'|'route_reinspection'|'terminal' created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

Row-Level Security (RLS)

-- Scouts only access storage rechecks for their assigned warehouses CREATE POLICY scout_storage_recheck_scope ON storage_rechecks USING ( warehouse_id IN ( SELECT warehouse_id FROM scout_assignments_warehouses WHERE scout_id = current_user_id() AND active = true ) ); CREATE POLICY scout_storage_recheck_write ON storage_rechecks FOR INSERT WITH CHECK (scout_id = current_user_id());

Strict Input Validations & Error Enforcements

  • Outcome Dependency: If seal integrity is marked broken, the clean outcome option is hard-blocked client-side.
  • Recheck Reason Codes: Broken locks/nets require mandatory coded explanation inputs before continuing.

Offline Caching & Queue Synchronization

  • Offline Lock: All parameters save to local SQL drafts. Enqueues atomic submission records.
  • Pipeline Workflows: On syncing non-clean outcomes, the server triggers instant listing suspensions, notifies admin, and dynamically schedules reinspections or logistics returns.

Security & Media Protection Protocols

  • Coordinate Locks: Scout arrival location is recorded at initialization to verify presence at popper warehouses.
  • Immutability: Submitted recheck details are finalized and read-only. Corrections require new sequential audits.
Storage Re-Check Steps Directory
5 Steps
1 Confirm Sealed Lot
Purpose
Identifies the target sealed lot and outlines remaining shelf-life schedules.
Business Requirement

Guards against checking invalid lots.

Asserts that the lot seal status is currently active.

Notes the recheck trigger (Routine half-life or anomaly report).

User Experience (UX)

Displays lot identification, stored seals, and assessed grades.

Shows remaining shelf-life days countdown.

Details trigger alerts.

Main button reads "Begin Re-check".

Technical Details
Initializes wizard context parameters in device cache, preparing integrity indicators.
2 Seal & Net Integrity
Purpose
Audits the physical state of the applied lock and net to identify tampering or breakdown.
Business Requirement

Mandatory logging of: Seal ID match (Pass/Fail), Seal integrity (Intact/Broken), and Net integrity (Intact/Broken).

Coded explanations are required on failures.

User Experience (UX)

Three checkbox groups with Pass/Fail segments.

Failures open nested selectors displaying failure codes (e.g.

cut net, broken seal loop).

Amber alerts warn that fails block clean classifications.

Technical Details
Checks matches and writes integrity outcomes into wizard local states.
3 Storage Conditions
Purpose
Audits current environmental and placement qualities of the host warehouse since the previous check.
Business Requirement

Inspects floor dampness, humidity indicators, ventilation, and pest signs.

Failures trigger warnings suggesting reinspection routes.

User Experience (UX)

Touch-friendly check rows listing key points (Dampness, pests, space).

Fails reveal notes boxes enabling short descriptions.

Dynamic warning bars display triage guidance.

Technical Details
Validates parameters to ensure all checklists are fully resolved before allowing page advancement.
4 Outcome & Submit
Purpose
Selects the authoritative outcome classification and commits re-check records to database tables.
Business Requirement

Outcomes: Clean (Lot continues listed), Void (Seal broken/tampered, listing suspended), Route Reinspection (Environmental moisture failure), or Terminal (Severe pest/rot defects).

Atomic writes run on submit.

User Experience (UX)

Recaps integrity audits and coordinates.

Renders segmented buttons selecting the outcome (Fails restrict Clean).

Prompts validation signature checkbox.

Solid cyan button reads "Submit Re-check".

Technical Details
Invokes submitStorageRecheckOutcome, updating lot_seals status and conditional failure mapping routes.
5 Submission Confirmation
Purpose
Confirms completion of the audit check and details next steps.
Business Requirement
Displays sync queue badges and warns if admin channels were alerted regarding suspended lots.
User Experience (UX)

Centered glassmorphic modal with success badges.

Echoes selected outcomes in color-coded pills.

Primary solid cyan button returns back to dashboard lists.

Technical Details

Resets wizard parameters.

Marks SQLite re-check tasks completed.
