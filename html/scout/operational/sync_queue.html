Workflow Module 7

Sync Queue

Offline data sync management. Lists locally captured items awaiting upload (Inspection/Verification/Seal/Registration). Tracks sync state (Waiting/Syncing/Synced/Conflict) with conflict resolution screen for mismatches.

Sync Queue Screen

List View
Chronological list of locally captured items awaiting upload.
Row Information
  • Item type icon (Inspection/Verification/Seal/Registration)
  • Entity reference (lot ID, farmer name, etc.)
  • Capture time (e.g., "2 hours ago")
  • Photo count + form size (e.g., "4 photos, 2.3MB")
  • State badge (Waiting/Syncing/Synced/Conflict)
State Definitions
Each item tracks its upload status.
Item States
  • Waiting: Captured locally, not yet synced
  • Syncing: In progress upload
  • Synced: Successfully uploaded to server
  • Conflict: Upload conflict detected (local vs server mismatch)
User Actions
Scout can manage sync workflow.
Available Actions
  • Sync Now button (top) â€” force immediate upload of all waiting items
  • Tap conflicted item â†’ resolution screen (see next)
  • Pull-to-refresh for manual refresh

Conflict Resolution Screen

Conflict Detection
When sync detects a mismatch between local and server records.
Conflict Scenarios
  • Item was modified on server after local capture
  • Server-side deletion (item no longer exists on server)
  • Field value mismatch (e.g., different lot status)
Resolution Options
Scout resolves conflict by choosing authoritative source.
Choices
  • Keep Local: Local version is correct; overwrite server
  • Use Server: Server version is authoritative; discard local
  • Escalate to Admin: Unclear; request Admin manual review
Conflict Flow
Scout sees both versions side-by-side for comparison.
Comparison
  • Local version (what scout captured)
  • Server version (what's on backend)
  • Highlighted differences (red/green for changed fields)
  • Scout chooses which to keep

Offline-First Architecture

Capture Pattern
All scout workflows are offline-first by design.
Workflow Sequence
  • Scout captures (Inspection/Verification/Seal/Registration) entirely offline
  • Data + photos stored locally in app database
  • Record added to sync_queue automatically
  • When network available, sync_queue uploads all items
Photo Handling
Photos compressed during capture for sync efficiency.
Compression
  • Each photo compressed to â‰¤1.5MB at capture time
  • Compression happens locally (no wait for network)
  • Original full-res stored locally; compressed version synced
Sync Trigger
Scout controls when data is uploaded.
Options
  • Manual: Scout taps "Sync Now" from Sync Queue or Today Dashboard
  • Automatic (future): Background sync when network available
  • No forced sync; scout works offline all day if needed

Data Integrity & Immutability

Immutable Records
Once synced, records are immutable by scout.
After Sync
  • Scout cannot edit synced record
  • Admin can correct via backend only
  • Audit trail preserved (timestamps, scout ID)
Version Control
Tracks both local and server versions.
Tracking
  • Local version has capture timestamp + scout ID
  • Server version has server-side timestamp
  • Conflict shows both with edit history (if available)
