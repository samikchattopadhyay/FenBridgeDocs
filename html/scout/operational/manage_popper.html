Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (5 Screens)

This dashboard manages offline popper profiles, document indices, and warehouse parameters.

State Operations:
• Basic edits persist directly. Document changes and warehouse deactivations require logging historical audits.

Key Procedures

  • updatePopperProfile — Protected. Mutates identity details.
  • updatePopperDocuments — Protected. Saves GST details and resets verified statuses to pending.
  • upsertWarehouseAsset — Protected. Saves warehouse metadata or appends new coordinates.
  • toggleWarehouseStatus — Protected. Sets warehouse state.

React Component Decomposition (5 Screens)

The Manage Popper workflow breaks into 11 reusable components: 8 shared across Scout module workflows, 3 workflow-specific for popper account management. Components follow atomic design principles with strict TypeScript interfaces.

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% components shared across 20 workflows
• Variant management: Props define UI variants, not separate components

Screen-by-Screen Component Breakdown

Screen 1: Edit Identity Profile
Purpose: Modify popper basic profile data (name, phone, address)
Layout: TopAppBar → ProfileHeaderCard → TextInput/AddressForm → VerificationBadges → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Edit Identity Profile", menu
• ProfileHeaderCard (workflow-specific) — Shows popper name, current status, registration date
• TextInput (shared) — "Business Name" with validation
• PhoneInput (shared) — "+91 Mobile" with read-only flag if verified
• AddressForm (shared) — Street, district, panchayat fields
• VerificationBadges (workflow-specific) — Inline badges showing phone/email verified status
• CTAFooter (shared) — "Update Profile" (enabled on change), "Cancel"
State Management: Profile edits in local component state, submitted via updatePopperProfile()
Validation: Name length 3-100, valid phone format, address completeness
Screen 2: Update Business Documents
Purpose: Replace or update GST, trade licenses, tax forms
Layout: TopAppBar → DocumentCardsGrid → InfoBanner → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Business Documents", menu
• DocumentStatusCard (workflow-specific) — Card for each doc type (GST, License, Tax) with:
  ├ Status badge (Verified/Under Review/Missing)
  ├ TextInput (shared) — Document number
  ├ Document upload camera trigger
  └ Expiry date if applicable
• InfoBanner (shared) — "Document updates require re-verification"
• CTAFooter (shared) — "Submit Documents", "Cancel"
State Management: Document fields + file URIs in local state, submitted via updatePopperDocuments()
Validation: GST valid format (15 chars), doc number non-empty, file upload required
Side Effect: Upon submit, all document verification states reset to 'pending'
Screen 3: Edit Warehouse Details
Purpose: Modify existing warehouse parameters (capacity, location, storage type)
Layout: TopAppBar → WarehouseSelectionCard → WarehouseEditForm → WarningBanner → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Edit Warehouse Details", menu
• WarehouseSelectionCard (workflow-specific) — Grid of existing warehouses, single-select with capacity display
• WarehouseEditForm (workflow-specific) — Form for selected warehouse:
  ├ TextInput (shared) — "Warehouse Name"
  ├ NumberInput (shared) — "Capacity (MT)" with validation
  ├ SelectInput (shared) — "Storage Type" (open, covered, refrigerated)
  ├ LocationPicker (shared) — GPS coordinates capture
  └ TextInput (shared) — "Landmark/Address"
• WarningBanner (shared) — Shows if warehouse has active lots: "Active storage in progress. Capacity changes will affect ongoing operations."
• CTAFooter (shared) — "Update Warehouse" (disabled if lots active + capacity changed), "Cancel"
State Management: Selected warehouse_id + form edits in local state
API Call: Requires getWarehouseStatus(warehouse_id) to check active lots before allowing capacity edits
Validation: Capacity > 0, name length 3-50, valid GPS coordinates
Screen 4: Add New Warehouse Asset
Purpose: Register new warehouse to popper profile
Layout: TopAppBar → WarehouseRegistrationForm → WarehouseReviewCard → ConfirmationCheckbox → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Add New Warehouse", menu
• WarehouseRegistrationForm (workflow-specific) — Form inputs for new warehouse:
  ├ TextInput (shared) — "Warehouse Name" (required)
  ├ NumberInput (shared) — "Capacity (MT)" with validation
  ├ SelectInput (shared) — "Storage Type" (open, covered, refrigerated)
  ├ SelectInput (shared) — "District" from geo-hierarchy
  ├ TextInput (shared) — "Village/Panchayat"
  ├ LocationPicker (shared) — GPS capture (mandatory, locks position on capture)
  └ TextInput (shared) — "Landmark"
• WarehouseReviewCard (workflow-specific) — Read-only recap: name, capacity, location, storage type
• ConfirmationCheckbox (shared) — "I confirm warehouse coordinates are accurate"
• CTAFooter (shared) — "Submit Warehouse" (enabled on checkbox), "Back"
State Management: Form fields in local state
API Call: submitWarehouse() generates unique warehouse_code, sets status to 'pending_approval'
Validation: All fields required, capacity > 0, GPS coordinates must differ from existing warehouses
Screen 5: Deactivate Warehouse Asset
Purpose: Suspend or permanently disable a warehouse from active use
Layout: TopAppBar → WarehouseSelectionCard → WarningBanner → DeactivationReasonCard → ConfirmationCheckbox → CTAFooter
Components (by hierarchy):
• TopAppBar (shared) — Header with back, "Deactivate Warehouse", menu
• WarehouseSelectionCard (workflow-specific) — Grid of warehouses, single-select for deactivation
• WarningBanner (shared) — Red critical banner: "Deactivation is irreversible. All pending allocations will be cancelled."
• ActiveAllocationsCard (workflow-specific) — Shows count of active lots if any: "2 active lots will be reassigned"
• DeactivationReasonCard (workflow-specific) — SelectInput with reasons:
  ├ "Lease Termination"
  ├ "Structural Damage"
  ├ "Safety Hazard"
  ├ "Regulatory Non-Compliance"
  └ "Other" (with TextInput for notes)
• ConfirmationCheckbox (shared) — "I confirm deactivation of [warehouse_name]"
• CTAFooter (shared) — "Deactivate Warehouse" (dark amber, enabled on checkbox), "Cancel"
State Management: Selected warehouse_id + deactivation_reason in local state
API Call: toggleWarehouseStatus(warehouse_id, 'inactive', reason)
Validation: Reason must be selected if "Other" then notes required

Cross-Workflow Component Reuse Pattern

⚠️ CRITICAL: Zero Duplication Rule

Already Shared Across Multiple Workflows:
• TextInput, PhoneInput, NumberInput, SelectInput — All shared. DO NOT recreate.
• AddressForm — Reused from Farmer/Popper Registration & other management. DO NOT duplicate.
• LocationPicker — Shared across 8+ workflows. DO NOT recreate.
• StatusBadge — Reused across all list screens. DO NOT create variant-specific badges.
• WarningBanner, InfoBanner — Shared. DO NOT create new banner components.
• ConfirmationCheckbox, CTAFooter, TopAppBar — All shared. DO NOT recreate.

This Workflow's New Components:
Only 3 components are UNIQUE to Manage Popper:
1. ProfileHeaderCard — Displays popper profile summary (not farmer profile)
2. DocumentStatusCard — Variant of status display for document types (could extend StatusBadge)
3. WarehouseSelectionCard — Warehouse grid selector (distinct from FarmerSelectionCard/POSelectionCard)
4. WarehouseEditForm — Form wrapper for warehouse edits
5. WarehouseReviewCard — Read-only recap (similar to other recap cards but warehouse-specific)
6. DeactivationReasonCard — Reason selector with "Other" notes field
7. ActiveAllocationsCard — Shows active lots count (could be generic warning card variant)

DO's:
✓ Check COMPONENT_ARCHITECTURE_GUIDE.md BEFORE creating any component
✓ Reuse TextInput, NumberInput, SelectInput, AddressForm, LocationPicker directly
✓ Reuse StatusBadge for all status displays (use variant props)
✓ Create only truly new components when representing unique data domains
✓ Use props to handle variations (doc types, warehouse actions, deactivation reasons)

DON'Ts:
✗ DO NOT create separate DocumentUpload, DocumentInput components — use existing DocumentStatusCard pattern
✗ DO NOT create new form input components — reuse TextInput/NumberInput/SelectInput with props
✗ DO NOT create WarehouseAddressForm — reuse AddressForm component
✗ DO NOT create different modal/card components for each action — use single card with variant props
✗ DO NOT duplicate LocationPicker logic — import from shared library
✗ DO NOT create new button/banner components — reuse CTAFooter, WarningBanner, InfoBanner

Consistency & Mobile Responsive Guidelines

  • ✓ Spacing: 16px gap between sections, 24px padding inside cards, 80px footer height
  • ✓ Colors: Use CSS variables (--accent-primary: #06b6d4, --text-main: #f3f4f6)
  • ✓ Typography: Outfit 800 for titles, Inter 400/600 for body, 12-14px labels
  • ✓ Mobile Breakpoints: 360px (mobile), 600px (tablet), 1024px (desktop)
  • ✓ Form Validation: Frontend mirrors backend Zod, real-time error display
  • ✓ Accessibility: aria-label on all buttons, labels paired with inputs, focus states
  • ✓ Status Display: Use StatusBadge component for all states (verified, pending, inactive)
  • ✓ Warning Severity: Red critical banners for deactivation, amber for capacity changes
  • ✓ State Management: Store form state in local component state, popper context for readonly data
  • ✓ Error Handling: Consistent red border + message for all validation errors
  • ✓ Loading States: Disabled buttons, spinner overlay during API calls
  • ✓ Selection Lists: Use single-select grid layout (not dropdown) for warehouse/document selection

Database Schemas (Drizzle ORM DDL)

warehouses (Secondary Storage Assets)

CREATE TABLE warehouses ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), popper_id UUID NOT NULL REFERENCES users(id), warehouse_code TEXT NOT NULL UNIQUE, name TEXT NOT NULL, capacity_mt NUMERIC(6,2) NOT NULL, district TEXT NOT NULL, panchayat_village TEXT NOT NULL, gps_coordinates POINT NOT NULL, status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','pending_approval','inactive')), created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

Strict Input Validations & Error Enforcements

  • GST Validation: Validates against expressions.
  • Capacity Limits: Must be greater than zero. Capacities exceeding typical warehouse specs trigger warning flags.

Offline Caching & Queue Synchronization

  • Offline Storage: Edits cache locally in SQLite tables, writing updates sequentially once network resolves.

Security & Media Protection Protocols

  • Scope Access: Restricts operations exclusively to scouts assigned to the target district.
Manage Popper Steps Directory
5 Steps
1 Edit Identity Profile
Purpose
Edits basic profile fields like spelling, phone numbers, and addresses.
Business Requirement
Restricts editing verified mobile numbers directly without OTP challenges.
User Experience (UX)

Text inputs are pre-filled with active profile data.

Verification badges flag status indicators.

Bottom CTA updates profiles.

Technical Details
Invokes updatePopperProfile mutations.
2 Update Business Documents
Purpose
Updates GST certificates, Trade Licenses, and regulatory tax forms.
Business Requirement
Updates require resubmitting verification files, locking active accounts until resolved.
User Experience (UX)

Displays current document status badges (Verified, Under Review, Missing).

Inputs accept new document numbers.

Camera pickers capture files.

Technical Details

Invokes updatePopperDocuments.

Sets document verification states back to pending.

3 Edit Warehouse Details
Purpose
Modifies existing warehouse metrics (Capacity, addresses, storage types).
Business Requirement
Locks changes to capacities if lots are currently active in storage facilities.
User Experience (UX)
Selects target warehouses. Entry fields handle storage details. GPS tools capture active positioning tags.
Technical Details
Invokes upsertWarehouseAsset mutations.
4 Add New Warehouse Asset
Purpose
Appends new warehouses to the popper's active profile directory.
Business Requirement
Requires exact GPS verification matching warehouse coordinates.
User Experience (UX)

Wizard flows input names, capacities, storage types.

GPS tools lock active positioning tags.

Reviews and submits new warehouse registrations.

Technical Details

Generates unique warehouse_code values.

Invokes upsertWarehouseAsset.

5 Deactivate Warehouse Asset
Purpose
Temporarily or permanently disables active warehouses from storage assignments.
Business Requirement

Requires selecting deactivation reasons (e.g.

structural damage, lease termination).

User Experience (UX)

Renders warning sheets listing active listing links.

Selection options choose deactivation codes.

Dark amber buttons suspend allocations.

Technical Details
Invokes toggleWarehouseStatus transactions to deactivate allocations.
