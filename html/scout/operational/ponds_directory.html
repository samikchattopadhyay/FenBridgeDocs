Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (3 Screens)

The Ponds Directory is designed as a unified workflow consisting of three linked views.

Screen Breakdown:
• Screen 1: Combined Pond List (search, filter chips, browse list)
• Screen 2: Pond Detail (3 tabs: Profile, Inspections, Spot Offers)
• Screen 3: Pond Inspection Record (read-only audit detail of historical inspection)

Key Procedures

  • listAssignedPonds — Protected. Returns ponds joined with scout_assignments_ponds and farmer user profile information. Supports search parameters (farmer, village, pond code) and filter by inspection recency.
  • getPondDetail — Protected. Hydrates single pond metadata + farmer user profile info + chronological pond inspections (ordered by checked_at DESC) + active spot offers.
  • getPondInspectionRecord — Protected. Reads detailed physical and biological readings from pond_inspections, joined with pond_bund_readings and pond_inspection_photos for a specific inspection ID.

Core Database Tables (Drizzle ORM)

ponds (Pond Identity & Meta)

CREATE TABLE ponds ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), farmer_user_id UUID NOT NULL REFERENCES users(id), pond_code TEXT UNIQUE NOT NULL, name TEXT, area_value NUMERIC(6,2), area_unit TEXT CHECK (area_unit IN ('acre','bigha','hectare')), est_annual_capacity_kg NUMERIC(10,2), district TEXT, panchayat_village TEXT, full_address TEXT, gps_lat NUMERIC(9,6), gps_lng NUMERIC(9,6), status TEXT NOT NULL DEFAULT 'pending_approval' CHECK (status IN ('pending_approval','active','inactive','rejected')), created_via_scout_id UUID REFERENCES scouts(id), created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_ponds_farmer ON ponds (farmer_user_id); CREATE INDEX idx_ponds_scout ON ponds (created_via_scout_id);

scout_assignments_ponds (Access Scope Enforcement)

CREATE TABLE scout_assignments_ponds ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES scouts(id), pond_id UUID NOT NULL REFERENCES ponds(id), assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(), active BOOLEAN NOT NULL DEFAULT true, UNIQUE (scout_id, pond_id) ); CREATE INDEX idx_sapo_scout ON scout_assignments_ponds (scout_id, active);

Related Tables

  • users: Base farmer user profiles (full_name, mobile) linked via farmer_user_id
  • pond_inspections: Chronological logs of physical, chemical, and biological measurements
  • pond_bund_readings: Granular sub-readings detailing density and biological maturity
  • pond_inspection_photos: Uploaded inspection site verification media
  • spot_offers: Bids and offers placed with a pond as the supply source

Enforcement & Scope Rules

Server-Side Scope Security

Scope Constraint:
All procedures must enforce server-side constraints. A scout is strictly restricted to viewing and interacting only with ponds that are actively linked to their profile:

WHERE scout_assignments_ponds.scout_id = session.scout_id AND scout_assignments_ponds.active = true

Ponds Directory Operational Workflow
3 Screens
1 Combined Ponds List
Purpose

Direct browse index for all assigned farmer ponds that the scout is responsible for managing.

Provides rapid lookup, filtering, and visual urgency cues.

Business Requirement

Enforces strict server-side scoping via scout_assignments_ponds.

The list must display key parameters (Farmer name, Pond ID, Village, Area) and support partial string search alongside inspection recency filters (Today, This Week, Overdue).

User Experience (UX)

Mirrors the Warehouse Directory.

Top navigation bar holds a text search field and active filter chips.

List items render as sleek interactive cards with robust layout architecture: farmer name (heavy font), pond identifier, village location, area values, and a high-visibility inspection status badge ("Never Inspected" in amber/danger, or "Inspected [Date]" in subtle gray).

User Actions
  • Type search query (filters instantaneously by farmer, village, or pond code)
  • Toggle recency filter chips (All, Today, This Week, Overdue)
  • Tap any pond card → navigates to the Pond Detail view
Technical Details

Queries ponds joined with scout_assignments_ponds where scout_assignments_ponds.scout_id = session.user_id.

Aggregates the last inspection timestamp by querying MAX(checked_at) from pond_inspections.

Filters are evaluated post-fetch or dynamically injected in the database query.

Database Schema & Scope

Reads from: ponds, scout_assignments_ponds, users (farmer profiles).

Enforces scope using scout_assignments_ponds.active = true.

2 Pond Detail Dashboard
Purpose

Comprehensive, multi-tab single pond file.

Serves as the central control dashboard for reviewing pond conditions, auditing history, checking associated transaction records, and launching new pond inspections.

Business Requirement

Must mirror the Warehouse Detail screen design.

Organize data into three tabs: Profile, Inspections, and Spot Offers.

The primary CTA "Start Inspection" must be prominent and easily accessible, serving as the entry point for starting a fresh inspection wizard.

User Experience (UX)
Sleek header card containing the pond name, farmer name, active/inactive status badge, and total area. An amber banner warns if the pond is in "Pending Approval" state. Below, three equal-width tabs (Profile Inspections Spot Offers) structure independent scrolling panels. Action buttons populate respective tabs.
Tab Contents & Layout
  • Profile Tab: Pond Code, capacity estimation, exact GPS coordinates with map embed (latitude/longitude), full address description, village, and panchayat info.
  • Inspections Tab: Chronological log list. Row cards show inspection date, checked-by scout, water level, algae/disease tags, readiness scores, and verified photo count. Tap row → Inspection Record.
  • Spot Offers Tab: Lists recent bids/offers using this pond as supply source. Shows offer ID, lot status, created date, and bid pricing metrics.
Technical Details

Combines multiple SQL queries to hydrate all tabs: profile metadata from ponds, logs from pond_inspections ORDER BY checked_at DESC, and bids from spot_offers WHERE source_type = 'pond' AND source_id = pond_id.

Offline-first profile caching is activated for active assignments.

3 Pond Inspection Record
Purpose

Read-only historical audit report detailing the physical, biological, and chemical parameters captured during a past pond inspection.

Provides critical historic evolutionary context.

Business Requirement

Must remain strictly read-only after submission.

Modifying historical audits is prohibited for compliance.

Must show all captured details: GPS arrival coordinates, check type (periodic/initial), physical conditions, bund readings, and verification media.

User Experience (UX)

Upper panel: inspection header with date, scout name, GPS arrival accuracy, check type, and physical indicators (water level, algae count, pest flags).

Middle panel: detailed bund readings table with row items for density and biological maturity percentages, with a computed average row highlighted at the bottom.

Lower panel: verified photo strip (fixed-ratio preview tiles) with full-screen viewer lightbox on tap.

Technical Details & Schema

Hydrates from three tables: pond_inspections (header logs), pond_bund_readings (granular measurements), and pond_inspection_photos (media storage references).

Images are automatically cached in local device storage for rapid offline viewing.
