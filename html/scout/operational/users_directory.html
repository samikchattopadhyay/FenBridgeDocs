Workflow Overview
Registered Users Directory is the unified interface for browsing all farmers and poppers registered by the scout or explicitly assigned to them. The workflow replaces the previous "My Farmers" pattern with a role-agnostic list that supports search, role filtering (All / Farmer / Popper / Buyer), and status filtering (Pending Approval / Active / Suspended). Each user has a tabbed detail view supporting role-specific management: Farmer Detail shows ponds and pond-related actions (add pond, edit); Popper Detail shows warehouses and warehouse-related actions. A FAB provides quick access to the Register New User workflow. Both farmer and popper profiles support multi-tab organization for associated resources (ponds/warehouses, bank accounts, documents, activity) and Manage actions (edit identity, update bank, upload documents).
4.1 Combined Users List
Purpose
Single browse surface for all farmers and poppers the scout has registered or been assigned.
Business Requirement
Replaces the previous "My Farmers" pattern with a unified directory that handles both roles. Scope-limited to users the scout registered (users.created_via_scout_id = self) plus users explicitly assigned (via scout_user_assignments table for ongoing management).
User Experience (UX)
Search bar + role filter chips (All / Farmer / Popper / Buyer-rare) + status filter (Pending Approval / Active / Suspended). Each row: name, role pill, mobile, district, last activity, status. FAB → "Register New User" (opens role chooser → Farmer or Popper registration workflow).
User Actions
• Search by user name or mobile
• Filter by role (All / Farmer / Popper / Buyer)
• Filter by status (Pending Approval / Active / Suspended)
• Tap row → Farmer/Popper Detail
• FAB: Register New User
Technical Details
Reads users joined farmer_profiles / popper_profiles where created_via_scout_id = self OR id IN (scout_user_assignments). Real-time subscription on users for assigned set. Search queries user name, mobile, district. Role and status filters applied client-side or server-side per architecture.
Design Details
Standard list pattern. Search bar at top, filter chips below (role toggles, status toggles). Each row: avatar/initials, name (Title 3), role pill (Farmer: teal, Popper: purple, Buyer: blue), mobile, district, last activity timestamp, status pill (Pending Approval: amber, Active: accent, Suspended: danger). FAB in bottom-right: "+ Register User".
Database Schema
CREATE TABLE scout_user_assignments ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES scouts(id), user_id UUID NOT NULL REFERENCES users(id), assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(), active BOOLEAN NOT NULL DEFAULT true, UNIQUE (scout_id, user_id) ); CREATE TABLE farmer_profiles ( user_id UUID PRIMARY KEY REFERENCES users(id), full_name TEXT NOT NULL, dob DATE, district TEXT NOT NULL, panchayat_village TEXT, full_address TEXT, alt_phone TEXT, preferred_language TEXT NOT NULL DEFAULT 'en' ); CREATE TABLE popper_profiles ( user_id UUID PRIMARY KEY REFERENCES users(id), contact_name TEXT NOT NULL, business_name TEXT NOT NULL, pan_number TEXT, gstin TEXT, fssai_number TEXT, district TEXT NOT NULL, panchayat_village TEXT, full_address TEXT, alt_phone TEXT, preferred_language TEXT NOT NULL DEFAULT 'en' );
4.2 Farmer Detail
Purpose
Full record of a farmer, organised in tabs.
Business Requirement
Five tabs — Profile, Ponds, Bank, Documents, Activity. Each tab supports the relevant Manage Farmer sub-action.
User Experience (UX)
Header card (avatar/initials, name, status pill, mobile, district), tab row, scrollable tab content. Edit/action buttons surface in each tab.
User Actions
• Tap tabs (Profile / Ponds / Bank / Documents / Activity)
• Profile tab: "Edit Identity" link / phone controls
• Ponds tab: Tap pond row → Pond Detail, "+ Add Pond" → Add Pond workflow
• Bank tab: "Edit Bank" → Bank workflow
• Documents tab: Upload documents
• Activity tab: View chronological feed
Technical Details
Reads users, farmer_profiles, ponds, user_bank_accounts, user_documents, plus activity rolled up from bids, spot_offers, purchase_orders, payments. Each tab queries its subset; tabs are lazy-loaded or all hydrated on entry. Pending Approval state triggers a banner across the screen.
Design Details
Header card: full name (Title 2), status pill, mobile with copy icon, district. Tab row with counts on each label. Profile: avatar/initials, full name, DoB, district, village, address, alternate phone, language. Ponds: table of ponds (code, area, status) with "+ Add Pond" button. Bank: account holder, masked account, IFSC, pennydrop status, "Edit" button. Documents: grid of document uploads by type. Activity: chronological feed of cards. Pending Approval state shown as banner at top.
Database Schema
CREATE TABLE user_bank_accounts ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id), account_holder_name TEXT NOT NULL, account_number_masked TEXT NOT NULL, account_number_encrypted TEXT NOT NULL, ifsc_code TEXT NOT NULL, bank_name TEXT, branch_name TEXT, pennydrop_status TEXT NOT NULL DEFAULT 'pending' CHECK (pennydrop_status IN ('pending','passed','failed')), pennydrop_attempted_at TIMESTAMPTZ, is_active BOOLEAN NOT NULL DEFAULT true, created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_uba_user ON user_bank_accounts (user_id, is_active); CREATE TABLE user_documents ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id), doc_type TEXT NOT NULL CHECK (doc_type IN ('aadhaar','pan','land_record','pond_lease', 'gstin_cert','fssai_cert','business_pan','warehouse_lease','photo')), storage_path TEXT NOT NULL, uploaded_by_scout_id UUID REFERENCES scouts(id), uploaded_at TIMESTAMPTZ NOT NULL DEFAULT now(), verified_at TIMESTAMPTZ, is_active BOOLEAN NOT NULL DEFAULT true ); CREATE INDEX idx_ud_user ON user_documents (user_id, doc_type);
4.3 Popper Detail
Purpose
Same as Farmer Detail but for poppers.
Business Requirement
Five tabs — Profile, Warehouses, Bank, Documents, Activity. Warehouses tab links into the existing Warehouses Directory entries owned by this popper.
User Experience (UX)
Identical pattern to Farmer Detail.
User Actions
• Tap tabs (Profile / Warehouses / Bank / Documents / Activity)
• Profile tab: "Edit Identity" link / phone controls
• Warehouses tab: Tap warehouse row → Warehouse Detail, "+ Add Warehouse" → opens the Add Warehouse step from Manage Popper workflow
• Bank tab: "Edit Bank" → Bank workflow
• Documents tab: Upload documents
• Activity tab: View chronological feed
Technical Details
Reads users, popper_profiles, warehouses (via existing schema, poppers.user_id references users.id), user_bank_accounts, user_documents, activity from spot_offers, purchase_orders, payments. Each tab queries its subset; tabs are lazy-loaded. Pending Approval state triggers a banner.
Design Details
Header card: business name (Title 2), contact person name, status pill, mobile, district. Tab row with counts. Profile: avatar/initials, contact person name, DoB, business name, PAN, GSTIN, FSSAI, district, village, address, alternate phone, language. Warehouses: table of warehouses (code, capacity, status) with "+ Add Warehouse" button. Bank: same as Farmer Detail. Documents: grid organized by type. Activity: chronological feed. Pending Approval banner at top.
Database Schema
Uses popper_profiles (defined in Screen 4.1) + existing poppers/warehouses (existing schema must be updated so poppers.user_id references users.id). Bank accounts and documents schemas identical to Farmer Detail (user_bank_accounts, user_documents).
