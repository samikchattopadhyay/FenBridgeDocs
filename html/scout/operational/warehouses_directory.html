Workflow Module 5

Warehouses Directory

Scout's comprehensive directory of all assigned warehouses with fitness status tracking, active lot inventory, and complete storage failure audit trail. Enables fast access to any warehouse with at-a-glance status visibility and deep drilling into fitness checks, stored lots, and failure patterns.

Warehouse List API
Procedure: scout.warehouse.listAssigned(filters, searchQuery, sortBy)
Input: filter chips (Fit/Unfit/Unchecked/HasActiveLots), search string (popper name / warehouse code / area), sort order (byName/byRecentActivity/byFitnessStatus)
Output: List of warehouses with popper name, code, location, fitness status pill, active sealed lot count, open-issue flag
Cache: Returns from local cache first; real-time subscription updates live when online
Offline: Fully supported — list renders from cached data; pull-to-refresh syncs when online
Warehouse Detail API
Procedure: scout.warehouse.getDetail(warehouseId)
Output: Four tabs: Profile (identity + GPS), Fitness (check history), Lots (grouped by status), Failures (storage-failure roll-up)
Profile Tab: warehouses + poppers joined, GPS map preview, last seal date, last re-check date
Fitness Tab: warehouse_fitness_checks ordered by checked_at DESC, verdict pill, scout name, critical-failure marker
Lots Tab: lots grouped by status (Active Sealed / Listed / Sold / Expired / Voided / Closed Terminal), with lot code, grade pill, quantity, status dot, last-event date
Failures Tab: warehouse_storage_failures ordered by created_at DESC, failure outcome pill, lot code, reason, date
Start Fitness Check: Creates scout_tasks row with task_type='warehouse_fitness_check'; enabled only if no in-progress check
Lot Detail API
Procedure: scout.warehouse.getLotDetail(lotId)
Output: Single lot's full record — header card with lot code, popper, warehouse, current status banner (colour-coded by lifecycle state), then stacked sections: Summary, Inspection Timeline, Current Seal, Re-check Log, Documents
Inspection Timeline: inspections WHERE lot_id=? ORDER BY checked_at ASC, including reinspection_seq > 0
Current Seal: lot_seals WHERE lot_id=? AND status='active', with past seals (expired/broken/voided) under "Past seals" expander
Re-check Log: storage_rechecks WHERE lot_id=? ORDER BY checked_at ASC
Documents: grade_certificates WHERE lot_id=?, latest is primary
All sections: Read-only for scout, fully offline-capable (renders from cache)
Warehouses Table
CREATE TABLE warehouses ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), popper_user_id UUID NOT NULL REFERENCES users(id), warehouse_code TEXT UNIQUE NOT NULL, name TEXT, built_capacity_kg NUMERIC(10,2), capacity_unit TEXT CHECK (capacity_unit IN ('kg','bags')), district TEXT, village_area TEXT, full_address TEXT, gps_lat NUMERIC(9,6), gps_lng NUMERIC(9,6), fitness_status TEXT NOT NULL DEFAULT 'unchecked' CHECK (fitness_status IN ('unchecked','fit','unfit','needs_recheck')), last_fitness_check_at TIMESTAMPTZ, status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','inactive','pending_approval')), created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_warehouses_popper ON warehouses (popper_user_id); CREATE INDEX idx_warehouses_scout_assigned ON warehouses (id) WHERE status = 'active';
Scout Assignments (Warehouses)
CREATE TABLE scout_assignments_warehouses ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES users(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(), active BOOLEAN NOT NULL DEFAULT true, UNIQUE (scout_id, warehouse_id) ); CREATE INDEX idx_saw_scout ON scout_assignments_warehouses (scout_id, active);

Search & Filter Validation
Search: Fuzzy match on local cache first (popper name, warehouse code, village/area), then server-side parallel
Filter Chips: Multi-select allowed (All / Fit / Unfit / Unchecked / Has active lots)
Sort: By name (default) / by recent activity / by fitness status
Scope Validation: Scout sees only warehouses in scout_assignments_warehouses with active=true; Admin bypass enforced server-side
Start Fitness Check Validation
Precondition: No in-progress warehouse_fitness_check task for this warehouse (idempotency guard)
Permission: Scout role only; not available for read-only guest users
Error: If fitness check already in progress, show: "Fitness check for this warehouse is already in progress" (409 Conflict)
Lot Detail Access
Permission: Scout must have warehouse_id matching lot.warehouse_id via scout_assignments_warehouses
Read-only: All tabs are read-only; only deep-links to originating inspection/re-check records if those are tappable
Error: If lot not found or warehouse not assigned: 404 Not Found

Workflow Overview
Warehouses Directory is the scout's primary browse interface for all warehouses assigned to them. The workflow provides search and filtering capabilities to locate specific warehouses, view detailed profiles including GPS coordinates and linked popper information, browse complete fitness check history with pass/fail verdicts, examine all lots currently held (grouped by status), and review storage-failure patterns. Scout can initiate a fresh warehouse fitness check directly from Warehouse Detail. Scope is enforced server-side — scouts see only warehouses assigned via scout_assignments_warehouses. The directory mirrors the Pond List/Detail pattern for consistency with existing pond management flows. Every lot visible in the Lots tab links through to a comprehensive Lot Detail screen showing the full lifecycle history from inspection through sealing, re-checks, and any closure.
5.1 Warehouse List
Purpose
The scout's directory of warehouses — every warehouse in their assigned area, browsable and filterable.
Business Requirement
Scout works a fixed set of warehouses over time and needs fast access to any, whether or not there's a task today. List should make warehouse's current standing visible at a glance — fitness status, active sealed lot count, any open issues. Scouts only see assigned warehouses; Admin sees all.
User Experience (UX)
Standard searchable list. Search bar at top, filter chips below, vertical list of warehouse cards. Each card shows: popper name, warehouse code, location, fitness status pill, active sealed lot count, danger marker if issues. Pull-to-refresh syncs.
User Actions
• Search by popper name / warehouse code / area
• Toggle filter chips: All / Fit / Unfit / Unchecked / Has active lots
• Sort: by name (default) / by recent activity / by fitness status
• Tap warehouse card → Warehouse Detail
• Pull down to refresh
Technical Details
Loads via: SELECT * FROM warehouses w JOIN scout_assignments_warehouses sa ON w.id = sa.warehouse_id WHERE sa.scout_id = ? (scope guard). Aggregates per warehouse: active_sealed_lot_count from lot_seals WHERE status='active', open_issue_flag if any lot_seals.status='broken' unresolved, recent warehouse_storage_failures, or fitness_status='unfit'. Search is local fuzzy match first, then server-side in parallel. Offline-first: list renders from cache; pull-to-refresh syncs. Real-time subscription on warehouses for scout's assigned set.
Design Details
Top bar: "Warehouses" title (left), filter icon (right). Search bar: "Search popper, code, area…" (sticky). Filter chips: All · Fit · Unfit · Unchecked · Has active lots (multi-select, scrollable). Warehouse cards: popper name (Body strong), code · village/area (Body), fitness pill (accent "Fit" / amber "Unchecked" / danger "Unfit"), "X active lots" chip, ⚠ open-issue dot if applicable. Sort button "Sort by: Name ↓". Empty state: "No warehouses assigned yet". Bottom nav: Today · Warehouses (active) · Sync · Profile.
Database Schema
CREATE TABLE scout_assignments_warehouses ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES scouts(id), warehouse_id UUID NOT NULL REFERENCES warehouses(id), assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(), active BOOLEAN NOT NULL DEFAULT true, UNIQUE (scout_id, warehouse_id) ); CREATE INDEX idx_saw_scout ON scout_assignments_warehouses (scout_id, active);
5.2 Warehouse Detail
Purpose
The single warehouse's full record — profile and GPS, fitness history, lots held, and storage-failure history — in one tabbed screen.
Business Requirement
Everything scout (or Admin) needs to know about a warehouse must be reachable from this screen. Four views: Profile (identity, address, GPS, contact), Fitness History (every fitness check ever performed, with outcomes), Lots (active sealed + sold/expired lots), Storage Failures (rollup of failed re-checks). Scout can initiate fresh fitness check from here if due or warranted.
User Experience (UX)
Header section with warehouse identity and primary action (Start Fitness Check, when permitted), then tab row: Profile · Fitness · Lots · Failures. Each tab is scrollable section. Current fitness status always visible in header.
User Actions
• View warehouse header (name, code, fitness status pill, active lot count)
• Tap "Start Fitness Check" → enters Warehouse Fitness Check workflow
• Tap "Navigate" → maps app
• Tap "Call popper" → dialer
• Switch tabs: Profile · Fitness · Lots · Failures
• Inside Lots tab → tap lot row → Lot Detail
• Inside Fitness tab → tap check row → read-only detail
• Inside Failures tab → tap failure row → originating Storage Re-check record
Technical Details
Hydrates from cached warehouses, poppers. Tab data: Profile = warehouses + poppers joined; Fitness = warehouse_fitness_checks WHERE warehouse_id=? ORDER BY checked_at DESC; Lots = lots WHERE warehouse_id=? with joined lot_seals, grouped by status; Failures = warehouse_storage_failures WHERE warehouse_id=? ORDER BY created_at DESC with joined storage_rechecks. "Start Fitness Check" enabled only when no in-progress check task (idempotency); creates scout_tasks row with task_type='warehouse_fitness_check'. All data renders from cache; real-time subscriptions push updates live when online. Tab counts computed client-side from cache.
Design Details
Top bar: back, popper-name + warehouse-code (Title 3). Header card: popper name + warehouse code, fitness pill (large, prominent): "Fit · checked 2 Feb 2026" / "Unfit · since 14 Mar" / "Unchecked", counts chip "5 active lots · 2 open issues", action row: "Start Fitness Check" (primary, when permitted) · "Navigate" · "Call popper". Tab bar: Profile · Fitness · Lots · Failures (with counts). Profile tab: popper card (name, phone, district), warehouse code, address, GPS coordinates with map preview, last seal/re-check dates. Fitness tab: newest first, each row shows verdict pill (Pass/Fail), check date, scout, critical-failure marker if applicable. Lots tab: grouped collapsible sections (Active Sealed / Listed / Sold / Expired / Voided / Closed Terminal), lot row shows code, grade pill, quantity, status dot, last-event date. Failures tab: newest first, failure outcome pill, lot code, reason, date, or "No storage failures recorded" empty state.
Database Schema
Reads: warehouses, poppers, warehouse_fitness_checks, warehouse_fitness_items, lots, lot_seals, inspections, warehouse_storage_failures, storage_rechecks. No new tables required; directory queries join existing schema.
5.3 Lot Detail
Purpose
The single lot's full record — from declaration through inspection, sealing, re-checks, and any rejection/closure — on one screen.
Business Requirement
Lot can have complex history: declared → inspected (possibly rejected, reconditioned, re-inspected) → sealed → re-checked (possibly voided or rerouted) → listed → sold (or expired, or closed terminal). Screen must show all as coherent narrative, with every artefact reachable: inspection records, Grade Certificate, seal record (chain-of-custody), re-check log, and any closure record. Scout viewing lot they're about to work on should see at a glance: where is this lot in lifecycle, and what records exist.
User Experience (UX)
Header card with lot identity and current status banner, then stacked sections: Summary, Inspection History (timeline), Current Seal (if sealed), Re-check Log, Documents (certificates, seal record). Sections expand/collapse where lengthy.
User Actions
• Review lot header — lot code, popper, warehouse, current status
• Review status banner (colour-coded by lifecycle state, with closure banner for terminal lots)
• Scroll inspection timeline (oldest first, with connector lines)
• Tap inspection row → opens detail (accepted → inspection detail; rejected → Rejection Detail)
• Tap active seal record → Seal Record Detail
• Tap re-check row → opens read-only detail
• Tap Grade Certificate → Grade Certificate Viewer
• (If terminal) tap "View closing record" → closing inspection or storage re-check
• Back → Warehouse Detail (Lots tab)
Technical Details
Hydrates from cached lots (header) + queries: Inspections timeline = inspections WHERE lot_id=? ORDER BY checked_at ASC (full chain including re-inspections where reinspection_seq > 0); Current seal = lot_seals WHERE lot_id=? AND status='active' LIMIT 1, with historical seals (status IN ('expired','broken','voided')) under "Past seals" expander; Re-check log = storage_rechecks WHERE lot_id=? ORDER BY checked_at ASC; Documents = grade_certificates WHERE lot_id=? (latest is primary, earlier archived); Closure state (if terminal) = derived from lots.status IN ('voided','expired','terminal') — no separate closure table; closing record resolved via final inspection or storage_rechecks row. Each row deep-links to relevant detail surface, all read-only. Lifecycle state derived from lots.status + lot_seals.status + inspections.outcome — single computed banner in header. Real-time subscriptions on lot_seals and inspections. Fully offline (all details render from cache).
Design Details
Top bar: back, lot code (Title 3) + grade pill (if assessed). Header card: lot code (large), popper name, warehouse code, declared/verified quantity. Status banner (colour-coded): "Pending inspection" (text-lo) / "Verified — awaiting seal" (accent) / "Sealed — listed" (accent) / "Sold" (text-lo) / "Expired — needs re-inspection" (amber) / "Seal voided" (danger) / "Pending re-inspection" (amber) / "Closed terminal" (danger, full Terminal Closure banner). PO link chip if related_po_id. Summary section: declared vs verified quantity (with variance line), claimed vs assessed grade, source (Spot / Against PO). Inspection Timeline section: vertical list oldest first with connector lines. Tap inspection → detail (accepted or rejected). Current Seal: active record shown, past seals under expander. Re-check Log: list ordered by checked_at. Documents: grade certificates (latest primary, earlier archived). Fully offline.
Database Schema
Read-only. Reads from: lots, lot_seals, inspections, storage_rechecks, grade_certificates (all existing tables). Closure state derived from lots.status — no separate closure table. No new tables required. Deep-links to inspection/seal/re-check detail surfaces via existing schema.
