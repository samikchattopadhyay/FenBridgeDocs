Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Storage Re-check workflow follows a stepper-based form pattern with progressive integrity validation. The 5-screen flow systematically validates lot condition and seal integrity through a combination of QR scanning, environmental auditing, and photographic evidence. Each screen contributes to a final binary verdict (Pass/Void) that gates further listing eligibility.

Key patterns: Progressive disclosure (show next step only after current validation), integrity gates (break seal or critical condition failure voids the lot immediately), and evidence-backed decisions (photos required before verdict).

Screen Decomposition

Screen 1: Lot / Warehouse Header

Displays the lot context and checks scout is on-site. Captures GPS location for audit trail.

TopAppBar { title: "Re-check Storage", backBtn } | InfoBanner { lotId, sealId, shelfLife_days, countdown_timer, warning_if_lt_7_days } | TextInput { lotId, warehouseId [read-only] } | LocationPicker { captureGPS } | CTAFooter { "Start Re-check" → screen 2 }

Screen 2: Seal & Net Integrity Check

Scout scans the hardware seal QR code and visually confirms no tampering. Mismatch auto-fails.

TopAppBar { title: "Seal & Net Integrity", backBtn } | StepperPanel { active: 2 } | QRCodeScanner { scanAction, verifyAction } | StatusBadge { status: scan_result === stored_seal_id ? "Intact" : "Tampered", color: green|red } | CheckboxInput { "Netting secure?" } | CTAFooter { "Continue" [disabled if Tampered] → screen 3 }

Screen 3: Conditions Check

Quick 4-point audit. Any "No" answer flags for re-inspection or void.

TopAppBar { title: "Storage Conditions", backBtn } | StepperPanel { active: 3 } | CheckboxInput[] { "Bags appear dry?", "Lot elevated on pallets?", "No rodent signs?", "No strong odours?" } | CTAFooter { "Continue" → screen 4 }

Screen 4: Evidence Photos

Mandatory 2-photo documentation: seal macro closeup and lot stack in context.

TopAppBar { title: "Evidence Photos", backBtn } | StepperPanel { active: 4 } | PhotoCaptureCard[] { slot_1: "Seal State (macro)", slot_2: "Lot Stack Context" } [mandatory] | PhotoGrid { display_captured_photos, edit_actions } | CTAFooter { "Continue" [disabled if missing] → screen 5 }

Screen 5: Outcome

Final verdict banner with attestation checkbox. Determines lot status (Pass → continue listing, Void → remove from sale).

TopAppBar { title: "Re-check Verdict", backBtn } | StepperPanel { active: 5 } | StatusBadge { verdict: "PASS" | "VOID", color: green|red, size: large } | PhotoGrid { review_captured_photos } | ConfirmationCheckbox { "I attest to accuracy" } [mandatory] | CTAFooter { "Submit Audit" → backend transaction }

State Management & Backend Logic

Local Wizard State

const [wizardState, setWizardState] = useState({ lot_id: string; seal_id: string; scout_gps: { lat, lng }; seal_scan_result: string; seal_integrity_status: "Intact" | "Tampered"; netting_secure: boolean; conditions: { bags_dry: boolean; pallets_elevated: boolean; no_rodents: boolean; no_odours: boolean; }; photos: { seal_state: File, lot_context: File }; attestation_checked: boolean; verdict: "PASS" | "VOID" | null; });

Verdict Engine Logic

const computeVerdict = (state) => { // Critical failures → immediate VOID if (state.seal_integrity_status === "Tampered") return "VOID"; if (!state.netting_secure) return "VOID"; // Conditions failures → flag anomaly const conditionsFail = Object.values(state.conditions) .some(v => v === false); if (conditionsFail) return "VOID"; // All checks passed return "PASS"; };

Backend Transaction

BEGIN TRANSACTION; 1. INSERT INTO recheck_logs ( lot_id, scout_id, seal_scan_result, seal_integrity_status, conditions_json, scout_gps, audit_timestamp ) VALUES (...); 2. INSERT INTO recheck_photos ( recheck_id, photo_type, photo_url, upload_timestamp ) VALUES (...) x2; -- seal_state, lot_context 3. UPDATE lots SET last_recheck_at = now(), status = CASE WHEN $verdict = 'VOID' THEN 'voided' ELSE 'listed' END WHERE id = $1; 4. IF verdict = 'VOID' THEN INSERT INTO lot_rejections (lot_id, reason, timestamp) VALUES ($1, 'Storage re-check failed', now()); END IF; COMMIT;

Common Patterns & Reusability

  • Checklist Pattern Reuse: The 4-point conditions checklist (Screen 3) mirrors the Fitness Check audit pattern. Componentize as <ChecklistAudit items={[...]} onComplete={} /> for consistency across Warehouse Fitness, Storage Re-check, and Pond Inspection workflows.
  • Photo Management: 2-photo capture pattern (seal macro + context) reused from Lot Verification and Seal Application. Extract as <EvidencePhotoCapture slots={2} labels={[...]} mandatory />
  • QR Verification: Scan-to-verify logic (Screen 2) decoupled from domain. Create <QRVerifyComponent expectedId={seal_id} onMismatch={failHandler} /> for reuse in Seal Application and other verification workflows.
  • Status Progression: Verdict determination (Pass/Void) follows same binary gate pattern as Lot Verification Inspection. Stored in Context for UI state transitions and backend routing.
  • Offline Capability: Entire workflow (except photo upload) works offline. Sync queue captures recheck_logs + photo metadata on background sync. GPS and QR data cached locally until network availability.
