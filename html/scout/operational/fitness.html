For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Warehouse Fitness Check workflow uses a stepper-based form pattern with multi-point checklist audits. Each screen serves a distinct audit category (facilities, safety, hygiene) with pass/fail toggles, photo capture slots for defects, and a final verdict determination engine.

Screen Decomposition & Component Usage

Screen 1: Confirm Warehouse & Arrival

GPS arrival verification with warehouse context card.

Component Stack:
TopAppBar → StepperPanel → [InfoBanner (GPS accuracy warning)] → [Warehouse Context Card (warehouse_name, last_fitness_status)] → [LocationPicker with Geolocator] → StatusBadge (GPS Status) → CTAFooter (Continue)

Screen 2: Basic Facilities Audit

6-item checklist with inline photo slots for failed items.

Component Stack:
TopAppBar → StepperPanel → [ExpandableSection * 6] each containing: CheckboxInput (Pass/Fail) → PhotoCaptureCard (conditional on Fail) → SelectInput (Reason Code) → PhotoGrid (evidence) → CTAFooter (Next)

Screen 3: Safety & Hygiene Audit

6-item safety checklist with mandatory safety equipment photo capture.

Component Stack:
TopAppBar → StepperPanel → [InfoBanner (Safety Equipment reminder)] → [ExpandableSection * 6] → PhotoCaptureCard (mandatory) → PhotoGrid → CheckboxInput (attestation) → CTAFooter (Next)

Screen 4: Audit Summary & Submit

Verdict determination with attestation checkbox.

Component Stack:
TopAppBar → StepperPanel → [StatusBadge (Verdict: Fit/Unfit)] → [Summary Cards (Checklist recap)] → [CheckboxInput (Attestation)] → [CTAFooter (Submit disabled until checked)]

Screen 5: Submission Confirmation

Success state with sync status and next task recommendations.

Component Stack:
TopAppBar → [SuccessGraphic] → [Summary Chips (Verdict, Warehouse, Sync Status)] → [StatusBadge (Synced/Pending)] → [CTAFooter (Back to Dashboard, Next Task)]

State Management & Data Flow

Local Wizard State (Client-side):
• arrival_gps: { lat, lng, accuracy, timestamp }
• facilities_checklist: { [criterion_id]: { pass: bool, photo_paths: [], reason_code: string } }
• safety_checklist: { [criterion_id]: { pass: bool, photo_paths: [], reason_code: string } }
• interior_photos: PhotoCaptureCard files
• verdict: string ('fit' | 'unfit')
• attestation_checked: boolean

Backend Atomic Transaction (on Submit):
1. INSERT fitness_checks (warehouse_id, scout_id, arrival_lat, arrival_lng, verdict, created_at)
2. INSERT fitness_checklist_results * 6
3. UPDATE warehouses SET fitness_status = verdict
4. INSERT sync_queue entry (for offline reconciliation)

Verdict Engine Logic

// Screen 4 Verdict Determination const computeVerdict = (facilities, safety) => { const criticalCriteria = ['pests', 'roof', 'chemicals']; // Any critical fail = UNFIT for (let criterion of criticalCriteria) { if (facilities[criterion]?.pass === false || safety[criterion]?.pass === false) { return 'unfit'; } } // Any non-critical fail = FIT WITH OBSERVATIONS (amber banner) const hasNonCriticalFail = [...Object.values(facilities), ...Object.values(safety)] .some(item => item.pass === false); return hasNonCriticalFail ? 'fit_with_obs' : 'fit'; };

Common Patterns & Reuse

  • Checklist Pattern: Screens 2 & 3 use identical ExpandableSection + CheckboxInput + PhotoCapture pattern. DRY principle: extract to ChecklistAuditItem component.
  • Photo Management: All 5 screens use PhotoGrid and PhotoCaptureCard. Centralize photo compression and GPS-stamping in PhotoService util.
  • GPS Lock: Screen 1 blocks progression until accuracy < 20m. Reusable LocationPicker with accuracy threshold prop.
  • Status Persistence: Use React Context (FitnessCheckContext) to persist wizard state across all 5 screens.
