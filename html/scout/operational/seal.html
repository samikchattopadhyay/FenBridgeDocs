Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Seal Application workflow follows a sequential capture and verification pattern that interlinks digital lot records with physical tamper-evident hardware. The 4-screen flow combines identity verification, hardware binding, evidence documentation, and confirmation of the chain-of-custody seal. This workflow is critical for marketplace integrity—once sealed, lots are frozen pending logistics.

Key patterns: Hardware biometrics (QR scan for seal ID), multi-angle evidence capture, state progression gates, and irreversible state transitions.

Screen Decomposition

Screen 1: Confirm Lot & ID

Identity verification ensuring the physical stack matches the digital Certificate. Gate check to ensure only verified lots proceed.

TopAppBar { title: "Sealing", backBtn } | StepperPanel { active: 1 } | StatusBadge { status: "READY_TO_SEAL", color: green } | TextInput[] { lotId, popperName, weight, grade [all read-only] } | ExpandableSection { "Lot Details" } | CTAFooter { "Start Sealing" → screen 2 }

Screen 2: Seal Application

Hardware seal ID capture via QR scan. Uniqueness check. Netting application confirmation.

TopAppBar { title: "Apply Seal", backBtn } | StepperPanel { active: 2 } | QRCodeScanner { scan_action, extract_seal_id, validate_uniqueness } | StatusBadge { seal_id, status: "LOCKED", color: green } | CheckboxInput { "Protective netting applied?" } | CTAFooter { "Proceed to Evidence" → screen 3 }

Screen 3: Photo Evidence

4-way mandatory photo documentation: Seal ID closeup, stack with netting, scout with stack, warehouse context.

TopAppBar { title: "Evidence Photos", backBtn } | StepperPanel { active: 3 } | PhotoCaptureCard[] { slot_1: "Seal ID (macro)", slot_2: "Stack with Netting", slot_3: "Scout & Stack", slot_4: "Warehouse Context" } [mandatory] | PhotoGrid { display_captured, edit_actions } | ConfirmationCheckbox { "I verify accuracy" } [mandatory] | CTAFooter { "Finalize Seal" [disabled if incomplete] → screen 4 }

Screen 4: Submission Confirmation

Final success screen. Lot is now sealed and frozen from further modification. Display seal registry and next steps.

TopAppBar { title: "Seal Confirmed", backBtn } | StepperPanel { active: 4 } | StatusBadge { status: "LOT_SEALED", color: green, size: large } | ExpandableSection { display_seal_id, timestamp } | CTAFooter { "Back to Dashboard" → home }

Local Wizard State

const [wizardState, setWizardState] = useState({ lot_id: string; seal_id: string; netting_applied: boolean; evidence_photos: { seal_id_macro: File; stack_with_netting: File; scout_with_stack: File; warehouse_context: File; }; attestation_checked: boolean; sealed_at?: timestamp; });

Seal ID Validation Logic

const validateSealId = async (scannedId) => { // Check global uniqueness const existing = await db.seals.findOne({ id: scannedId }); if (existing) { return { valid: false, error: "Seal already registered" }; } // Check format (alphanumeric, length 8-16) if (!/^[A-Z0-9]{8,16}$/.test(scannedId)) { return { valid: false, error: "Invalid seal ID format" }; } return { valid: true }; };

Backend Transaction

BEGIN TRANSACTION; 1. INSERT INTO lot_seals ( lot_id, seal_id, netting_applied, scout_id, sealed_at ) VALUES ($1, $2, $3, $4, now()); 2. INSERT INTO seal_evidence ( seal_id, photo_type, storage_url, geotag, timestamp ) VALUES ($2, 'seal_id_macro', ...) x4; -- 4 photos 3. UPDATE lots SET status = 'sealed', seal_id = $2, sealed_at = now() WHERE id = $1; 4. INSERT INTO audit_log ( action, lot_id, seal_id, scout_id, timestamp ) VALUES ('SEAL_APPLIED', $1, $2, $4, now()); COMMIT; -- Post-transaction: Send notification to Seller triggerNotification('lot_sealed', $1, seller_id);

Common Patterns & Reusability

  • QR Verification Component: Seal ID scanning (Screen 2) reused from Storage Re-check. Extract as <HardwareQRVerify onCapture={handleSealId} validateFn={validateSealId} />
  • Multi-Photo Capture Pattern: 4-photo evidence pattern (Screen 3) reused from Lot Verification and Dispatch. Centralize <EvidencePhotoCapture slots={4} labels={[...]} />
  • State Lock Pattern: Once sealed, lot is frozen. Use read-only context flag to prevent back-navigation or edits. Mirror Lot Verification's "irreversible decision" pattern.
  • Success Confirmation: Final screen follows standard success pattern (green badge, navigation CTAs). Consistent with other workflow completions.
  • Offline Capability: Entire workflow except photo upload works offline. Sync queue captures seal data + photo metadata; unsealing requires admin override (always online).
