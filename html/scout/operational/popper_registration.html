Scout Operations

Register Popper (Scout-Assisted)

Complete 15-step workflow for scout-assisted popper registration with business identity capture, warehouse equipment documentation, warehouse management, real-time mobile OTP verification, and admin approval queue.

A 15-screen wizard including new popping capability profiling and ancillary equipment documentation. The scout sits with the popper (at their business/warehouse), explains the consent line, collects answers, captures business documents and warehouse photos, documents equipment specifications and capabilities, and submits. The popper's contact person's phone receives the OTP via bottom sheet overlay; scout reads it from the phone and enters it. On submit, the popper enters Pending Approval state and warehouse enters fitness check queue.

tRPC Procedure Architecture (15 Procedures)

The Register Popper workflow consists of 15 tRPC procedures organized by authentication level.

Procedure Breakdown:
• 2 Public procedures: Send OTP, Verify OTP
• 12 Protected procedures: Identity, Language, Business (3), Warehouse (3), Equipment (2), Bank (2)
• 1 Atomic transaction: Popper Registration Submit (8 simultaneous inserts)

File: /src/server/api/routers/popper.ts

export const popperRouter = t.router({ // Steps 1-2: Public Auth (OTP) sendPopperMobileOtp: publicProcedure, verifyPopperMobileOtp: publicProcedure, // Steps 3-13: Protected Setup Wizard submitPopperIdentity: protectedProcedure, selectLanguage: protectedProcedure, submitBusinessDetails: protectedProcedure, submitBusinessLegalDocuments: protectedProcedure, submitBusinessAddress: protectedProcedure, submitWarehouseAddress: protectedProcedure, submitWarehouseDetails: protectedProcedure, submitWarehouseGeofencing: protectedProcedure, submitPoppingCapability: protectedProcedure, submitAncillaryEquipment: protectedProcedure, submitBankAccount: protectedProcedure, verifyBankAccount: protectedProcedure, // Step 14: Atomic Review & Submit submitPopperRegistration: protectedProcedure, });

Equipment Procedures (Steps 11-12)

Step 11: submitPoppingCapability
Input Schema: { warehouse_id: UUID, chula_count: number (0-50), chula_throughput_kg_per_day: number (5-10), machine_count: number (0-20), machine_throughput_kg_per_day: number (50-250), unit_count: number (0-10), unit_throughput_kg_per_day: number (500-2000), equipment_photos: File[] (max 5), producible_grades: string[] (SUTA6P, SUTA5, SUTA45, MURRAH, CHIPTA), has_qa_capability: boolean, qa_methods: string[] (MANUAL, MOISTURE, SORTING, ICAR_CERTIFIED) } Output Schema: { success: boolean, warehouse_id: UUID, daily_capacity_kg: number (auto-calculated), equipment_verified: boolean } Validation Rules: • At least one equipment type required (chula_count > 0 OR machine_count > 0 OR unit_count > 0) • Daily capacity = (chula_count × chula_throughput) + (machine_count × machine_throughput) + (unit_count × unit_throughput) • Producible grades: multi-select, at least one required • QA methods: only validated if has_qa_capability = true • Photos: uploaded to Supabase bucket popper-equipment-photos DB Write: warehouses table: Update record with equipment inventory + daily_capacity_kg + equipment_photos (JSONB)
Step 12: submitAncillaryEquipment (Optional)
Input Schema: { warehouse_id: UUID, ancillary_equipment: { pre_popping: { cleaning_winnowing: boolean, drying: boolean, pre_roast_grading: boolean }, roasting: { sand_roaster: boolean, rotary_roaster: boolean }, post_popping: { de_shelling: boolean, polishing: boolean, post_pop_grading: boolean }, value_add: { flavor_coating: boolean, packaging_vacuum: boolean, packaging_nitrogen: boolean } }, ancillary_equipment_photos: File[] (max 5) } Output Schema: { success: boolean, warehouse_id: UUID, facility_score_boost: number (0-15 points), skip_confirmed: boolean } Scoring Logic: • +1 point per equipment type available (max 12 base) • +3 bonus for rare combination (e.g., all 4 categories present) • Max 15 points total Skip Behavior: • If user skips: skip_confirmed = true, facility_score_boost = 0 • Continue to next step (Bank Account Details) DB Write: warehouses table: Update record with ancillary_equipment (JSONB) + ancillary_equipment_photos + facility_score_boost

Branching Logic (Step 2 - Mobile Verification OTP)

  • If OTP matches EXISTING popper: Load popper detail screen (Update mode)
  • If OTP for NEW user: Continue setup wizard (Create mode)

Atomic Transaction (Step 14 - Review & Submit)

  • 1. Insert user record (role=popper, status=pending_approval)
  • 2. Insert popper_profiles
  • 3. Insert business_profiles
  • 4. Insert warehouses (first warehouse)
  • 5. Insert user_documents (business docs + warehouse ownership)
  • 6. Insert user_bank_accounts (encrypted)
  • 7. Insert popper_approval_queue
  • 8. Insert popper_registration_events (audit trail)
  • → On success: Trigger async Inngest jobs
  • → On failure: Rollback all 8 inserts (no partial state)

React Component Decomposition & Reusability

Popper Registration breaks into reusable React components across the Scout module's 20 workflows. Follow atomic design principles: atoms (inputs) → molecules (form groups) → organisms (form sections) → templates (screens).

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% of components shared across 20 workflows
• Variant management: Props define UI variants, not separate components

State Management (Context + Local)

Global (RegistrationContext): • wizard_id: string • currentStep: number (1-15) • formData: { [step]: value } (wizard-wide state) • errors: { [field]: string[] } (validation errors) • loading: boolean • savedDraft: boolean Local (useRegistrationForm hook): • Persists formData to AsyncStorage (mobile) • Auto-saves after 3s of inactivity • Manages optimistic UI (loading, disabled CTAs) • Debounces API calls on Continue Per-Screen (Local state): • File uploads (before compression) • Date picker modal open/close • Dropdown open/close • Temporary focus states

Consistency Checklist

  • ✓ Props Interface: All components export TypeScript interfaces for props
  • ✓ Error Handling: All inputs display errors consistently (red border + error message below)
  • ✓ Spacing: Use CSS variables (--spacing-xs, --spacing-sm, --spacing-md) not hardcoded px
  • ✓ Colors: Use CSS variables (--accent-primary, --text-main) from root design system
  • ✓ Validation: Frontend validation mirrors backend Zod schemas (shared-types)
  • ✓ Accessibility: Labels paired with inputs, ARIA attributes for complex components
  • ✓ Mobile-First: Responsive breakpoints: 360px, 600px, 1024px
  • ✓ No Duplication: Extract common patterns into reusable components immediately
  • ✓ Variant Props: Use variant/size/state props, not separate components
  • ✓ No Local State: Business logic state lives in Context, component state only for transient UI

⚠️ Encryption:
• GSTIN: AES-256-GCM encrypted (sensitive business identifier)
• Business legal name: AES-256-GCM encrypted at-rest
• Bank account numbers: AES-256-GCM encrypted at-rest
• Account holder names: AES-256-GCM encrypted
• OTP: Hashed with bcrypt (never stored plaintext)
• IFSC code: NOT encrypted (public information)
• FSSAI: Masked in logs (last 4 digits only)

• Backend always validates (never trust frontend)
• Frontend uses same schemas for UX feedback

• Return 400 BAD_REQUEST with field-level errors

1. popper.send-otp-sms (Fire-and-Forget)

Trigger: Step 2 (Mobile Verification OTP)
Steps: Generate 6-digit → Hash with bcrypt → Store in DB → Send via Twilio SMS → Log
Retry: 4 attempts with exponential backoff (1s, 5s, 30s)
DLQ: Move to dead-letter queue after 4 failures, alert ops

2. popper.penny-drop-verification (Wait-for-External)

Trigger: Step 13 (Bank Account Details)
Steps: Initiate Razorpay penny-drop → Store job ID → Wait for webhook (max 5 min) → Verify match → Update status
Webhook: POST /api/webhooks/razorpay/verification-complete

3. popper.registration-submitted (Chained)

Trigger: Step 14 (Review & Submit)
Steps: Verify popper exists → Send confirmation SMS (in preferred_language) → Notify Admin (dashboard alert)
Dependencies: Ensure popper record created before SMS

Security Best Practices

Sensitive Data Handling

Never Log:
✗ Full mobile numbers (log only last 4: ***6789)
✗ GSTIN or business legal name
✗ PAN number
✗ Full account numbers (log hash or last 4)
✗ FSSAI numbers (log hash only)
✗ Tokens or OTP
✗ Encryption keys or secrets

Safe to Log:
✓ Last 4 digits of mobile (padded: ***6789)
✓ District/state (non-identifying)
✓ Popper type (processor, trader, warehouse)
✓ Approval status (pending, approved, rejected)
✓ Event types and timestamps
✓ Warehouse code (WARE-1024 format)

Data Encryption

FieldMethodRotation
GSTINAES-256-GCMNever (stable)
Business Legal NameAES-256-GCMNever (stable)
Bank Account NumberAES-256-GCMNever (stable)
Account Holder NameAES-256-GCMNever (stable)
OTPbcrypt 10 roundsN/A (hashed, never retrieved)

OTP Security

  • Generate 6-digit OTP locally (cryptographically random)
  • Hash with bcrypt (10 rounds) before storage
  • Set expiration: 10 minutes
  • Max 5 attempts before blocking (10-min cooldown after 5 failures)
  • Compare using bcrypt.compare() (constant-time, timing-attack resistant)
  • Resend cooldown: 30 seconds enforced server-side

CORS & Security Headers
