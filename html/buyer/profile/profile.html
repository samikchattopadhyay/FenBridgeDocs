Screen 01 Buyer Profile Hub
buyer_profile
High-fidelity screen capture unavailable. Refer to source code directly.
Purpose
Central hub for buyer to view and manage account, business info, delivery addresses, and settings.
Business Requirement
Single source of truth for buyer's account data. Allows quick edits and navigation to detailed management screens.
User Experience (UX)
Single scrollable screen with avatar, editable key fields, status/approval info, and menu buttons.
User Actions
  • Tap avatar to upload/change profile photo.
  • Tap name, mobile, or address to inline-edit (or open edit sheet).
  • Tap menu buttons to navigate to detailed screens (Edit Business, Delivery Addresses, etc.).
  • Tap Logout to sign out.
Technical Details
  • Avatar upload: max 5 MB, JPEG/PNG, stored at fenbridge/users/{user_id}/avatar.jpg.
  • Inline edits (name, mobile) auto-save on blur + brief toast.
  • Approval status: fetch users.status (pending_approval / active / suspended).
  • Trust score: computed server-side, cached, updated daily. Fetch from users.trust_score (nullable).
  • Primary address: fetch from user_addresses where is_primary = true.
  • WhatsApp status: check whatsapp_links table for this user.
Design Details
Header Section (surface-hi background, top padding 16 dp): - Avatar: 120 × 120 dp circle, surface-hi border with accent outline. Camera icon corner. Tap to upload. - Name: Title 2, text-hi, tap to edit (inline or sheet). - Role badge: "Buyer" (pill, info colour). - Edit profile link (small secondary button with pen icon). Section 1 — Contact & Location (surface card, border-hi, 12 dp padding): - Label: "Contact & Location" (Title 3, accent). - Row 1: Mobile icon, "+91 98765 43210", verified badge (✓ accent), tap to edit, edit icon. - Row 2: WhatsApp icon, "Not linked" (text-md) OR "+91 98765 43211" (if linked), "+ Link" button (info) OR edit icon (if linked). - Row 3: Location icon, primary address (2-line truncate), edit icon. Section 2 — Business Information (surface card): - Label: "Business Information" (Title 3, accent). - Row 1: Business name, edit icon. - Row 2: GSTIN (if set) or "Not provided" (text-lo), edit icon. - Row 3: PAN (if set) or "Not provided", edit icon. - Row 4: Divider. - Row 5: "Uploaded Documents" link (info colour). Tap to view gallery. Section 3 — Account Status (surface card): - Label: "Account Status" (Title 3, accent). - Row 1: Status pill. If pending_approval: ⏳ "Pending Approval" (warn colour), helper text "Expected within 24 hours". If active: ✓ "Active" (accent colour). If suspended: ✗ "Suspended" (danger colour), helper text "Contact support". - Row 2: Trust score (if available): "⭐ 4.5 / 5.0 — Established Buyer" (info colour). If new: "ⓘ Your trust score will appear after your first order." - Row 3: Linked account: "Google" (text-md), edit icon (for future multi-account). Section 4 — Menu Buttons (stacked, gap 8 dp): - "Edit Business Details" (secondary button, full-width, icon ✎). - "Delivery Addresses" (secondary button, full-width, icon 📍). - "Settings" (secondary button, full-width, icon ⚙️). - "Notification Preferences" (secondary button, full-width, icon 🔔). - "Help & Support" (secondary button, full-width, icon ❓). - "Logout" (destructive button, full-width). Bottom padding: 24 dp.
Database Schema
users ( id UUID PRIMARY KEY, email VARCHAR(255) NOT NULL UNIQUE, full_name VARCHAR(200) NOT NULL, mobile_number VARCHAR(20), mobile_verified_at TIMESTAMP, avatar_url VARCHAR(500), role ENUM ['buyer', 'farmer', 'scout'] NOT NULL, business_name VARCHAR(255), gstin VARCHAR(15), pan VARCHAR(10), status ENUM ['onboarding', 'pending_approval', 'active', 'suspended'] DEFAULT 'pending_approval', trust_score DECIMAL(3, 1), trust_tier ENUM ['new', 'established', 'trusted'], theme_preference ENUM ['dark', 'light'] DEFAULT 'dark', preferred_language VARCHAR(10) DEFAULT 'en', approval_status_updated_at TIMESTAMP, created_at TIMESTAMP )

Screen 03 Delivery Addresses
delivery_addresses
High-fidelity screen capture unavailable. Refer to source code directly.
Purpose
Manage all delivery locations for the buyer. Set primary address (used as default in new POs).
Business Requirement
Buyers may have multiple warehouses. Each PO specifies a delivery address. Primary address is the default.
User Experience (UX)
List of addresses with options to set as primary, edit, or delete. FAB to add new.
User Actions
  • View all saved addresses.
  • Tap "Set as Primary" on any address.
  • Tap address row to view/edit details.
  • Tap delete icon to remove address.
  • Tap FAB to add new address.
Technical Details
  • Fetch all user_addresses where user_id = current_user.
  • Only one address can be is_primary = true at a time.
  • Add/edit/delete operations update user_addresses table.
  • Geolocation optional: reverse-geocode if available during add/edit.
Design Details
Top bar: Back button, "Delivery Addresses" heading (Title 2). List of address cards (surface background, border-hi, 12 dp padding, gap 8 dp): Each card: - Address line 1 + line 2 (Body strong, 2-line truncate). - City, postal code (Body, text-md, 8 dp top margin). - If primary: "⭐ Primary" badge (accent colour, pill shape, top-right corner). - Action row: "Set as Primary" button (if not primary, info colour, secondary) | edit icon (pen, secondary button) | delete icon (trash, danger colour, secondary button). FAB: "+ Add Address" (accent colour, 56 dp diameter, bottom-right corner). Empty state: "No addresses yet. Add your first delivery location." + centered "Add Address" button.
Database Schema
user_addresses ( id UUID PRIMARY KEY, user_id UUID FOREIGN KEY, address_line VARCHAR(500) NOT NULL, city_district VARCHAR(100) NOT NULL, postal_code VARCHAR(10), latitude DECIMAL(10, 8), longitude DECIMAL(11, 8), is_primary BOOLEAN DEFAULT false, created_at TIMESTAMP )

Screen 06 Link WhatsApp (Common Flow)
link_whatsapp
High-fidelity screen capture unavailable. Refer to source code directly.
Purpose
Link user's WhatsApp number for future SMS/WhatsApp notifications and direct support contact.
Business Requirement
WhatsApp is widely used in India. Linking allows FenBridge to send updates via WhatsApp (v1.1), increasing engagement.
User Experience (UX)
Input field for WhatsApp number. Verification via OTP. Confirmation.
User Actions
  • Tap "Link WhatsApp" button (from Profile or Settings).
  • Enter WhatsApp number (pre-filled with registered mobile if same).
  • Send OTP via WhatsApp.
  • Enter OTP to verify.
  • Confirmation: "✓ WhatsApp linked."
Technical Details
  • WhatsApp number input: accepts +91 format or 10-digit format, validates pattern.
  • OTP sent via Twilio WhatsApp API or WhatsApp Business API (v1.1).
  • In v1.0, can skip OTP and just store the number (less critical).
  • Stored in whatsapp_links table, linked to user_id.
Design Details
Triggered from: Profile > Contact section > WhatsApp row > "+ Link" button. Step 1 — Enter Number: - Top bar: Back, "Link WhatsApp". - Heading: "Add your WhatsApp number" (Title 2). - Input: "WhatsApp number" (pre-filled with registered mobile, editable). Step 2 — Verify OTP: - Heading: "Verify your WhatsApp number" (Title 2). - Display: "We sent a code to +91 98765 43210". - OTP input: three boxes (auto-advance), or manual entry. - On verification: auto-submit. - Resend link after 30 sec. Step 3 — Confirmation: - Heading: "WhatsApp linked!" (Title 2, accent). - Text: "✓ You'll now receive updates on WhatsApp." - Button: "Go Back" (routes back to Profile).

Screen 08 Notification Preferences
notification_preferences
High-fidelity screen capture unavailable. Refer to source code directly.
Purpose
Fine-grained control over which events trigger notifications and via which channels.
Business Requirement
Users should control notification fatigue. Different events may be important to different users.
User Experience (UX)
List of event categories, each with toggles for in-app/push/SMS channels. Grouped by type (Order, Payment, System, etc.). Auto-save on toggle.
User Actions
  • View all notification event types.
  • Toggle on/off per channel for each event.
  • Implicit save triggers on toggle state commit.
Technical Details
  • Fetch users.notification_preferences JSON.
  • Structure: { "order_placed": { "in_app": true, "push": true, "sms": false }, "payment_due": {...}, ... }.
  • On toggle, update specific key-value pair and save to DB.
  • Default: all in-app and push enabled, SMS disabled for most events.
Design Details
Top bar: Back button, "Notification Preferences" heading (Title 2). Intro text: "Control which updates you receive and how." (Body, text-md, 16 dp top padding). Event Groups: Order Updates (only for Buyer) / Bid Updates (only for Farmer): - Event: "New requirement posted" → three toggles (in-app | push | SMS). - Event: "Bid awarded" / "Order confirmed". - Event: "Dispatched" → three toggles. - Event: "Delivered" → three toggles. Payment Updates: - Event: "Payment due" → three toggles. - Event: "Payment received" → three toggles. Account Updates: - Event: "Account status changed" → toggles. - Event: "Approval completed" → toggles. System Notifications: - Event: "New features" → toggles. - Event: "Promotional offers" → toggles. Each event row: event name (Body), three small toggles (in-app | push | SMS) aligned right. Toggle indicators: circular, 40 × 24 dp, accent when on, border when off. No Save button (auto-saves).
Database Schema
// Updating persistent state map JSONB property atomically: UPDATE users SET notification_preferences = jsonb_set(notification_preferences, '{order_placed,push}', 'false') WHERE id = current_user;
