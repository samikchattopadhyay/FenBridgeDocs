Screen 1QR Scan / Load Confirmation
load_confirmation
Purpose
Ensures the specific 'Sealed' lot is being loaded onto the matched truck (Section 6.1).
Business Requirement
  • Scout must scan the Seal ID on the stack before it enters the truck.
  • System verifies that the truck plate matches the PO logistics plan.
User Experience (UX)
QR scanner view. Real-time validation card showing "Truck Matched" or "Error: Wrong Truck". Truck Driver ID entry.
User Actions
  • Scan Seal.
  • Enter Driver Phone.
  • Confirm Match.
Technical Details
Cross-references `lots.seal_id` and `logistics_orders.truck_plate`. Updates status to 'loading'.
Design Details
  • Status badges (Green for Match, Red for Conflict).
  • Large numeric input for driver phone.
Database Schema
UPDATE logistics_orders SET status = 'loading', loaded_at = NOW() WHERE lot_id = $1;

Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

Dispatch Logistics is a 3-screen workflow that interlock seal verification, evidence collection, and departure confirmation. Each screen ensures correct truck loading, weight validation via OCR, and terminal state transition to 'In-Transit' status.

Screen Decomposition & Component Usage

Screen 1: QR Scan / Load Confirmation

Seal ID verification against truck match.

Component Stack:
TopAppBar → StepperPanel → [QRCodeScanner] → [StatusBadge (Match/Error)] → [ValidationCard (Truck Plate, Driver)] → TextInput (Driver Phone) → CTAFooter (Confirm)

Screen 2: Weight Bridge Slip / Truck Photos

Evidence capture with OCR-powered weight recognition.

Component Stack:
TopAppBar → StepperPanel → [PhotoCaptureCard (mandatory x 3: slip, truck rear, plate)] → [OCRExtractedWeight (auto-fill)] → NumberInput (verified_weight) → CheckboxInput (Weight confirmed) → StatusBadge (Within tolerance?) → CTAFooter (Next)

Screen 3: Departure Confirmation

Terminal state transition with attestation.

Component Stack:
TopAppBar → StepperPanel → [SummaryCard (Truck, Destination, ETA)] → SelectInput (Destination) → ConfirmationCheckbox (Attestation) → [StatusBadge (Ready to dispatch)] → CTAFooter (Mark Dispatched - disabled until checked)

State Management & Data Flow

Local Wizard State (Client-side):
• seal_id: string (scanned from QR)
• truck_plate: string (matched from PO)
• driver_phone: string
• evidence_photos: { slip_url, truck_url, plate_url }
• extracted_weight: number (from OCR)
• verified_weight_confirmed: boolean
• destination: string
• attestation_checked: boolean

Backend Atomic Transaction (on Screen 3 Submit):
1. UPDATE lots SET status = 'in_transit'
2. INSERT logistics_evidence (photos)
3. Trigger webhook to Buyer (escrow hold)
4. INSERT audit_trail (departure_confirmed_at)

Common Patterns & Reuse

  • QR Scanning: Screen 1 uses QRCodeScanner. Extract to QRScannerComponent for reuse in other inventory flows.
  • Photo Evidence: Screens 1 & 2 capture mandatory photos. Centralize photo compression and upload logic in PhotoService.
  • Weight Validation: Screen 2 OCR + tolerance check. Create a reusable validateWeight utility for any lot/weight verification.
  • Status Progression: All 3 screens use StatusBadge for real-time validation feedback. Use Context to persist wizard state.
