tRPC Procedures (5 Screens)

This dashboard manages offline SQLite mutations queues, processing payloads sequentially.

State Operations:
• Changes register inside local tables. Synced items move to historical archives upon resolution.

Key Procedures

  • getSyncQueue — Protected. Fetches pending offline changes from SQLite buffers.
  • processSyncQueue — Protected. Atomic mutator. Sequentially runs transactions against PostgreSQL tables.
  • resolveSyncConflict — Protected. Applies merge strategies (e.g. Server-Wins, Client-Wins).

React Component Decomposition & Reusability

Sync Queue workflow decomposes into reusable React components following atomic design principles. This is a specialized offline-sync dashboard (not a form), so component patterns differ from registration/inspection workflows.

✅ Design Philosophy — Offline-First Sync Dashboard:
• Single Responsibility: Each component inspects/controls ONE aspect of sync state
• Prop-based composition: Components accept queue items, payloads, conflicts as data
• Read-only payload inspection: Displays JSON/metadata without mutations
• Conflict resolution UI: Dual-pane comparison with resolution strategy selector
• Reuse across workflows: 90% shared components (TopAppBar, StatusBadge, RadioButton, CTAFooter)
• Variant management: Props define UI variants (pending/synced/failed/conflict states)

Critical: Zero Duplication Warnings

❌ WRONG: Create "QueueStatusBadge" (similar to StatusBadge)
✅ RIGHT: Use existing StatusBadge + SyncStatusBadge variant for pending/synced/failed/conflict states. SyncStatusBadge is workflow-specific because it requires special sync logic. StatusBadge handles generic approval/rejection/review states across other 19 workflows.

❌ WRONG: Create "SyncItemListCard" + "QueueItemCard" (both show queued items)
✅ RIGHT: Use single QueueItemCard with flexible props for all queue items in Screen 1.

❌ WRONG: Create "ClientValueDisplay" + "ServerValueDisplay" for conflict pane
✅ RIGHT: Use ClientServerComparisonPane (single component with props for left/right values).

❌ WRONG: Create "SyncRetryButton" separate from CTAFooter
✅ RIGHT: Use CTAFooter with primary="Retry Sync" action prop. RetryButton is micro-component used within footer, not standalone.

❌ WRONG: Create specialized "CodeBlock" just for sync payloads
✅ RIGHT: Use CodeViewport (shared across all workflows that display code: payload inspection, batch results, API responses).

Sync Queue Specific UX Pattern: Read-Only Inspection + Conflict Resolution

Unlike registration/inspection workflows:
• No form inputs: Screens 2–3 display data, never collect user input (except conflict strategy selection)
• Payload transparency: JSONviewport shows exact payloads queued for sync (debugging, diagnostics)
• Conflict handling: Screen 3 is not a form—it's a resolution UI where scout picks strategy (client-wins, server-wins, manual)
• Limited actions: Only "Retry Sync" (Screen 2) and "Resolve Conflict" (Screen 3)—no edit/create/delete on payloads
• Network awareness: Connection status always visible (Screen 1), informs retry logic

Component Usage Guidelines

When to Create a New Component:
✓ Logic/JSX repeats 2+ times
✓ Testable in isolation
✓ Clear, single responsibility
✗ Used in only 1 workflow (inline)
✗ Just a styling wrapper (use CSS)
Sync Queue: Always Inline (Not Components):
• Queue list header layout (use Flexbox + CSS)
• Conflict field comparison labels (use styled spans)
• Metric card grid (use CSS Grid)
Always Reuse (Never Create Variants):
• TopAppBar — all workflows
• StatusBadge — approval/rejection/generic states (use variant prop)
• SyncStatusBadge — pending/synced/failed/conflict (workflow-specific, but reused within Screen 1 multiple times)
• RadioButton — conflict strategy, language selection, other radio groups (use variant prop)
• CTAFooter — all screens with actions (use variant prop for button labels)

Database Schemas (Drizzle ORM DDL)

sync_queue (SQLite / Offline Cache Ledger)

CREATE TABLE sync_queue ( id TEXT PRIMARY KEY, scout_id TEXT NOT NULL, endpoint TEXT NOT NULL, payload TEXT NOT NULL, priority INTEGER NOT NULL DEFAULT 1, status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','synced','failed','conflict')), error_log TEXT, created_at TEXT NOT NULL, synced_at TEXT );

Strict Input Validations & Error Enforcements

  • Idempotency Keys: Required on every queue entry to prevent duplicate PostgreSQL writes during retries.

Offline Caching & Queue Synchronization

  • Background Syncs: Scheduled jobs trigger uploads in background modes when internet connectivity returns.

Security & Media Protection Protocols

  • Payload Encryption: Payloads are stored securely in device memory.
