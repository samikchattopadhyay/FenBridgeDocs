Scout Operations

Registered Users Directory

Comprehensive directory of all farmers and poppers registered by the scout, with unified search, filtering by role and status, and detailed user management screens with multiple tabs.

Scout can browse all registered farmers and poppers, search by name/mobile/district, filter by role and approval status, and access detailed user profiles with tabs for identity, assets (ponds/warehouses), payment methods, documents, and activity history.

Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (3 Screens)

The Users Directory consists of 3 primary screens with supporting procedures.

Screen Breakdown:
• Screen 1: Combined Users List (search, filter, browse)
• Screen 2: Farmer Detail (5 tabs: Profile, Ponds, Bank, Documents, Activity)
• Screen 3: Popper Detail (5 tabs: Profile, Warehouses, Bank, Documents, Activity)

Key Procedures

  • listRegisteredUsers — Protected. Returns farmer_profiles + popper_profiles joined with users. Supports role filter (farmer/popper/all) + status filter (pending_approval/active/suspended) + search (name/mobile).
  • getFarmerDetail — Protected. Hydrates farmer_profiles + ponds + user_bank_accounts + user_documents + activity aggregation.
  • getPopperDetail — Protected. Hydrates popper_profiles + warehouses + user_bank_accounts + user_documents (business-only) + activity.
  • searchUsers — Protected. Full-text search on name, mobile, district. Client-side or server-side implementation.

React Component Decomposition & Reusability (3 Screens)

This directory workflow breaks into 14 workflow-specific components and 9 shared components. List screen optimized for search/filter/role-selection, detail screens use 5-tab navigation for comprehensive user profile management.

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% of components shared across 20 workflows
• Variant management: Props define UI variants, not separate components
📖 Cross-Workflow Component Library
All scout workflows share a unified component library. See COMPONENT_ARCHITECTURE_GUIDE.md for the master reference documenting all 20 workflows and their component usage.

Core Database Tables (Drizzle ORM)

farmer_profiles (Farmer Identity)

CREATE TABLE farmer_profiles ( user_id UUID PRIMARY KEY REFERENCES users(id), full_name TEXT NOT NULL, dob DATE, district TEXT NOT NULL, panchayat_village TEXT, full_address TEXT, alt_phone TEXT, preferred_language TEXT NOT NULL DEFAULT 'en' );

popper_profiles (Business Identity)

CREATE TABLE popper_profiles ( user_id UUID PRIMARY KEY REFERENCES users(id), contact_name TEXT NOT NULL, business_name TEXT NOT NULL, pan_number TEXT, gstin TEXT, fssai_number TEXT, district TEXT NOT NULL, panchayat_village TEXT, full_address TEXT, alt_phone TEXT, preferred_language TEXT NOT NULL DEFAULT 'en' );

Related Tables

  • users: Base user table with role, status, created_via_scout_id
  • ponds: Farmer's pond details (area, GPS, location)
  • warehouses: Popper's warehouse details (capacity, location, status)
  • user_bank_accounts: Payment methods (encrypted account number, IFSC, penny-drop status)
  • user_documents: Uploaded documents (doc_type, verified_at, storage_path)
  • scout_user_assignments: Explicit assignment of users to scouts for management

Registered Users Directory Workflow
3 Screens
1Combined Users List
Purpose

Single unified directory for all farmers and poppers the scout has registered or been assigned to manage.

Provides rapid searching, filtering by role and approval status, and quick navigation to user profiles.

Business Requirement

Scope is limited to users where created_via_scout_id = self OR explicitly assigned via scout_user_assignments.

Must distinguish between all user roles (Farmer, Popper, Buyer) and provide filtering by status (Pending Approval, Active, Suspended).

Supports launching new farmer/popper registration from FAB.

User Experience (UX)

Top bar with search field (placeholder: "Search by name, mobile, district...").

Below: four role filter chips (All / Farmer / Popper / Buyer, default All) and three status filter chips (Pending Approval [amber], Active [accent], Suspended [danger]).

User list shows rows with avatar/initials (teal for farmer, purple for popper), name, role pill, mobile, district, last activity timestamp, status pill.

Floating action button (FAB) bottom-right: "Register New User" + icon.

User Actions
Search by name, mobile, or district (any field, client or server-side)→filters list. Tap role chip to filter (All / Farmer / Popper / Buyer). Tap status chip to narrow (Pending Approval / Active / Suspended). Tap user row→navigates to Farmer Detail or Popper Detail (role-dependent). Tap FAB→role picker dialog→"Register a Farmer" or "Register a Popper" wizard.
Technical Details

Query: SELECT users.*, farmer_profiles.* FROM users LEFT JOIN farmer_profiles ON users.id = farmer_profiles.user_id LEFT JOIN popper_profiles ON users.id = popper_profiles.user_id WHERE created_via_scout_id = self OR id IN (SELECT user_id FROM scout_user_assignments WHERE scout_id = self).

Real-time subscription on users table.

Search: full-text on name + mobile (client or server).

Filters: applied post-fetch or as part of query (role determined by presence in farmer_profiles vs popper_profiles; status from users.status).

Database Schema
Reads from: users (role, status, created_via_scout_id, mobile), farmer_profiles, popper_profiles, scout_user_assignments. Computes: last activity from max(timestamp) across bids, spot_offers, purchase_orders, payments tied to user_id.
2Farmer Detail Profile (5 Tabs)
Purpose

Comprehensive single-farmer record view, organized into five tabs.

Hub for all "Manage Farmer" sub-actions (profile edits, pond management, bank updates, document uploads, activity viewing).

Shows complete audit trail and enables field-by-field edits.

Business Requirement

Every farmer registered or assigned to the scout must have full audit trail and edit capability.

Five-tab layout separates concerns: identity (Profile), production assets (Ponds), payment method (Bank), compliance (Documents), transaction history (Activity).

Pending Approval status prominently warned — farmer cannot transact until approved.

User Experience (UX)

Header card: small avatar/initials (teal bg), farmer name (Title 2), status pill (Pending Approval [amber] / Active [accent] / Suspended [danger]), mobile, district, optional alt phone.

If Pending Approval: full-width amber banner above tabs ("Awaiting Admin approval — farmer cannot transact until approved").

Below: five equal-width tabs (Profile / Ponds / Bank / Documents / Activity).

Tab content scrolls independently.

Relevant action buttons in each tab (Edit, Add Pond, Upload Document, etc.).

Tab Details
Profile: Full Name, DoB, District, Village, Address, Alt Phone, Preferred Language. Edit button→Edit Identity workflow.
Ponds: List of pond cards (code, name, area, last inspection date, status pill). Tap row→Pond Detail. "+" button→Add Pond. Edit/Deactivate via menu.
Bank: Account holder name, masked account number, IFSC, bank name, penny-drop status pill. Edit button→Edit Bank Account. Retry button (if failed).
Documents: Tiles by type (Aadhaar, PAN, Land Record, Pond Lease, etc.). Upload date, verified badge. Tap to view. Camera icon→Add/Replace Documents.
Activity: Chronological feed (newest first) of transactions (bids, spot offers, POs, payments). Timestamp, activity summary. View-only.
Database Schema
Reads: users, farmer_profiles, ponds, pond_inspections (last 5-10 per pond), user_bank_accounts, user_documents, bids, spot_offers, purchase_orders, payments.
3Popper Detail Profile (5 Tabs)
Purpose

Comprehensive popper (buyer/processor) record, mirroring Farmer Detail but with business-centric focus.

Five tabs organize business identity, warehouse assets, payment method, legal compliance, and transaction history.

Business Requirement

Poppers are businesses with named contact person, multiple warehouses, legal documentation (PAN, GSTIN, FSSAI), and bank accounts.

Warehouses tab links to existing warehouse records.

Pending Approval state warned prominently — popper cannot transact until approved.

User Experience (UX)

Identical pattern to Farmer Detail: header card with popper business name, contact person, status, mobile, district.

If Pending Approval: amber warning banner.

Tab row: Profile / Warehouses / Bank / Documents / Activity.

Tab content includes role-appropriate actions (add warehouse, edit business, manage GSTIN/FSSAI certs, etc.).

Tab Details
Profile: Business Name, Contact Person Name, PAN (masked), GSTIN (if present), FSSAI (if present), Address. Edit button→Edit Business Details.
Warehouses: List of warehouse cards (code, name, capacity, district, village, status pill). Tap→Warehouse Detail. "+" button→Add Warehouse. Edit/Deactivate via menu.
Bank: Identical to Farmer Detail — account holder, masked number, IFSC, penny-drop status. Edit/Retry buttons.
Documents: Business-specific: Business PAN (mandatory), GSTIN Cert, FSSAI Cert, Warehouse Lease. Upload date, verified badge. Tap to view. Camera icon→Add/Replace Documents.
Activity: Chronological feed (newest first) of popper transactions (spot offers, POs, lot seals, payments, verifications). Tap to drill into detail.
Database Schema
Reads: users, popper_profiles, warehouses, user_bank_accounts, user_documents (business-only: PAN, GSTIN, FSSAI, warehouse lease), spot_offers, purchase_orders, lot_seals, payments.
