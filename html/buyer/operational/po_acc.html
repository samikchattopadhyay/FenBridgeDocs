Step 1Review PO Draft
PO_ACCEPTANCE/review_po_draft
Purpose
Present the full draft PO to the buyer for comprehensive review covering lot specifications, dynamic pricing summaries, delivery schedules, and payment milestone configurations.
Business Requirement
Enforces the Zero-Leak privacy isolation standard. Strips original farmer names, source ponds, and platform procurement pricing matrices completely. Implements a 72-hour hard expiry timer.
Technical Details
Loads purchase orders via SELECT * FROM purchase_orders WHERE id = ? AND state IN ('draft','pending_acceptance'). Subscribes to real-time status deltas server-side.
Database Security Masking
-- Secure view stripping counterparty identities: CREATE VIEW v_po_buyer AS SELECT id, bid_id, buyer_id, lot_id, grade, quantity_kg, unit_price_per_kg, sub_total, grand_total, buyer_receivable, state, pending_acceptance_expires_at FROM purchase_orders;

Step 2Insurance Options
PO_ACCEPTANCE/insurance_options
Purpose
Configure dynamic liability insurance riders covering raw seed transit degradation or unforeseen moisture absorption events prior to arrival. Allows overriding baseline requirement presets.

Step 4Digital Signature Acceptance
PO_ACCEPTANCE/digital_signature
Purpose
Provide legally compliant client-side hand drawing or corporate stamp inputs binding signing executive keys straight to finalized order ledgers.
Business Requirement
Complies with Section 5 of the Information Technology Act, 2000. Uses standard mobile OTP cryptographic hashes supplemented by optional canvas thumbnail drawings.
Technical Details
Validates generated challenges via backend RPC verify_otp(user_id, otp_code) while logging source IP signatures and client user agents to secure non-repudiation.
Database Schema
-- Persists cryptographically verified challenge arrays: digital_signatures ( id UUID PK, po_id UUID FK, signed_by_user_id UUID, signer_role ENUM, otp_verification_id UUID, verified_mobile TEXT, ip_address INET, handwritten_signature_path TEXT, signed_at TIMESTAMP DEFAULT now() );

Step 5Advance Escrow Allocation
PO_ACCEPTANCE/advance_payment
Purpose
Present explicit escrow routing account parameters itemizing required initial down payments required to lock active farmer production commitments.
Business Requirement
Outlines static Current Account transfer structures while awaiting verified external netbanking UTR rows. Directs 15% advance holdings to independent Castler/Escrowpay escrows.
Technical Details
Atomic table writes mark order state as Accepted with advance_awaited sub-status. Triggers asynchronous background Edge Functions rendering final immutable PDF contract documents.
Database Schema
-- Captures payment intent tracking models: payments ( id UUID PK, payer_party ENUM, po_id UUID FK, buyer_id UUID, leg ENUM, expected_amount NUMERIC, utr TEXT, bank_screenshot_path TEXT, status ENUM DEFAULT 'pending', escrow_held BOOLEAN DEFAULT true );
