Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (5 Screens)

This dashboard manages offline farmer properties and physical pond allocations.

State Operations:
• Profile edits save directly. Bank changes and pond deactivations log historical audits before committing mutations.

Key Procedures

  • updateFarmerProfile — Protected. Mutates identity details.
  • updateFarmerBankDetails — Protected. Updates bank rows, sets approval to pending, and schedules checks.
  • upsertPondAsset — Protected. Saves pond edits or appends new coordinates.
  • togglePondStatus — Protected. Sets pond state.

React Component Decomposition & Reusability

This 5-screen farmer portfolio management workflow breaks into reusable React components following atomic design principles. Focus on profile edits, bank verification, and pond asset management.

✅ Design Philosophy — Farmer Portfolio Proxy:
• Single Responsibility: Each component manages ONE farmer asset (profile, bank, pond)
• Prop-based composition: Components accept farmer data, ponds list, bank details as props
• Tabbed navigation: TabbedNavigation component switches between Profile/Bank/Ponds without page reload
• Conditional rendering: Different screens for edit vs. read-only modes
• Reuse across workflows: 80% shared components (TopAppBar, TextInput, PhoneInput, SelectInput, CTAFooter)
• Variant management: Props define edit mode, approval status, asset state

Critical: Zero Duplication Warnings

⚠️ DO NOT CREATE DUPLICATE COMPONENTS:
❌ WRONG: Create "FarmerPhoneField" + "FarmerAddressField" (both are form inputs)
✅ RIGHT: Reuse PhoneInput and AddressForm (shared across Farmer Registration, Popper Registration, Manage Farmer).

❌ WRONG: Create "KYCStatusBadge", "BankStatusBadge", "PondStatusBadge" (all are status variants)
✅ RIGHT: Use single StatusBadge component with label prop (e.g., label="KYC", label="Bank", label="Pond").

❌ WRONG: Copy TopAppBar for "FarmerManagementHeader"
✅ RIGHT: Reuse TopAppBar across all 20 workflows—same component, different title/progress props.

❌ WRONG: Create "PondEditCard" + "PondViewCard" (both show pond data)
✅ RIGHT: Use single PondCard with isEditing prop that toggles between view/edit modes.

❌ WRONG: Create "ApprovalPendingBadge" separate from StatusBadge
✅ RIGHT: Use StatusBadge with status prop (e.g., status="pending", status="approved") + use ApprovalBadge only for extended details (reviewer name, date).

Manage Farmer Specific UX Pattern: Tabbed Portfolio Management

Portfolio Proxy Pattern:
• Tabbed Navigation: Profile/Bank/Ponds tabs (no page reloads)
• Edit-in-Place: Each tab has read-only view + edit mode toggle
• Approval Workflow: Changes trigger "pending approval" status
• Conditional Rendering: Verification badges change based on KYC/bank/pond states
• Asset Management: Screen 4 allows add/edit/delete ponds (CRUD on child collection)
• Summary Before Submit: Screen 5 shows all changes in read-only before final submission

Component Usage Guidelines

When to Create a New Component:
✓ Logic/JSX repeats 2+ times (PondCard displayed multiple times)
✓ Testable in isolation (BankEditForm, PondEditModal)
✓ Clear, single responsibility (VerificationBadges handles only badges)
✗ Used in only 1 workflow (inline)
✗ Just a styling wrapper (use CSS)
Always Reuse (Never Create Variants):
• TopAppBar — all 20 workflows
• TextInput — name, bank fields (same component, different labels)
• PhoneInput — farmer/popper/scout phone inputs
• AddressForm — farmer/popper address (shared component)
• StatusBadge — KYC/bank/pond status (use label prop for variants)
• LocationPicker — pond GPS, warehouse location, farmer address
• CTAFooter — all screens (use variant prop for button labels)

Database Schemas (Drizzle ORM DDL)

ponds (Source Asset Modifications)

CREATE TABLE ponds ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), farmer_user_id UUID NOT NULL REFERENCES users(id), pond_code TEXT NOT NULL UNIQUE, name TEXT NOT NULL, area_value NUMERIC(5,2) NOT NULL, area_unit TEXT NOT NULL DEFAULT 'acres', district TEXT NOT NULL, panchayat_village TEXT NOT NULL, gps_coordinates POINT NOT NULL, status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','pending_approval','inactive')), created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

Strict Input Validations & Error Enforcements

  • IFSC Code Format: Checked against expressions.
  • GPS Coordinates: Asserts that location parameters exist within regional borders.

Offline Caching & Queue Synchronization

  • Offline Storage: Edits cache locally in SQLite tables, writing updates sequentially once network resolves.

Security & Media Protection Protocols

  • Scope Access: Restricts operations exclusively to scouts assigned to the target district.
Manage Farmer Steps Directory
5 Steps
1 Edit Identity Profile
Purpose
Edits simple profile fields like spelling, phone numbers, and addresses.
Business Requirement
Restricts editing verified mobile numbers directly without OTP challenges.
User Experience (UX)

Text inputs are pre-filled with active profile data.

Verification badges flag status indicators.

Bottom CTA updates profiles.

Technical Details
Invokes updateFarmerProfile mutations.
2 Replace Bank Account Details
Purpose
Updates bank routing information for offline payouts.
Business Requirement
Requires uploading pictures of cancelled cheques to verify validity before payouts run.
User Experience (UX)

Displays current active banking fields.

Entry panels handle Account and IFSC parameters.

Document cameras log media verification check images.

Technical Details

Invokes updateFarmerBankDetails routines.

Sets status flags back to pending approvals.

3 Edit Pond Parameters
Purpose
Modifies existing pond variables (Area, landmarks, locations).
Business Requirement
Locks changes to pond sizes if bids have already run against their allocations.
User Experience (UX)
Selects target ponds. Dynamic panels handle size and capacity variables. GPS buttons capture active positioning points.
Technical Details
Invokes upsertPondAsset mutations.
4 Add New Pond Asset
Purpose
Appends new ponds to the farmer's active directory profile.
Business Requirement
Requires exact GPS verification matching pond coordinates.
User Experience (UX)

Wizard flows input names, sizes, villages.

GPS trackers lock active coordinate tags.

Reviews and submits new pond registrations.

Technical Details

Generates unique pond_code values.

Invokes upsertPondAsset.

5 Deactivate Pond Asset
Purpose
Temporarily or permanently disables active ponds from market lists.
Business Requirement

Identifies deactivation reasons (e.g.

seasonal changes, water contamination).

User Experience (UX)

Renders warning sheets listing active listing links.

Selection options choose deactivation codes.

Dark amber buttons suspend allocations.

Technical Details
Invokes togglePondStatus transactions to deactivate allocations.
