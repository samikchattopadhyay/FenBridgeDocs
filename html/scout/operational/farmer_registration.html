Farmer Registration Workflow
15 Steps
1Intro & Consent
Purpose
Establishes that the scout is acting on the farmer's behalf and the farmer consents.
Business Requirement
Consent must be explicitly recorded — scout-assisted registration creates a binding profile for the user, so a consent timestamp + scout's identity + farmer's mobile is the audit minimum. The farmer's verbal consent is the source; the scout attests via a checkbox.
User Experience
Single screen with plain-language explanation, consent checkbox, primary CTA.
User Actions
Read consent line aloud to farmer → tick "Farmer has given verbal consent" → Continue.
Technical Details
Wizard context initialised: { role: 'farmer', consent_at: now, scout_id }. No DB write yet.
Design Details
Headline "Register a Farmer" + plain-language explanation of what will be collected. Consent checkbox: "The farmer has given verbal consent to register on FenBridge". Disabled CTA until ticked.
Database Schema
None at this step.
2Mobile Verification — OTP Send
Purpose
Sends OTP to farmer's mobile for verification.
Business Requirement
OTP confirms the mobile entered is real and belongs to the farmer (the farmer's phone must physically receive it). Without verified mobile, the farmer can't later log in themselves.
User Experience
Mobile echoed for confirmation; "Send OTP" CTA; 60s cooldown after send.
User Actions
Confirm mobile → Send OTP → tells farmer to check their phone.
Technical Details
Calls Supabase auth.signInWithOtp with farmer's mobile — but with a flag so the OTP is delivered without creating a session on this device.
Design Details
Mobile chip + "Change" link · Send OTP primary · cooldown timer.
Database Schema
CREATE TABLE user_otp_verifications ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), intended_user_id UUID, mobile TEXT NOT NULL, otp_hash TEXT NOT NULL, expires_at TIMESTAMPTZ NOT NULL, consumed_at TIMESTAMPTZ, attempt_count INT NOT NULL DEFAULT 0, requested_by_scout_id UUID REFERENCES scouts(id), context TEXT NOT NULL CHECK (context IN ('scout_assisted_registration','reverify')), created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_uov_mobile ON user_otp_verifications (mobile, created_at DESC);
3Mobile Verification — OTP Entry
Purpose
Scout enters the OTP read from farmer's phone.
Business Requirement
Six-digit OTP. 3 retries; resend after 30s.
User Experience
Six-box input. Helper line "Read the OTP from the farmer's phone."
User Actions
Enter OTP → auto-verify on 6th digit → Continue.
Technical Details
Verifies via Supabase / Edge Function. On success, user_otp_verifications.consumed_at set. Unified Auth Branching Check: If mobile exists in users table, the app cancels the rest of the wizard and routes directly to that User's Detail profile. If not, the scout proceeds to the new user Onboarding Setup Wizard starting with Screen 5.5.
Design Details
Same OTP pattern as scout login OTP screen, with scout-context helper text.
Database Schema
Updates user_otp_verifications.
4User Identity Details
Purpose
Captures farmer's personal details first to perform dynamic uniqueness verification.
Business Requirement
Name (mandatory), date of birth (optional but recommended for age compliance), mobile (mandatory; must be the farmer's own — used for OTP and SMS), alternate phone (optional).
User Experience
Form fields with helper text emphasising mobile must be the farmer's own.
User Actions
Enter name → DoB → mobile → optional alt phone → Continue.
Technical Details
Mobile validated as 10-digit Indian. Uniqueness check against users.mobile runs on Continue — if already registered, screen blocks with "This mobile is already registered" and a link to that user's record (Admin-controlled escalation if it's truly the same farmer with a duplicate registration attempt).
Design Details
Standard form. Mobile field with country chip. Helper line below mobile: "This is the farmer's mobile — they'll receive OTP and SMS here."
Database Schema
Writes to users (mobile) and farmer_profiles (full_name, dob) at submit.
5Language Selection
Purpose
Sets the farmer's preferred app language.
Business Requirement
Determines language used in SMS notifications, dashboards, and the farmer's own future logged-in sessions. Three options: English / Hindi / Bengali.
User Experience
Three large tiles with native scripts.
User Actions
Tap one → Continue auto-enabled.
Technical Details
Context: preferred_language.
Design Details
Same pattern as scout's own Language Selection (workflow 11). Helper: "We'll send notifications in this language."
Database Schema
Written to farmer_profiles.preferred_language at submit.
6Confirm Role = Farmer
Purpose
Final role confirmation before the role-specific fork begins.
Business Requirement
Catches misclassification at the scout's last opportunity. Locked once past this screen.
User Experience
Read-only card showing "Role: Farmer — produces raw makhana from ponds" + Confirm CTA + Back link to choose Popper instead.
User Actions
Confirm · or Back → Role chooser (return to Register a Popper workflow).
Technical Details
Context: role='farmer'.
Design Details
Single card with role description + Confirm button.
Database Schema
None.
7Address Details
Purpose
Captures farmer's residential address.
Business Requirement
District is mandatory and must come from a controlled list. Village/panchayat is searchable but free text fallback supported.
User Experience
District dropdown · Panchayat/village text or searchable · Full address textarea · optional landmark.
User Actions
Pick district → enter village → enter address → Continue.
Technical Details
Context: { district, panchayat_village, full_address }. Districts loaded from a static reference list.
Design Details
Standard form pattern. Live validation on district required.
Database Schema
Written to farmer_profiles at submit.
8First Pond Details
Purpose
Captures the farmer's first pond's basic details.
Business Requirement
Farmer must have at least one pond at registration. Additional ponds added later via Manage Farmer. Pond name, area + unit, estimated annual capacity.
User Experience
Compact form.
User Actions
Pond name → area + unit (Acre / Bigha / Hectare) → estimated capacity (kg) → Continue.
Technical Details
Context: pond.{name, area_value, area_unit, est_annual_capacity_kg}. Pond code auto-generated server-side (POND-{seq}).
Design Details
Three fields stacked. Unit segmented (Acre/Bigha/Hectare).
Database Schema
Written to ponds at submit.
9First Pond Legal Documents
Purpose
Captures legal proof of pond ownership / lease.
Business Requirement
Pond lease or land record is required for legitimacy. Mandatory at least one document; up to 3 photos per document. Scout uses camera to photograph the physical document the farmer presents.
User Experience
Document type picker → photo capture slots → "Add another document" optional.
User Actions
Pick doc type (Land Record / Pond Lease / Other) → capture up to 3 photos per doc → optionally add more docs → Continue.
Technical Details
Photos compressed, stored locally, uploaded to Supabase Storage on sync. Wizard context: documents: [{ doc_type, photo_paths: [] }].
Design Details
Document cards stack; each card has type label + photo grid + delete.
Database Schema
Written to user_documents at submit.
10First Pond Address
Purpose
Captures pond's address (often differs from farmer's residential).
Business Requirement
District + village + free-text address. Pre-fill with farmer's address as a starting point; scout edits if pond is elsewhere.
User Experience
Same pattern as Address Details with "Same as farmer's address" toggle.
User Actions
Toggle same-as · or edit fields · Continue.
Technical Details
Context: pond.{district, panchayat_village, full_address}.
Design Details
Toggle at top; form below.
Database Schema
Written to ponds at submit.
11First Pond Geo-Fencing
Purpose
Captures the pond's GPS centre point.
Business Requirement
Year-1: single GPS point. Year-2 polygon optional. Mandatory because scout-task routing depends on it.
User Experience
Map view with a draggable pin. "Use my location" auto-places pin at scout's current GPS.
User Actions
Tap map to place / drag pin · or "Use my location" · Continue.
Technical Details
Map: Google Maps Flutter. Captures lat/lng + accuracy. Wizard context: pond.{gps_lat, gps_lng}.
Design Details
Map (300dp height), pin centre overlay, coordinates strip below, "Use my location" secondary button, primary Continue.
Database Schema
Written to ponds.gps_lat/lng at submit.
12Bank Account Details
Purpose
Captures farmer's bank account for payments.
Business Requirement
Account holder name (must match identity), account number (entered twice for confirmation), IFSC. Penny-drop verification triggered at submit (₹1 test transfer, name-match check). Bank account is one per farmer in Year 1.
User Experience
Form with account number confirmation field. IFSC auto-resolves bank + branch name on entry.
User Actions
Enter holder name → account number → confirm account number → IFSC → see resolved bank/branch → Continue.
Technical Details
IFSC validated regex ^[A-Z]{4}0[A-Z0-9]{6}$; bank/branch resolved via Razorpay IFSC API or local lookup. Account number encrypted at-rest. Penny-drop is an async server-side job triggered at submit; the wizard doesn't block on it — bank shows pennydrop_status='pending' initially.
Design Details
Standard form. Bank/branch displayed read-only after IFSC entry. Helper: "We'll send ₹1 to confirm this account — penny-drop verification."
Database Schema
Written to user_bank_accounts at submit.
13Theme Preference
Purpose
Sets app theme for the farmer's eventual login.
Business Requirement
Trivial preference; defaults to Dark. Scout can skip or set on behalf.
User Experience
Two preview tiles · or "Use default (Dark)" link.
User Actions
Tap a tile · or skip · Continue.
Technical Details
Context: theme: 'dark'|'light'. Written to user_preferences at submit.
Design Details
Two tile previews, same as scout's theme pattern.
Database Schema
user_preferences (existing).
14Review & Submit
Purpose
Final recap before commit.
Business Requirement
Scout reviews every captured field; farmer can be shown the screen for transparency. Explicit final consent checkbox + submit.
User Experience
Sectioned recap (Identity, Address, Pond, Bank, Documents). Each section editable via inline "Edit" link → returns to the relevant step.
User Actions
Review · tick final consent · Submit · or Edit any section.
Technical Details
On submit (atomic, offline-capable): writes users, farmer_profiles, ponds, user_bank_accounts (+ pennydrop async), user_documents, user_preferences. Status: users.status='pending_approval', ponds.status='pending_approval'. created_via_scout_id = self. All queued to sync_queue as a single atomic batch. Notifications to Admin: farmer_registration_submitted.
Design Details
Sectioned recap, edit links, danger-tinted confirm checkbox, primary "Submit" disabled until ticked.
Database Schema
All tables defined above.
15Submission Confirmation
Purpose
Closes the wizard and tells the scout what happens next.
Business Requirement
Plain language outcome — "Submitted, pending Admin approval. Farmer will be notified by SMS when approved."
User Experience
Success state echoing farmer's name, mobile, pond code, and sync status.
User Actions
"Back to Users" · "Register another farmer" · "Next task →" if relevant.
Technical Details
Reads back local writes. No further state changes.
Design Details
Success icon, headline, summary chips, sync line.
Database Schema
Read-only.
