React Component Decomposition & Reusability

Scout Dashboard (3 screens) breaks into reusable React components following atomic design principles. Components are shared across Scout module's 20 workflows.

✅ Design Philosophy:
• Single Responsibility: Each component does ONE thing
• Prop-based composition: Components accept data + actions as props
• No local state for business logic: Context + hooks for state
• Reuse across workflows: 80% of components shared across 20 workflows
• Variant management: Props define UI variants, not separate components
📖 Cross-Workflow Component Library
All scout workflows share a unified component library. See COMPONENT_ARCHITECTURE_GUIDE.md for the master reference documenting all 20 workflows and their component usage.

Screen 1Today's Task List
scout_today_dashboard
Purpose

Primary field hub for the scout's daily workflow.

Surfaces all active assignments (Inspections, Fitness Checks, Re-checks) based on proximity and priority.

Business Requirement
  • Dynamic task list scoped to the logged-in scout.
  • Status grouping: New, In-Progress, Completed.
  • Priority markers for lots linked to active Buyer POs.
User Experience (UX)

Scrollable list of task cards.

Each card shows Warehouse/Lot Name, Task Type, Distance (km), and Priority Pill.

Pull-to-refresh for latest sync.

User Actions
  • Tap Task Card: Opens Task Detail Bottom Sheet.
  • Search: Filter tasks by warehouse name or lot ID.
  • Map Toggle: View tasks in a geofenced map view.
Technical Details

Polls `scout_tasks` table.

Uses device GPS to calculate `distance_km` client-side via Haversine.

Offline-first caching enabled.

Design Details
  • Dark-themed list with surface-hi card separation.
  • Accent color (Cyan) for active icons.
  • Skeletal loaders for data transitions.
Database Schema
SELECT * FROM scout_tasks WHERE scout_id = auth.uid() AND status != 'completed' ORDER BY priority DESC, scheduled_date ASC;

Screen 2Overdue Tasks Screen
overdue_tasks_screen
Purpose
Escalation view surfacing tasks that have missed their 24-48 hour service level agreement (SLA) window.
Business Requirement
  • Automatic flag for tasks > 24h past `scheduled_date`.
  • Visual warning: Red indicator for overdue status.
  • Prevents cherry-picking of newer tasks while old ones linger.
User Experience (UX)

Separate tab or filtered list with "Overdue" header.

Time-elapsed counter shown on each card (e.g., "Overdue by 14 hrs").

User Actions
  • Acknowledge: Adds a note why the task is delayed.
  • Tap Task: Standard Task Detail Sheet opens.
Technical Details

Calculates `now() - scheduled_date`.

Filter: `status = 'pending' AND scheduled_date < now()`.

Design Details
  • Danger Red (#ef4444) highlights for overdue counts.
  • Warning icon on list items.
Database Schema
CREATE INDEX idx_overdue_tasks ON scout_tasks (scheduled_date, status) WHERE status = 'pending';

Screen 3Notifications Center
scout_notifications
Purpose
Audit trail of critical alerts, task re-assignments, and system-level messages for the scout.
Business Requirement
  • Real-time alerts for new high-priority tasks.
  • Feedback on Admin-reviewed inspections (e.g., Re-inspection requested).
  • Sync conflict warnings.
User Experience (UX)

Chronological list of notification cards.

Grouped by Today/Yesterday/Older.

Badge count on the bell icon in Top Bar.

User Actions
  • Tap Notification: Deep-link to the relevant task or sync item.
  • Clear All: Mark all as read.
Technical Details

Uses Supabase real-time subscriptions for the `notifications` table.

FCM (Firebase Cloud Messaging) for background pushes.

Design Details
  • Distinct icons for Task, Warning, and Message.
  • Unread state indicated by a subtle blue dot.
Database Schema
CREATE TABLE notifications ( id UUID PRIMARY KEY, user_id UUID REFERENCES users(id), title TEXT, body TEXT, metadata JSONB, -- { "task_id": "...", "type": "task_assigned" } read_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT now() );
