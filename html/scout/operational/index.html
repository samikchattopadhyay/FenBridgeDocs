Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

Entry & Authentication is a lightweight 3-screen flow emphasizing security, user onboarding, and role-based routing. The splash screen handles cold-start initialization, the login screen integrates SMS-based OTP verification, and the routing screen automatically directs authenticated scouts to their dashboard.

State Management & Data Flow

Local Auth State (Client-side):
• auth_token: JWT (15-30 min expiry)
• refresh_token: Single-use rotation token (7-14 days)
• user_id: UUID
• user_role: 'scout' | 'farmer' | 'buyer' | 'popper'
• mobile_number: string (verified via OTP)
• otp_session_id: temporary session token

Backend Authentication Flow:
1. Phone number → initiate OTP via Twilio
2. OTP input → verify against SMS record
3. Query scouts table with phone_number
4. Issue JWT + refresh token
5. Decode JWT role claim → router.navigate(role-specific-path)

Common Patterns & Reuse

  • OTP Input Pattern: 6 sequential digit inputs with auto-advance to next field. Extract to OtpInput component for reuse in other SMS-based workflows.
  • Phone Input: Country code selector + mobile number. Reusable PhoneInput component across farmer/popper registration.
  • Token Persistence: Store JWT in secure localStorage with refresh token rotation. Use AuthContext to share across app.
  • Error Handling: Rate-limit OTP requests (max 3 per minute). Display user-friendly error messages for invalid OTP.
  • Splash Screen: Implement with setTimeout + navigate pattern. Can be reused for app version checks.
