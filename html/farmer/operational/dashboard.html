Screen 1Farmer Primary Dashboard
farmer_dashboard
Purpose
Primary landing screen after login. Surface what needs the farmer's attention today — new bid invitations, active orders, payments, ready lots — in a single scrollable view.
Business Requirement
Year 1 farmers are small-pond operators with limited screen time and patchy connectivity. They need to see "what's happening with my money and my produce" in under 10 seconds. Anything that requires action must surface here.
UX & User Actions
UX: Single scrollable screen with stacked sections. Each section is a glanceable card or list with a CTA. No nested navigation from dashboard — every tap deep-links to the relevant module.

Actions:
• View counts of new bid invitations, active POs, spot listings, pending payments.
• Tap a section to navigate to the full list.
• Tap a featured card to jump directly to the detail.
• Pull to refresh.
• Tap floating action button (FAB) → "Request Spot Listing" wizard.
Technical Details
• Fetch aggregated counts via single Supabase RPC farmer_dashboard_summary(user_id) returning all section counts in one call.
• Cache last successful response locally (Hive/SharedPreferences) for offline rendering.
• Realtime subscription on bid_invitations and purchase_orders filtered by farmer_id.
• Capability score and tier read from users.capability_score.
Design Details
• Top bar: Home title, bell icon with unread badge, profile avatar.
• Greeting strip: Title 2 greeting, pond count caption, capability tier pill.
• Section 1 (Attention Strip): Horizontal scroll of compact non-zero notification alerts.
• Section 2 (Quick Stats): 2x2 grid tracking Active POs, Spot Listings, Earned This Month, Pending Receipts.
• Sections 3-5: Direct previews of Open Invitations, Active POs, and Spot Listings.
• Persistent primary FAB triggers Request Spot Listing flows.
Database Schema
-- Read-only aggregation RPC; no new tables. Referenced schemas: users (id UUID PK, full_name TEXT, role ENUM, capability_score DECIMAL); bid_invitations (id UUID PK, requirement_id UUID, farmer_id UUID, status ENUM); purchase_orders (id UUID PK, buyer_id UUID, farmer_amount NUMERIC); spot_listings (id UUID PK, status ENUM, expires_at TIMESTAMP); payments (id UUID PK, status ENUM, utr TEXT);

Screen 2Notifications Inbox
notifications_inbox
Purpose
Single in-app place to see all notifications the farmer has received — bid invitations, awards, scout visits, payments, lot status changes, system messages.
Business Requirement
Push notifications and SMS reach farmers in real-time but are easily missed or buried. The inbox is the durable, searchable record they can return to when they have time. It also serves as a navigation index — every notification deep-links to the source entity.
UX & User Actions
UX: Reverse-chronological list. Unread items have an accent dot prefix and surface-hi background. Tapping a row navigates to the deep-linked entity and marks the notification as read.

Actions:
• View list of all notifications.
• Tap a row to navigate directly to the deep-linked entity sheet.
• Long-press context menu: Mark unread / Delete / Mute this type.
• Filter by category: All / Bids / POs / Payments / Spot / System.
Technical & Design Details
• Query: SELECT * FROM notifications WHERE user_id = current_user ORDER BY created_at DESC LIMIT 50.
• Local push (FCM) and SMS fired in parallel for critical events.
• Unread count exposed via Supabase RPC unread_notification_count(user_id).
• Retained for 180 days; older records archived server-side.
Database Schema
notifications ( id UUID PRIMARY KEY, user_id UUID FK, notification_type ENUM, title TEXT, body TEXT, deep_link TEXT, is_read BOOLEAN DEFAULT false, created_at TIMESTAMP ); notification_preferences ( user_id UUID PK FK, preferences JSONB );
