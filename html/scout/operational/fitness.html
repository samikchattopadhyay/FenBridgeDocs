Screen 1Confirm Warehouse & Arrival
confirm_warehouse_arrival
Purpose
Confirms scout is at the correct physical location and captures the audit's arrival GPS coordinates.
Business Requirement
  • Audit is only valid if captured on-site.
  • Haversine distance check against registered warehouse GPS.
  • Tolerance: 200m warning (allows for rural GPS drift).
User Experience (UX)
Warehouse ID card with last fitness status (e.g., "Fit - Nov 2023"). Large "Capture Arrival" button. Interactive map pin showing current vs. expected location.
User Actions
  • Capture GPS: Triggers location lock.
  • Continue: Enabled once GPS accuracy is within 20m.
Technical Details
Uses `Geolocator` with `high` accuracy. Compares against `warehouses.gps_lat/lng`. Time-stamps `arrival_at`.
Design Details
  • Prominent Warehouse Name header.
  • Pulsing GPS indicator during capture.
Database Schema
INSERT INTO fitness_checks (warehouse_id, scout_id, arrival_lat, arrival_lng, arrival_at) VALUES (...);

Screen 4Audit Summary & Submit
verdict_submit
Purpose
Recaps the entire audit, presents computed verdict, and atomic commit.
Business Requirement
  • Decision Engine: Any Critical Fail = Terminal Unfit. Non-critical fails = Verdict Banner "Fit with Observations".
  • Updates `warehouses.fitness_status` to 'Fit' or 'Unfit'.
  • Immutable history. No edits post-submit.
User Experience (UX)
Verdict banner (Color-coded: Green/Amber/Red). Checklist recap rows. Explicit confirmation checkbox "I attest this audit was conducted on-site...".
User Actions
  • Tick Attestation: Unlocks Submit button.
  • Submit Audit: Atomic local write and sync queue entry.
Technical Details
On submit, triggers `sync_queue` write. If Fit, warehouse becomes eligible for Lot Verification. If Unfit, sends Admin push notification.
Design Details
  • Large verdict icons.
  • Cyan Submit button (disabled state until checked).
Database Schema
UPDATE warehouses SET fitness_status = $1, last_fitness_check_at = now() WHERE id = $2;
