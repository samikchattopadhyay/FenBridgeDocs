Workflow Overview
Pond Inspection is the scout's on-site fieldwork for raw-makhana supply intelligence. The workflow validates the scout's physical presence via GPS, captures three coded assessments (water level, algae, pest/disease), collects 3–5 spot-sampled maturity and density readings from bund edges, and requires minimum photographic evidence. All data is captured offline and synced when network is available. Submissions are immutable; scout cannot edit after confirm. Year 1 use is learning-only—no commercial commitments, no grades, no seals produced. The workflow integrates with the scout_tasks queue and feeds pond readiness signals for future makhana supply projections.
2.1 Confirm Pond & Arrival
Purpose
Confirms the scout is at the right pond and captures arrival GPS as evidence of physical presence.
Business Requirement
Like warehouse fitness checks, a pond inspection is only credible if the scout was there. GPS mismatch warns (not blocks) — rural ponds may have weak signal. Prior inspection summary helps the scout calibrate expectations.
User Experience (UX)
Single read-only screen — farmer + pond identity card, prior-inspection summary strip, GPS capture, primary CTA.
User Actions
• Review pond identity (farmer name, pond code, village, pond area)
• Review last-inspection summary (date, readings averages)
• Tap "Capture location" → GPS + accuracy
• (If mismatch) acknowledge or re-capture
• Tap "Begin Inspection" → Screen 2.2
Technical Details
Hydrates ponds + joined users (farmer) + last pond_inspections row. Haversine vs ponds.gps_lat/lng; warn > 200 m at accuracy < 50 m. Wizard context: { pond_id, farmer_user_id, task_id, arrival_gps, check_type }. check_type = 'initial' if no prior, else 'periodic'. Fully offline.
Design Details
Top bar: Close, "Pond Inspection", progress "1 of 5". Pond card (surface-hi): farmer name (Title 3), pond code, village, area in acres/bigha. History strip: "Last inspection: 12 Feb 2026 — water level Normal, readiness 6/10" or "First inspection". GPS block + mismatch warning (same pattern as fitness check). Action row: Navigate · Call farmer. Primary CTA: "Begin Inspection".
Database Schema
Reads ponds, users, pond_inspections. References defined in §6 below.
2.2 Pond Conditions
Purpose
Captures the high-level pond state — water level, algae, pest/disease, harvest readiness.
Business Requirement
Three coded categorical fields plus a 1–10 readiness slider. All coded, no free text. These four fields together drive the "is this pond near harvest?" intelligence signal.
User Experience (UX)
Single scrollable form. Each field is a labelled card with a segmented control or slider. Continue enables when all four are set.
User Actions
• Set water level — Low / Normal / High
• Set algae presence — None / Light / Heavy
• Tap pest/disease flags (multi-select coded chips)
• Set readiness score (slider 1–10)
• Tap "Continue" → Screen 2.3
Technical Details
Pest/disease flag set loaded from versioned reference set pond_pest_flags (codes like aphids, leaf_spot, stem_rot, etc.). Held in wizard context; no persistence yet.
Design Details
Top bar: back, "Pond Conditions", progress "2 of 5". Heading "Assess pond conditions". Four field cards (surface-hi): segmented controls for first two; chip multi-select for pest/disease; slider for readiness with live numeric echo. Primary CTA: "Continue".
Database Schema
Written at Screen 2.5 — fields on pond_inspections (water_level, algae_presence, pest_disease_flags, readiness_score).
2.3 Bund-Edge Readings
Purpose
Captures 3–5 spot samples taken at different bund edges, each with seed density and maturity percentage.
Business Requirement
Project memory: large pond scouting in Year 1 uses scout-averaged 3–5 bund-edge readings. The screen must enforce the minimum of 3 readings and accept up to 5.
User Experience (UX)
Stacked reading rows, each with two numeric inputs (density per sq m, maturity %). "Add reading" button until 5 rows. Live average shown at bottom.
User Actions
• For each reading: enter density + maturity %
• Tap "Add reading" → new row (max 5)
• Delete a row (min 3 retained)
• View live averages
• Tap "Continue" → Screen 2.4
Technical Details
Held as array in wizard context: bund_readings: [{ density, maturity_pct }]. Validation: at least 3 rows complete. Average computed client-side.
Design Details
Top bar: back, "Bund Readings", progress "3 of 5". Helper: "Take 3–5 readings around the pond bund." Numbered reading rows. Live averages chip at bottom: "Avg density 14.2/sqm · Avg maturity 62%". Primary CTA: "Continue".
Database Schema
CREATE TABLE pond_bund_readings ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), pond_inspection_id UUID NOT NULL REFERENCES pond_inspections(id) ON DELETE CASCADE, reading_seq INT NOT NULL, density_per_sqm NUMERIC(6,2), maturity_pct NUMERIC(5,2) ); CREATE INDEX idx_pbr_inspection ON pond_bund_readings (pond_inspection_id);
2.4 Photos & Notes
Purpose
Captures 1–3 pond photos and optional notes.
Business Requirement
Photos are mandatory for visual verification of the pond and the scout's claimed presence. Notes are optional.
User Experience (UX)
Photo grid (3 fixed slots — wide pond, sample, anomaly-or-extra), camera-first, plus a multiline notes field at the bottom.
User Actions
• Capture pond wide + sample photos (required)
• Optionally capture anomaly photo
• Add optional notes
• Tap "Continue" → Screen 2.5
Technical Details
Photos slot_type (pond_wide / sample / anomaly_or_extra); compression ≤ 1.5 MB; GPS stamped from arrival.
Design Details
Photo tile grid; notes multiline (max 500 chars). Primary CTA "Continue".
Database Schema
CREATE TABLE pond_inspection_photos ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), pond_inspection_id UUID NOT NULL REFERENCES pond_inspections(id) ON DELETE CASCADE, slot_type TEXT NOT NULL, storage_path TEXT NOT NULL, captured_at TIMESTAMPTZ NOT NULL );
2.5 Review & Submit
Purpose
Final recap and atomic commit. Submission confirmation echoes the readiness score with sync state.
Business Requirement
Like every other scout workflow, immutable on submit; corrections Admin-only.
User Experience (UX)
Recap card + confirmation checkbox + "Submit Inspection". On submit → full-screen confirmation echoing the readiness score with sync state.
User Actions
• Confirm → submit → "Back to Today" / "Next task"
Technical Details
Atomic local write: 1 pond_inspections + N pond_bund_readings + M pond_inspection_photos. Updates scout_tasks.status='completed'. Queues to sync_queue.
Design Details
Same pattern as inspection / fitness-check submission confirmations.
Database Schema
Inserts only; all tables defined above.

Complete Database Schema

CREATE TABLE pond_inspections ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), pond_id UUID NOT NULL REFERENCES ponds(id), farmer_user_id UUID NOT NULL REFERENCES users(id), scout_id UUID NOT NULL REFERENCES scouts(id), task_id UUID REFERENCES scout_tasks(id), -- conditions water_level TEXT CHECK (water_level IN ('low','normal','high')), algae_presence TEXT CHECK (algae_presence IN ('none','light','heavy')), pest_disease_flags JSONB, -- array of coded flags readiness_score INT CHECK (readiness_score BETWEEN 1 AND 10), -- meta notes TEXT, arrival_lat NUMERIC(9,6) NOT NULL, arrival_lng NUMERIC(9,6) NOT NULL, arrival_accuracy_m NUMERIC(6,1), checked_at TIMESTAMPTZ NOT NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT now(), synced_at TIMESTAMPTZ ); CREATE INDEX idx_pi_pond ON pond_inspections (pond_id, checked_at DESC);
