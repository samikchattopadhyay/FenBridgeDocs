Workflow Module 6

Storage Re-check

Periodic audit of sealed lots during the 30/45-day listing window to ensure continued storage compliance and seal integrity.

5 Screens
Screen 1Lot / Warehouse Header
recheck_lot_header
Purpose
Identifies the specific lot due for periodic audit (Section 6.1). Triggered at 50% shelf-life or by Admin.
Business Requirement
  • Bounded risk: BM guarantees quality for 30/45 days; re-checks prevent "seal-and-forget" failures.
  • Must show remaining shelf-life.
User Experience (UX)
Context card with Lot ID, Seal ID, and Countdown Timer for expiry. GPS distance check to ensure on-site audit.
User Actions
  • Capture Location.
  • Start Re-check.
Technical Details
Loads from `lots` inner joined with `lot_seals`. Computes `days_remaining`.
Design Details
  • Amber warning text if < 7 days remaining.
Database Schema
SELECT * FROM lots l JOIN lot_seals s ON l.id = s.lot_id WHERE l.id = $1;

Screen 2Seal & Net Integrity Check
seal_net_integrity
Purpose
Physical verification that the hardware seal and protective netting are untampered (Section 6.2).
Business Requirement
  • Broken seal = Immediate Void.
  • Netting must be secure.
User Experience (UX)
Scan-to-verify. Scout must scan the existing seal; system matches it against the stored `seal_id`. Logic: "It's the same seal, and it's physically intact".
User Actions
  • Scan Seal QR.
  • Toggle Integrity Status (Intact / Tampered).
Technical Details
QR comparison logic. If mismatch, forces 'Tampered' state.
Database Schema
// Logic check: scan_result == lot.seal_id

Screen 3Conditions Check
conditions_check
Purpose
Spot audit of the current warehouse environment (Section 6.3).
Business Requirement
  • Check: 1. Bags dry? 2. Still off-floor on pallets? 3. No new rodent signs? 4. No strong odours?
User Experience (UX)
Lightweight checklist (similar to Fitness Check but only 4 critical items). Single-tap pass/fail.
User Actions
  • Audit Conditions.
Technical Details
Stored as a `recheck_log` record.

Screen 5Outcome
recheck_outcome
Purpose
Final verdict (Section 6.5).
Business Requirement
  • Clean -> Log & Continue listing.
  • Anomaly -> Route to Re-inspection or Void.
User Experience (UX)
Big Success (Pass) or Critical Fail (Void) banner. "Submit & Finish" button.
User Actions
  • Submit Audit.
Technical Details
If Anomaly: sets `lots.status = 'voided'`. If Clean: updates `last_recheck_at`.
Database Schema
UPDATE lots SET last_recheck_at = now() WHERE id = $1;
