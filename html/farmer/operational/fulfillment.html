Screen 1My Active POs Hub
my_active_pos
Purpose
Show all Purchase Orders where this farmer is the awarded producer. Track production stage, dispatch readiness, and payment status against each PO.
Business Requirement
Binding production obligation post bid award. Producers must know expectations (grade, quantity, ready-by date), update status when ready, and track clear payments due from BM.
UX & User Actions
UX: Filterable list of active and recently closed POs. Stage badges visually dominate the card layout.

Actions:
• Filter by stage: All / Advance Pending / In Production / Ready / Dispatched / Delivered / Completed.
• Search by PO ID string.
• Tap row to load PO Detail sheet.
• Trigger primary Mark Lot Ready flows.
Technical & Design Details
• Query: SELECT * FROM purchase_orders WHERE farmer_id = current_user.
• Fully masks wholesale buyer identities and buyer-side receivable margins.
• Real-time subscriptions enabled per row.
Database Schema
purchase_orders (id UUID PK, buyer_id UUID, farmer_id UUID, state ENUM, farmer_payable NUMERIC); po_state_transitions (id UUID PK, po_id UUID, from_state TEXT, to_state TEXT); po_documents (id UUID PK, po_id UUID, doc_type ENUM, storage_path TEXT);

Screen 2PO Detail — Farmer View
po_detail_farmer
Purpose
Provide a detailed tabbed overview detailing delivery deadlines, expected payouts, and verified field metrics. Expose context-aware action triggers routing straight to harvest completion flows.
Business Requirement
Single durable source of truth itemizing assigned field scouts, operational checklists, and multi-leg BM-to-farmer payout schedules.
UX & User Actions
UX: Tabbed navigation interface: Overview · Production · Payments · Documents.

Actions:
• Review strict quality standard metrics (moisture, foreign matter, broken seeds).
• Track production checklists dynamically.
• Contact assigned scout directly via phone/messaging link.
• Download generated grading certificates.
Technical Details
• Exposes purely producer-relevant columns while stripping central margin ledgers cleanly.
Row Level View Mapping
-- Secure view mapping enforcing producer column boundaries: SELECT id, lot_id, grade, quantity_kg, farmer_payable, ready_by_date, state, buyer_region FROM purchase_orders WHERE id = target_id AND farmer_id = current_user;

Screen 3Dispute Management & Evidence
dispute_management
Purpose
Enable farmers to file disputes with photographic and documentary evidence when quality is questioned, payment discrepancies occur, or delivery issues arise. Admin reviews, awards compensation, rejects, or escalates to Director. Full audit trail for trust and transparency.
Business Requirement
• BR-141: Dispute filing window — within 7 days of PO closure or payment.
• BR-142: Evidence types — photos, lab reports, bank screenshots, witness statements.
• BR-143: Dispute types — Quality downgrade, Qty mismatch, Payment error, Delivery delay, Seal breach.
• BR-144: BM review SLA — <48 hours from filing.
• BR-146: Appeal once if rejected — escalates to Director for final decision.
Entry Point
PO Detail (after PO closed) → "File Dispute" button.
Or: Notifications → "Quality Downgrade Alert" → "File Dispute".
UX & User Actions
Step 1: Select dispute type + enter claim summary.
Step 2: Upload evidence (photos, lab test, witness statement).
Step 3: Supporting notes + submit → receive dispute reference #.
Track: Under Review (48h) → Resolved / Rejected / Appealed.
Database Schema
disputes ( id UUID PK, po_id UUID FK -> purchase_orders.id, filed_by UUID FK -> users.id, dispute_type ENUM ['quality','qty','payment','delivery','seal','other'], claim_summary TEXT, claimed_amount NUMERIC(12,2), status ENUM ['filed','under_review','resolved','rejected','appealed','closed'], decision TEXT, awarded_amount NUMERIC(12,2), filed_at TIMESTAMP, resolved_at TIMESTAMP ); dispute_evidence ( id UUID PK, dispute_id UUID FK, evidence_type ENUM ['photo','lab_report','bank_screenshot','witness'], storage_path TEXT, uploaded_at TIMESTAMP );
