Workflow Overview
Ponds Directory is the scout's primary browse interface for all ponds assigned to them. The workflow provides search and filtering capabilities to locate specific ponds, view detailed profiles including GPS coordinates and linked farmer information, browse complete inspection history, and access spot offers linked to each pond. Scout can initiate new pond inspections directly from Pond Detail. Scope is enforced server-side — scouts see only ponds assigned via scout_assignments_ponds. The directory mirrors the Warehouse List/Detail pattern for consistency with existing warehouse management flows.
3.1 Pond List
Purpose
Directory of every pond the scout is assigned to.
Business Requirement
Scope-limited via scout_assignments_ponds. Search by farmer / village / pond code; filter by inspection recency.
User Experience (UX)
Same pattern as Warehouse List: search + filter chips + cards. Each card: farmer name, pond code, village, area, last inspection date.
User Actions
• Search by farmer name, village, or pond code
• Apply filter chips (Today, This Week, Overdue)
• Tap card → Pond Detail
Technical Details
Reads ponds joined scout_assignments_ponds. Aggregate "last inspection" from pond_inspections. Scope enforced via WHERE scout_id = current_user. Search queries pond_code, panchayat_village, and joined farmer name.
Design Details
Card layout identical to Warehouse List; pill shows "Inspected 12 Feb" / "Not inspected". Each card displays: farmer name (Title 3), pond code, village, area (Acre/Bigha/Hectare), and last inspection pill with date or "Never inspected".
Database Schema
CREATE TABLE scout_assignments_ponds ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), scout_id UUID NOT NULL REFERENCES scouts(id), pond_id UUID NOT NULL REFERENCES ponds(id), assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(), active BOOLEAN NOT NULL DEFAULT true, UNIQUE (scout_id, pond_id) ); CREATE INDEX idx_sapo_scout ON scout_assignments_ponds (scout_id, active);
3.2 Pond Detail
Purpose
Full record of a single pond — profile, GPS, inspection history, linked farmer, linked spot offers.
Business Requirement
Mirrors Warehouse Detail. Scout can start a fresh inspection from here.
User Experience (UX)
Header + tabs: Profile · Inspections · Spot Offers.
User Actions
• Switch tabs
• Tap inspection row → Pond Inspection Record
• Tap spot offer → Spot Offer Detail
• Primary CTA: "Start Inspection"
Technical Details
Reads ponds, users (farmer), pond_inspections, spot_offers WHERE source_type='pond' AND source_id=?. Profile tab: hydrates farmer contact and pond metadata. Inspections tab: queries pond_inspections ordered by checked_at DESC, joined with pond_bund_readings and pond_inspection_photos for display. Spot Offers tab: queries spot_offers for this pond source with status and pricing.
Design Details
Header card with pond name (Title 2), farmer name, status pill (Pending Approval / Active / Inactive), and area. Tab row with counts. Profile tab: pond code, GPS coordinates with map embed, district, village, address. Inspections tab: chronological rows showing date, water level, algae, readiness score, photo count. Spot Offers tab: shows linked offers with lot status and price. "Start Inspection" CTA floats or appears in header.
Database Schema
CREATE TABLE ponds ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), farmer_user_id UUID NOT NULL REFERENCES users(id), pond_code TEXT UNIQUE NOT NULL, name TEXT, area_value NUMERIC(6,2), area_unit TEXT CHECK (area_unit IN ('acre','bigha','hectare')), est_annual_capacity_kg NUMERIC(10,2), district TEXT, panchayat_village TEXT, full_address TEXT, gps_lat NUMERIC(9,6), gps_lng NUMERIC(9,6), status TEXT NOT NULL DEFAULT 'pending_approval' CHECK (status IN ('pending_approval','active','inactive','rejected')), created_via_scout_id UUID REFERENCES scouts(id), created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_ponds_farmer ON ponds (farmer_user_id); CREATE INDEX idx_ponds_scout ON ponds (created_via_scout_id);
3.3 Pond Inspection Record
Purpose
Full view of a single past pond inspection.
Business Requirement
Read-only audit detail. No edits. Provides historical context for pond evolution over time.
User Experience (UX)
Header, readings table, photos strip, notes.
User Actions
• View inspection details (read-only)
• Tap photos for full-screen lightbox
• Back to Pond Detail
Technical Details
Reads pond_inspections + children (pond_bund_readings, pond_inspection_photos). No write operations. Photos cached locally if previously synced; available offline.
Design Details
Header section: inspection date, scout name, arrival GPS + accuracy, check type (initial/periodic). Conditions summary: water level, algae, pest/disease flags, readiness score. Bund readings table: all 3-5 readings with densities and maturity percentages, live average row at bottom. Photo strip: all 1-3 photos in fixed-width grid (wide pond, sample, anomaly-or-extra), tappable for lightbox. Notes section (if present): full text display.
Database Schema
Read-only. Reads from pond_inspections, pond_bund_readings, and pond_inspection_photos (all defined in Pond Inspection workflow §2).
