5 Concrete Steps
Step 1Basic Parameters
POST_REQUIREMENT/post_requirement_step_1
Purpose
Capture root wholesale purchasing intent and core product sorting tier definitions. Buyer selects high-level makhana categorizations establishing target filtering context.
Business Requirement
Bharat Makhana sells in defined grade tiers (A, B, C) with versioned quality specifications. The buyer must explicitly choose one grade—no mixed options.
UX & User Actions
UX: Single screen showing visual radio tiles with spec previews.

Actions:
• Tap a grade tile to expand its quality spec parameters.
• Select target product variant.
Technical Details
Grade options are fetched from the versioned grade_specifications table to ensure strict typing and absolute structural consistency.

Step 2Grade & Specifications
POST_REQUIREMENT/post_requirement_step_2
Purpose
Input definitive sorting grade criteria along with precise sizing millimeter diameters. Forms core inputs matching against pre-assessed farmer stock levels.
Business Requirement
Locks the current active version of the grade spec into the requirement at submission, guaranteeing that subsequent platform spec updates do not retroactively alter active trade promises.
Technical Details
Displays live preview parameters promised to the buyer: maximum moisture, foreign matter limits, broken seeds cap, and target screen retention sizes.
Database Schema
-- Resolves authoritative quality boundaries per variant: SELECT id, grade, version, moisture_max_pct, foreign_matter_max_pct FROM grade_specifications WHERE is_active = true;

Step 3Quantity & Pricing Caps
POST_REQUIREMENT/post_requirement_step_3
Purpose
Record total minimum bulk weight commitments (kg) alongside maximum intended buyer acquisition offer prices per kilogram. Triggers backend platform margin mapping logic.
Business Requirement
Establishes commercial bounds. The ceiling price represents the maximum total payout from the buyer. Internally, BM applies margin spread models to compute counterparty farmer-ceiling thresholds automatically.
UX & User Actions
UX: Numeric input fields integrated with reference market price comparison cards.

Actions:
• Enter required quantity in kg (enforcing minimum viable operational limits).
• Input acquisition ceiling price per kg.
Technical Details
Live validations trigger warnings if proposed pricing falls below 70% of reference indices to avoid zero-bid deadlocks. Enforces strict ceiling constraints.

Step 4Delivery Deadlines & Location
POST_REQUIREMENT/post_requirement_step_4
Purpose
Establish hard delivery completion targets using calendar controls and specify intended end-destination corporate dispatch addresses.
Business Requirement
Ensures logistics synchronization. The delivery window must accommodate local processing lead times while filtering routing availability based on saved destination regions.
Technical Details
Queries the buyer_addresses list and applies calendar range bounds preventing impossible backward-dated delivery targeting. Stores selected address reference ID.

Step 5Final Summary & Submission
POST_REQUIREMENT/post_requirement_step_5
Purpose
Final aggregated configuration check review block. Committing writes to real-time arrays dispatches pre-filtered custom invitation rows to capable tier sellers instantly.
Business Requirement
Instantiates a valid baseline wholesale requisition. Sets transaction state to Draft or Open, initiating automated matchmaking workflows while preserving counterparty privacy.
Technical Details
Atomic payload validation writes core structures and dispatches background notifications to area administrators and matching producer pools simultaneously.
Database Schema
-- Authoritative submission definition: requirements ( id UUID PK, buyer_id UUID, grade ENUM, quantity_kg INT, buyer_ceiling_price_per_kg NUMERIC, farmer_ceiling_price_per_kg NUMERIC, delivery_window_start DATE, delivery_window_end DATE, delivery_address_id UUID, status ENUM DEFAULT 'draft' );
