Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Sync Queue workflow implements an offline-first reconciliation engine for atomic data propagation from field devices to the central repository. The 3-screen flow provides visibility into pending items, active transmission with retry logic, and manual conflict resolution for edge-case state conflicts. This workflow is critical infrastructure enabling Scout to operate with patchy connectivity while ensuring zero data loss.

Key patterns: Batch processing with atomic transactions, exponential backoff retry, conflict detection and quarantine, progress indication for long operations.

State Management & Backend Logic

Sync Queue Data Structure

interface SyncQueueItem { id: UUID; workflow_type: "inspection" | "seal" | "fitness" | "photo"; lot_id: string; payload: JSONB; blob_paths?: string[]; // File references in app storage status: "pending" | "syncing" | "success" | "error" | "conflict"; error_message?: string; retry_count: number; created_at: timestamp; last_retry_at?: timestamp; conflict_resolution?: "discard" | "retry" | null; }

Batch Sync with Exponential Backoff

const syncBatch = async (items: SyncQueueItem[]) => { for (const item of items) { try { // Atomic RPC call with retry logic const result = await rpc.submitSyncItem({ workflow: item.workflow_type, payload: item.payload, attachments: await loadBlobsFromStorage(item.blob_paths) }); // Update local state await db.syncQueue.update(item.id, { status: 'success' }); emitProgress(items.length, successCount + 1); } catch (error) { if (isRetryable(error)) { const nextRetry = Date.now() + exponentialBackoff(item.retry_count); await db.syncQueue.update(item.id, { status: 'error', error_message: error.message, last_retry_at: nextRetry, retry_count: item.retry_count + 1 }); } else if (isConflict(error)) { // State conflict: quarantine for manual review await db.syncQueue.update(item.id, { status: 'conflict', error_message: error.serverMessage }); } } } };

Conflict Detection & Quarantine

const isConflict = (error: Error) => { // Server error codes indicating state mismatch const conflictCodes = [409, 422, 'ENTITY_NOT_FOUND', 'STATUS_MISMATCH']; return conflictCodes.some(code => error.message.includes(code) || error.code === code ); }; // Conflict stays in sync_queue with status='conflict' // Admin must manually review and authorize discard or manual reconciliation // Prevents automatic replay of stale commands

Common Patterns & Reusability

  • Batch Processing: Generic batch sync engine used across all workflows. Extract as <BatchSyncEngine items={queueItems} onProgress={callback} />
  • Progress Indication: Progress bar + per-item status (Screens 2-3) reused pattern for any long-running async operation. Consistent with photo upload progress.
  • Conflict Resolution UI: Screen 3 pattern (quarantine + manual review) applies to any offline collision scenario. Centralize conflict handler <ConflictResolverUI conflictItems={[...]} />
  • Exponential Backoff: Retry logic with exponential backoff applied to all network-dependent RPC calls. Shared utility across module.
  • Atomic Transactions: Each sync item wrapped in atomic transaction to prevent partial state. Coordinated with backend RPC contract to ensure "at-least-once" semantics.
