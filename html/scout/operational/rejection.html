Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

Design Philosophy

The Rejection Handling workflow follows a binary decision flow pattern determining the fate of failed lots. The 2-screen flow maps inspection failures to one of two pathways: reconditioning (with specific treatment guidance) or terminal closure (permanent removal from marketplace). This workflow is triggered only after a lot fails inspection and encodes critical business logic for quality assurance.

Key patterns: Evidence-backed decision (show defect photos), risk classification (reconditionable vs. food-safety terminal), and pathway determination (actionable next steps).

Binary decision determining lot's final status. Reconditionable defects show treatment pathway; food-safety risks trigger permanent closure.

State Management & Backend Logic

Local Wizard State

const [wizardState, setWizardState] = useState({ lot_id: string; inspection_id: string; defect_reason_code: string; // from catalogue defect_photos: string[]; decision: "recondition" | "terminal" | null; reconditioning_path?: string; // e.g. "Re-dry", "Re-sort" terminal_notes?: string; attestation_checked: boolean; });

Defect Classification Logic

const DEFECT_CATALOGUE = { // Reconditionable "high_moisture": { path: "Re-dry on clean cloth", terminal: false }, "foreign_matter": { path: "Re-clean & sort", terminal: false }, "mixed_size": { path: "Re-sort by size", terminal: false }, // Terminal (food-safety critical) "mold": { path: null, terminal: true }, "infestation": { path: null, terminal: true }, "rancidity": { path: null, terminal: true }, "rodent_damage": { path: null, terminal: true }, "adulteration": { path: null, terminal: true } }; const getReconditioningPath = (defectCode) => { return DEFECT_CATALOGUE[defectCode]?.path || null; };

Backend Transaction

BEGIN TRANSACTION; 1. INSERT INTO lot_rejections ( lot_id, inspection_id, defect_reason_code, photos_json, decision_timestamp ) VALUES (...); 2. UPDATE lots SET status = CASE WHEN $decision = 'terminal' THEN 'terminal_rejected' WHEN $decision = 'recondition' THEN 'awaiting_reconditioning' ELSE status END WHERE id = $1; 3. IF decision = 'terminal' THEN INSERT INTO lot_terminal_closures ( lot_id, reason, scout_notes, closure_date ) VALUES ($1, $defect_code, $notes, now()); END IF; 4. IF decision = 'recondition' THEN INSERT INTO reconditioning_tasks ( lot_id, treatment_type, expected_completion ) VALUES ($1, $path, now() + interval '5 days'); END IF; COMMIT;

Common Patterns & Reusability

  • Defect Photo Gallery: Photo display pattern (Screen 1) reused from Lot Verification and Storage Re-check. Extract as <DefectPhotoGallery defectPhotos={[...]} reviewMode />
  • Binary Decision Pattern: Radio selection between two outcome paths mirrors Pond Inspection pass/fail logic. Use Context to persist decision state across screens.
  • Status Badge Theming: Red critical status for rejection, amber warning for decision-required state. Consistent with other workflow verdict displays.
  • Defect Catalogue Lookup: Centralize DEFECT_CATALOGUE in shared config for cross-workflow consistency. All rejection/inspection workflows reference same set of defect codes.
  • Offline Support: Both screens work offline (no real-time calls). Entire workflow cached locally; sync queues rejection decision + photos on network availability.
