Scout Operations

Register Farmer (Scout-Assisted)

Complete 12-step workflow for scout-assisted farmer registration with real-time mobile OTP verification, farm details capture, and admin approval queue.

A 12-screen wizard. The scout sits with the farmer (often the farmer's house or pondside), explains the consent line, collects answers, captures photos of documents, and submits. The farmer's phone receives the OTP; scout reads it from the phone and enters it. On submit, the farmer enters Pending Approval state.

tRPC Procedure Architecture (13 Procedures)

The Register Farmer workflow consists of 13 tRPC procedures organized by authentication level.

Procedure Breakdown:
• 3 Public procedures: Identity, OTP Send, OTP Verify
• 10 Protected procedures: Language, Address, Pond Details (3), Bank (2), Verify Bank
• 1 Atomic transaction: Farmer Registration Submit (7 simultaneous inserts)

File: /src/server/api/routers/farmer.ts

export const farmerRouter = t.router({ // Screen 1-2: Consent & Mobile Verification initFarmerRegistration: publicProcedure, sendFarmerMobileOtp: publicProcedure, verifyFarmerMobileOtp: publicProcedure, // Screen 3: User Identity submitFarmerIdentity: publicProcedure, // Screens 4-10: Setup Wizard (Protected) selectLanguage: protectedProcedure, submitResidentialAddress: protectedProcedure, submitFirstPondDetails: protectedProcedure, submitPondLegalDocuments: protectedProcedure, submitPondAddress: protectedProcedure, submitPondGeofencing: protectedProcedure, submitBankAccount: protectedProcedure, verifyBankAccount: protectedProcedure, // Screen 11: Review & Submit submitFarmerRegistration: protectedProcedure, });

Branching Logic (Screen 2 - Mobile Verification)

  • If OTP matches EXISTING farmer: Load farmer detail screen (Update mode)
  • If OTP for NEW user: Continue setup wizard (Create mode)

Atomic Transaction (Screen 11 - Review & Submit)

  • 1. Insert user record (role=farmer, status=pending_approval)
  • 2. Insert farmer_profiles
  • 3. Insert ponds (first pond)
  • 4. Insert user_documents (ownership photos)
  • 5. Insert user_bank_accounts (encrypted)
  • 6. Insert farmer_approval_queue
  • 7. Insert farmer_registration_events (audit trail)
  • → On success: Trigger async Inngest jobs
  • → On failure: Rollback all 7 inserts (no partial state)

  • ✓ Props Interface: All components export TypeScript interfaces for props
  • ✓ Error Handling: All inputs display errors consistently (red border + error message below)

  • ✓ Colors: Use CSS variables (--accent-primary, --text-main) from root design system
  • ✓ Validation: Frontend validation mirrors backend Zod schemas (shared-types)

  • ✓ Mobile-First: Responsive breakpoints: 360px, 600px, 1024px

Core Database Tables (Drizzle ORM)

users (Universal User Record)

CREATE TABLE users ( id UUID PRIMARY KEY, mobile VARCHAR(15) UNIQUE NOT NULL, role ENUM('scout','farmer','popper','admin'), status ENUM('pending_approval','active','suspended'), preferred_language ENUM('en','hi','bn') DEFAULT 'en', created_via_scout_id UUID REFERENCES scouts(id), created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() );

farmer_profiles (Farmer-Specific)

CREATE TABLE farmer_profiles ( id UUID PRIMARY KEY, user_id UUID NOT NULL UNIQUE REFERENCES users(id), full_name VARCHAR(100) NOT NULL, date_of_birth DATE, district VARCHAR(50) NOT NULL, residential_address TEXT NOT NULL, created_at TIMESTAMP DEFAULT NOW() );

user_bank_accounts (Encrypted)

CREATE TABLE user_bank_accounts ( id UUID PRIMARY KEY, user_id UUID NOT NULL REFERENCES users(id), account_holder_name_encrypted BYTEA NOT NULL, account_number_encrypted BYTEA NOT NULL, ifsc_code VARCHAR(11) NOT NULL, verified BOOLEAN DEFAULT FALSE, verified_at TIMESTAMP, penny_drop_job_id VARCHAR(255), created_at TIMESTAMP DEFAULT NOW() );
⚠️ Encryption:
• Bank account numbers: AES-256-GCM encrypted at-rest
• Account holder names: AES-256-GCM encrypted
• OTP: Hashed with bcrypt (never stored plaintext)
• IFSC code: NOT encrypted (public information)

farmer_approval_queue (Workflow)

CREATE TABLE farmer_approval_queue ( id UUID PRIMARY KEY, farmer_id UUID NOT NULL REFERENCES users(id), status ENUM('pending','approved','rejected'), approved_by_admin_id UUID, created_at TIMESTAMP DEFAULT NOW(), approved_at TIMESTAMP );

Additional Tables

  • ponds: Farmer's pond details (area, GPS, geofence)
  • user_documents: Pond ownership/lease photos
  • user_otp_verifications: OTP storage with expiration (10 min)
  • farmer_registration_events: Complete audit trail

Async Jobs with Inngest (Serverless Queue)

1. farmer.send-otp-sms (Fire-and-Forget)

Trigger: Screen 2 "Mobile Verification"
Steps: Generate → Hash → Store → Send via Twilio → Log
Retry: 4 attempts with exponential backoff (1s, 5s, 30s)
DLQ: Move to dead-letter queue after 4 failures

2. farmer.penny-drop-verification (Wait-for-External)

Trigger: Screen 10 "Bank Account Details"
Steps: Initiate → Wait for webhook (max 5 min) → Verify match → Update status
Webhook: POST /api/webhooks/razorpay/verification-complete

3. farmer.registration-submitted (Chained)

Trigger: Screen 11 "Review & Submit"
Steps: Verify farmer exists → Send confirmation SMS → Notify admin
Dependencies: Ensure farmer record created before SMS

Inngest Configuration

import { Inngest } from "inngest"; export const inngest = new Inngest({ id: "makhana-scout", retryConfig: { initial: 1000, // 1 second maxAttempts: 4, multiplier: 5, // 5x backoff }, });

Security Best Practices

Sensitive Data Handling

Never Log:
✗ Full mobile numbers (log only last 4: ***6789)
✗ Full account numbers (log hash or last 4)
✗ Tokens or OTP
✗ Encryption keys or secrets

Safe to Log:
✓ Last 4 digits of mobile (padded: ***6789)
✓ District/state (non-identifying)
✓ Farmer status (pending, approved)
✓ Event types and timestamps

Data Encryption

FieldMethodRotation
Bank Account NumberAES-256-GCMNever (stable)
Account Holder NameAES-256-GCMNever (stable)
OTPbcrypt 10 roundsN/A (hashed)

OTP Security

  • Generate 6-digit OTP locally (don't ask user for it)
  • Hash with bcrypt (10 rounds) before storage
  • Set expiration: 10 minutes
  • Max 5 attempts before blocking
  • Compare using bcrypt.compare() (constant-time)

Purpose
Establishes that the scout is acting on the farmer's behalf and the farmer consents.
Business Requirement
Consent must be explicitly recorded — scout-assisted registration creates a binding profile for the user, so a consent timestamp + scout's identity + farmer's mobile is the audit minimum. The farmer's verbal consent is the source; the scout attests via a checkbox.

Single screen with plain-language explanation, consent checkbox, primary CTA.
User Actions

Technical Details
Wizard context initialised: { role: 'farmer', consent_at: now, scout_id }. No DB write yet.
Design Details
Headline "Register a Farmer" + plain-language explanation of what will be collected. Consent checkbox: "The farmer has given verbal consent to register on FenBridge". Disabled CTA until ticked.
Database Schema
None at this step.

2Mobile Verification (OTP Send & Verify)
Purpose
Enter farmer's mobile number and trigger SMS verification. OTP entry is handled inside this screen via a bottom sheet overlay.
Business Requirement
OTP confirms the mobile entered is real and belongs to the farmer (the farmer's phone must physically receive it). Without verified mobile, the farmer can't later log in themselves.
User Experience
Mobile input field with helper text, country code prefix (+91), "Send OTP" CTA. Brings up OTP entry bottom sheet.
User Actions
Type farmer's mobile number → Tap "Send OTP" to open interactive bottom sheet overlay. Enter OTP to auto-verify.
Technical Details
Calls sendFarmerMobileOtp and verifyFarmerMobileOtp. Triggers background SMS async job. Opens inline sheet overlay for OTP Entry to avoid page navigation. Note that verify_mobile_entry exists in the codebase as a standalone fallback screen (Step 3 of 13 if using a 13-screen standalone progression) but the production app integrates it as an interactive sheet inside Step 2, resulting in a streamlined 11-step wizard + 1 confirmation page workflow.
Design Details
Mobile chip + "Change" link · Send OTP primary · cooldown timer.
Database Schema
