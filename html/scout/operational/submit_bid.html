Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedures (5 Screens)

This proxy system records supply contracts for offline farmers against targeted buyer bids.

State Guard:
• Buyer requirements must be exactly open. Expired or fully awarded listings block bid placement.

Key Procedures

  • scout.bid.initProxy — Protected. Fetches open buyer requirements, highlights active parameters, and pulls farmer profile coordinates.
  • scout.bid.submitProxy — Protected. Atomic mutator. Inserts a new row in bids with submitted_by_scout = true and logs historical trail.

DO's and DON'Ts

✅ DO:
• Reuse TopAppBar, StepperBar, InfoBanner, ExpandableSection, CTAFooter in new screens
• Use NumberInput for all numeric fields (price, quantity) with min/max/step props
• Use DateInput for all date selections with validation
• Reuse SearchInput pattern for farmer/requirement filtering
• Calculate and display live revenue as user enters price/quantity data
• Check COMPONENT_ARCHITECTURE_GUIDE.md BEFORE creating any new component
• Use multi-select for lot selection to allow sourcing from multiple ponds
• Validate bid price against buyer's min/max range in real-time
❌ DON'T:
• Create duplicate NumberInput, DateInput, or SearchInput components
• Create separate FarmerSearch vs RequirementSearch; reuse SearchInput with same pattern
• Inline styles — always use CSS variables
• Local state for API-driven data (use Context/hooks)
• Create separate QuantityCard or PriceCard; use NumberInput with props variants
• Skip live revenue calculation updates as user changes price/quantity
• Show success before confirming the confirmation checkbox
• Allow bid submission if price exceeds buyer's max or quantity below minimum

Database Schemas (Drizzle ORM DDL)

bids (Procurement Ledger Additions)

CREATE TABLE bids ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), requirement_id UUID NOT NULL REFERENCES buyer_requirements(id), farmer_id UUID NOT NULL REFERENCES farmers(id), bid_price_per_kg NUMERIC(6,2) NOT NULL, quantity_kg INT NOT NULL, delivery_window_days INT NOT NULL DEFAULT 7, dispatch_location TEXT NOT NULL, status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','accepted','rejected','withdrawn')), submitted_by_scout BOOLEAN NOT NULL DEFAULT true, scout_id UUID REFERENCES users(id), created_at TIMESTAMPTZ NOT NULL DEFAULT now() ); CREATE INDEX idx_bids_requirement ON bids (requirement_id); CREATE INDEX idx_bids_farmer ON bids (farmer_id);

Row-Level Security (RLS)

-- Scout can only submit proxy bids for farmers in their zone CREATE POLICY scout_bid_proxy_write ON bids FOR INSERT WITH CHECK ( submitted_by_scout_id = current_user_id() ); -- Bids are immutable once submitted CREATE POLICY bids_no_update ON bids FOR UPDATE USING (false);

Strict Input Validations & Error Enforcements

  • Bid Price Thresholds: Validates against target buyer budgets. Displays alerts if the bid price exceeds target parameters.
  • Minimum Quantity checks: Asserts that bid volume satisfies the buyer's minimum contract constraints.

Offline Caching & Queue Synchronization

  • Offline Lockouts: Bids register locally, freezing variables to prevent updates until synchronized.

Security & Media Protection Protocols

  • RLS Bounds: Asserts that scouts only bid on behalf of farmers officially mapped to their assigned zones.
Submit Bid Steps Directory
5 Steps
1 Choose Farmer User
Purpose
Identifies the specific offline farmer for whom the scout is submitting a proxy bid.
Business Requirement
Scouts are restricted to selecting active, registered farmers within their designated operational zone.
User Experience (UX)

Searchable index with real-time suggestions.

Details cards display active ponds, coordinates, and pending validation flags to help scouts select the correct account.

Technical Details
Validates and saves the target farmer_id in the current wizard memory block.
2 Select Active Buyer Requirement
Purpose
Identifies which open buyer request the scout is submitting the supply offer against.
Business Requirement
Pulls active procurement orders, showing required grades (A/B/C) and volume benchmarks.
User Experience (UX)

Cards display target price ranges, minimum supply metrics, and delivery dates.

Interactive filters categorize requirements by grade and district.

Technical Details

Registers the selected requirement_id.

Checks date parameters to block placing bids on closed listings.

3 Price Offer & Ready-By Date
Purpose
Enters the customized price per kg and estimated crop readiness date.
Business Requirement

Ensures bid price satisfies base requirements.

Records the estimated harvest readiness window.

User Experience (UX)

Slider or input widgets for Price/Kg and Quantity.

Interactive date pickers select target ready dates.

Renders calculated projected revenues.

Technical Details
Performs validations comparing pricing inputs to buyer limits.
4 Fulfillment Source Selection
Purpose
Maps the bid to specific physical ponds or registered warehouse storage lots.
Business Requirement
Enforces connection of bids to verified physical assets to guarantee supply authenticity.
User Experience (UX)

Renders list cards of the farmer's registered ponds and storage vaults.

Displays size, landmarks, and available crop quantities for each asset.

Technical Details
Saves target pond_id or lot_id parameters in wizard states.
5 Review & Confirm Bid Submission
Purpose
Summarizes all bidding parameters and registers the proxy bid contract.
Business Requirement

Commits atomic transaction in Postgres database.

Sets bid status to pending.

User Experience (UX)

Comprehensive recaps card displaying all entered details, target prices, ready dates, and mapped ponds.

Primary Cyan CTA submits bid.

Success popup redirects to dashboard.

Technical Details
Invokes submitProxyBid, updating bid lists and clearing wizard configurations.
