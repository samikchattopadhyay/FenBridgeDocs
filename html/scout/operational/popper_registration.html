Scout Operations

Register Popper (Scout-Assisted)

Complete 15-step workflow for scout-assisted popper registration with business identity capture, warehouse management, real-time mobile OTP verification, and admin approval queue.

A 15-screen wizard. The scout sits with the popper (at their business/warehouse), explains the consent line, collects answers, captures business documents and warehouse photos, and submits. The popper's contact person's phone receives the OTP; scout reads it from the phone and enters it. On submit, the popper enters Pending Approval state and warehouse enters fitness check queue.

Backend Architecture & Implementation Guide

For Developers — Complete Specifications for Backend Implementation

tRPC Procedure Architecture (16 Procedures)

The Register Popper workflow consists of 16 tRPC procedures organized by authentication level.

Procedure Breakdown:
• 4 Public procedures: Init Registration, Send OTP, Verify OTP, Submit Identity
• 12 Protected procedures: Language, Role, Address, Business Details (3), Warehouse (4), Bank (2)
• 1 Atomic transaction: Popper Registration Submit (8 simultaneous inserts)

File: /src/server/api/routers/popper.ts

export const popperRouter = t.router({ // Screens 1-4: Contact Person Info initPopperRegistration: publicProcedure, sendPopperMobileOtp: publicProcedure, verifyPopperMobileOtp: publicProcedure, submitPopperIdentity: publicProcedure, // Screens 5-14: Setup Wizard (Protected) selectLanguage: protectedProcedure, submitResidentialAddress: protectedProcedure, submitBusinessDetails: protectedProcedure, submitBusinessLegalDocuments: protectedProcedure, submitBusinessAddress: protectedProcedure, submitWarehouseAddress: protectedProcedure, submitWarehouseDetails: protectedProcedure, submitWarehousePhotos: protectedProcedure, submitWarehouseGeofencing: protectedProcedure, submitBankAccount: protectedProcedure, verifyBankAccount: protectedProcedure, // Screen 14: Review & Submit submitPopperRegistration: protectedProcedure, });

Branching Logic (Screen 11 - Mobile Verification)

  • If OTP matches EXISTING popper: Load popper detail screen (Update mode)
  • If OTP for NEW user: Continue setup wizard (Create mode)

Atomic Transaction (Screen 14 - Review & Submit)

  • 1. Insert user record (role=popper, status=pending_approval)
  • 2. Insert popper_profiles
  • 3. Insert business_profiles
  • 4. Insert warehouses (first warehouse)
  • 5. Insert user_documents (business docs + warehouse ownership)
  • 6. Insert user_bank_accounts (encrypted)
  • 7. Insert popper_approval_queue
  • 8. Insert popper_registration_events (audit trail)
  • → On success: Trigger async Inngest jobs
  • → On failure: Rollback all 8 inserts (no partial state)

Core Database Tables (Drizzle ORM)

users (Universal User Record)

CREATE TABLE users ( id UUID PRIMARY KEY, mobile VARCHAR(15) UNIQUE NOT NULL, role ENUM('scout','farmer','popper','admin'), status ENUM('pending_approval','active','suspended'), preferred_language ENUM('en','hi','bn') DEFAULT 'en', created_via_scout_id UUID REFERENCES scouts(id), mobile_verified_at TIMESTAMP, created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() );

popper_profiles (Popper-Specific)

CREATE TABLE popper_profiles ( id UUID PRIMARY KEY, user_id UUID NOT NULL UNIQUE REFERENCES users(id), contact_person_name VARCHAR(100) NOT NULL, date_of_birth DATE, popper_type ENUM('processor','trader','warehouse','multi') NOT NULL, residential_district VARCHAR(50) NOT NULL, residential_address TEXT NOT NULL, residential_landmark TEXT, business_name VARCHAR(150) NOT NULL, pan_number VARCHAR(10) NOT NULL, gstin VARCHAR(15), fssai_number VARCHAR(14), business_district VARCHAR(50), business_address TEXT, business_landmark TEXT, preferred_language ENUM('en','hi','bn') DEFAULT 'en', created_at TIMESTAMP DEFAULT NOW() );

warehouses (Warehouse Details)

CREATE TABLE warehouses ( id UUID PRIMARY KEY, popper_id UUID NOT NULL REFERENCES users(id), warehouse_code VARCHAR(20) UNIQUE NOT NULL, name VARCHAR(100), built_capacity_kg INTEGER NOT NULL, bag_capacity INTEGER, warehouse_type ENUM('processor','trader','storage') NOT NULL, district VARCHAR(50) NOT NULL, address TEXT NOT NULL, landmark TEXT, gps_lat DECIMAL(10,7), gps_lng DECIMAL(10,7), arrival_accuracy_m INTEGER, status ENUM('pending_approval','active','inactive','rejected') DEFAULT 'pending_approval', created_via_scout_id UUID REFERENCES scouts(id), created_at TIMESTAMP DEFAULT NOW() );

user_bank_accounts (Encrypted)

CREATE TABLE user_bank_accounts ( id UUID PRIMARY KEY, user_id UUID NOT NULL REFERENCES users(id), account_holder_name_encrypted BYTEA NOT NULL, account_number_encrypted BYTEA NOT NULL, account_number_masked VARCHAR(20), ifsc_code VARCHAR(11) NOT NULL, bank_name VARCHAR(100), branch_name VARCHAR(100), pennydrop_status ENUM('pending','verified','failed') DEFAULT 'pending', pennydrop_job_id VARCHAR(255), verified_at TIMESTAMP, created_at TIMESTAMP DEFAULT NOW() );
⚠️ Encryption:
• GSTIN: AES-256-GCM encrypted (sensitive business identifier)
• Business legal name: AES-256-GCM encrypted at-rest
• Bank account numbers: AES-256-GCM encrypted at-rest
• Account holder names: AES-256-GCM encrypted
• OTP: Hashed with bcrypt (never stored plaintext)
• IFSC code: NOT encrypted (public information)
• FSSAI: Masked in logs (last 4 digits only)

user_documents (Photos & Certificates)

CREATE TABLE user_documents ( id UUID PRIMARY KEY, user_id UUID NOT NULL REFERENCES users(id), doc_type ENUM('business_pan','gstin_cert','fssai_cert','warehouse_ownership','warehouse_interior','warehouse_exterior'), storage_path VARCHAR(255) NOT NULL, uploaded_by_scout_id UUID REFERENCES scouts(id), created_at TIMESTAMP DEFAULT NOW() );

CREATE TABLE popper_approval_queue ( id UUID PRIMARY KEY, popper_id UUID NOT NULL REFERENCES users(id), status ENUM('pending','approved','rejected') DEFAULT 'pending', approved_by_admin_id UUID REFERENCES admin_users(id), rejection_reason TEXT, created_at TIMESTAMP DEFAULT NOW(), approved_at TIMESTAMP ); CREATE TABLE popper_registration_events ( id UUID PRIMARY KEY, popper_id UUID NOT NULL REFERENCES users(id), event_type VARCHAR(50), event_data JSONB, created_at TIMESTAMP DEFAULT NOW() );

Async Jobs with Inngest (Serverless Queue)

1. popper.send-otp-sms (Fire-and-Forget)

Trigger: Screen 11 (Mobile Verification)
Steps: Generate 6-digit → Hash with bcrypt → Store in DB → Send via Twilio SMS → Log
Retry: 4 attempts with exponential backoff (1s, 5s, 30s)
DLQ: Move to dead-letter queue after 4 failures, alert ops

2. popper.penny-drop-verification (Wait-for-External)

Trigger: Screen 12 (Bank Account Details)
Steps: Initiate Razorpay penny-drop → Store job ID → Wait for webhook (max 5 min) → Verify match → Update status
Webhook: POST /api/webhooks/razorpay/verification-complete

3. popper.registration-submitted (Chained)

Trigger: Screen 14 (Review & Submit)
Steps: Verify popper exists → Send confirmation SMS (in preferred_language) → Notify Admin (dashboard alert)
Dependencies: Ensure popper record created before SMS

import { Inngest } from "inngest"; export const inngest = new Inngest({ id: "makhana-scout", retryConfig: { initial: 1000, // 1 second maxAttempts: 4, multiplier: 5, // 5x backoff }, });

Security Best Practices

Sensitive Data Handling

Never Log:
✗ Full mobile numbers (log only last 4: ***6789)
✗ GSTIN or business legal name
✗ PAN number
✗ Full account numbers (log hash or last 4)
✗ FSSAI numbers (log hash only)
✗ Tokens or OTP
✗ Encryption keys or secrets

Safe to Log:
✓ Last 4 digits of mobile (padded: ***6789)
✓ District/state (non-identifying)
✓ Popper type (processor, trader, warehouse)
✓ Approval status (pending, approved, rejected)
✓ Event types and timestamps
✓ Warehouse code (WARE-1024 format)

Data Encryption

FieldMethodRotation
GSTINAES-256-GCMNever (stable)
Business Legal NameAES-256-GCMNever (stable)
Bank Account NumberAES-256-GCMNever (stable)
Account Holder NameAES-256-GCMNever (stable)
OTPbcrypt 10 roundsN/A (hashed, never retrieved)

OTP Security

  • Generate 6-digit OTP locally (cryptographically random)
  • Hash with bcrypt (10 rounds) before storage
  • Set expiration: 10 minutes
  • Max 5 attempts before blocking (10-min cooldown after 5 failures)
  • Compare using bcrypt.compare() (constant-time, timing-attack resistant)
  • Resend cooldown: 30 seconds enforced server-side

Popper Registration Workflow
15 Steps
1Intro & Consent
Purpose
Scout acts on popper's behalf; verbal consent required before any data collected. Binding profile creation requires explicit attestation.
Business Requirement
Consent timestamp + scout ID + popper mobile = audit minimum. Binding profile creation requires attestation via checkbox. Popper's verbal consent is the source; scout attests via checkbox.
User Experience
Single screen, plain-language explanation, consent checkbox, disabled CTA until ticked. Scout reads consent line aloud to popper.
User Actions
Read consent line aloud to popper → tick "I confirm the popper has given verbal consent to register on FenBridge" → Continue.
Technical Details
Wizard context initialised: { role: 'popper', consent_at: now(), scout_id }. No database write yet. Disabled CTA enforced via form validation.
Design Details
Headline "Register a Popper" + explanation block with business context ("The popper is a business that buys raw makhana seed and processes it into finished product for sale. We will collect contact person's name and mobile, business identity (PAN, GSTIN, FSSAI), warehouse information, and bank account. Once submitted, the registration will be reviewed by our team — the popper will be notified by SMS when approved.") + consent checkbox + disabled CTA until ticked. Helper: "Ensure you have explained the above to the contact person and they have agreed."
Database Schema
None at this step. Context only.
